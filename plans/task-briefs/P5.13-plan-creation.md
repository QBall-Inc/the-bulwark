## Problem Statement

Create a plan-creation skill that produces structured implementation plans through a 4-role collaborative scrum team. The skill uses Claude Code's Task tool as the default orchestration mode and Agent Teams as an enhanced opt-in mode for users who want peer debate between team members.

### Distribution Priority

| Priority | Context | Role |
|----------|---------|------|
| P1 (highest) | CLEAR Framework (`/mnt/c/projects/clear-framework`) | Core workflow feature — produces the initial plan artifact that CLEAR's plan-management skill consumes |
| P2 | Essential Skills standalone repo | General-purpose plan-creation, no branding, portable |
| P3 (lowest) | Bulwark plugin | Optional add-on, Agent Teams proving ground |

Architecture must be portable across all 3 contexts. No hardcoded project paths. No CLEAR-specific dependencies.

## Functional Requirements

### FR1: Dual-Mode Orchestration

- **Task tool mode (default)**: PO first → [Architect, Eng+Delivery Lead] parallel → QA/Critic last
- **Agent Teams mode (enhanced, opt-in)**: PO first via Task tool → 3 teammates (Architect, Eng+Delivery Lead, QA/Critic) debate via mailbox with delegate-mode lead
- Mode detection: Pre-flight checks `$CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` env var
- Agent Teams mode requires BOTH the env var AND explicit user opt-in (not automatic)
- Graceful degradation: if env var absent when Agent Teams requested, notify user and fall back to Task tool

### FR2: 4-Role Scrum Team

| Role | Focus | Execution |
|------|-------|-----------|
| **Product Owner** | Requirements, scope, acceptance criteria, user value | Solo first (codebase exploration via Task tool, both modes) |
| **Technical Architect** | System design, component decomposition, integration, trade-offs | Parallel (Task tool) / Teammate (Agent Teams) |
| **Engineering & Delivery Lead** | Implementation sequencing, effort estimation, scheduling, milestones, dependencies, risk | Parallel (Task tool) / Teammate (Agent Teams) |
| **QA / Critic** | Challenge assumptions, identify gaps, kill criteria, adversarial review | Sequential last (Task tool) / Active participant throughout (Agent Teams) |

Key design decisions:
- PO always runs via Task tool even in Agent Teams mode (codebase exploration is solo, no debate benefit)
- QA/Critic participates throughout in Agent Teams mode (not gated until end) — this is the primary quality advantage of Agent Teams
- Combined Eng+Delivery Lead addresses the 3-4 agent coordination cap from research while preserving distinct functional coverage

### FR3: Implementation Plan Output

Output is a **hybrid Markdown preamble + YAML body** plan:

```markdown
# {Project Name} — Implementation Plan

## Executive Summary
{2-3 paragraphs: goal, approach, key risks — prose for human readability}

## Plan
```yaml
version: "1.0"
project_name: "{name}"
created: "{ISO date}"
created_by: "plan-creation skill"
mode: "task-tool | agent-teams"

phases:
  - id: "phase_1"
    name: "{Phase Name}"
    status: "not_started"
    workpackages:
      - id: "WP1"
        name: "{Workpackage name}"
        description: "{What this delivers}"
        estimated_sessions: {N}
        dependencies: []
      - id: "WP2"
        name: "{Workpackage name}"
        dependencies: ["WP1"]

milestones:
  - id: "m1"
    name: "{Milestone name}"
    phase: "phase_1"
    type: "major"
    requires: ["WP1", "WP2"]

dependency_graph:
  critical_path: ["WP1", "WP2", "WP5"]
  parallel_opportunities:
    - ["WP3", "WP4"]

risks:
  - id: "R1"
    description: "{Risk}"
    severity: "high | medium | low"
    mitigation: "{Mitigation}"

kill_criteria:
  - "{Condition under which project should be abandoned}"
```
```

**CLEAR compatibility**: YAML body field names match CLEAR's `MasterPlan` type interface (`phases[].id`, `phases[].status`, `milestones[].type`, `milestones[].requires`) without importing CLEAR types. When CLEAR resumes, its `parseMasterPlanContent()` can consume the YAML body with minor field mapping.

**Token efficiency**: YAML body for a 3-phase, 12-workpackage plan is ~400-600 tokens vs ~800-1200 for equivalent Markdown tables.

### FR4: SA2 Compliance

- **Task tool mode**: All agent output to `logs/plan-creation/{slug}/{NN}-{role}.md` (standard SA2)
- **Agent Teams mode**: Dual-output contract in every teammate prompt:
  - Full analysis → `logs/plan-creation/{slug}/{NN}-{role}.md` (SA2 artifact)
  - 3-5 sentence coordination summary → mailbox (Agent Teams peer messaging)
- Synthesis stage verifies all log files exist before proceeding

### FR5: Explore-Then-Commit Workflow

Borrowed from Claude Code's plan mode concept:
1. Agents research and produce analysis (explore phase)
2. Orchestrator synthesizes into draft plan
3. User reviews draft via AskUserQuestion approval gate
4. On approval, plan written to user-specified location (or default `plans/`)
5. Critical Evaluation Gate applied to user feedback (Preference/Technical Claim/Architectural Suggestion)

### FR6: Portability

- Self-contained directory: `skills/plan-creation/SKILL.md` + `references/` + `templates/`
- No hardcoded project paths — PO explores codebase autonomously (same as brainstorm SME)
- Only skill dependency: `subagent-prompting` (declared in frontmatter)
- Works in any project with no Bulwark or CLEAR infrastructure

## Technical Design

### Architecture: Sandwich Structure

```
Stage 1: Pre-Flight (SHARED)
├── Parse input (topic/document + optional research file)
├── Slugify topic, create logs/plan-creation/{slug}/
├── Detect mode: check env var + user preference
├── AskUserQuestion if ambiguous (2-3 questions per round)
├── Load subagent-prompting skill
├── Token budget check (warn if >30%)
└── Load references/role-product-owner.md

Stage 2: Context Gathering (SHARED)
├── Spawn PO agent (Opus, Task tool in both modes)
├── PO explores codebase autonomously
├── PO output: logs/plan-creation/{slug}/01-product-owner.md
└── Token budget check

Stage 3A: Scrum Team — Task Tool Mode
├── Spawn [Architect, Eng+Delivery Lead] in parallel (2 Task tool calls)
├── Each reads PO output + problem statement + research synthesis
├── Each writes to logs/plan-creation/{slug}/{NN}-{role}.md
├── Then spawn QA/Critic sequentially with all prior outputs
└── Token budget check (checkpoint if >55%)

Stage 3B: Scrum Team — Agent Teams Mode
├── Lead enters delegate mode (coordination only)
├── Spawn 3 teammates: Architect, Eng+Delivery Lead, QA/Critic
├── Each prompt includes dual-output contract (logs + mailbox)
├── Peer debate: explicit challenge prompts between teammates
├── QA/Critic participates throughout (active adversarial role)
├── In-process display mode (WSL2 safe)
└── Token budget check

Stage 4: Synthesis (SHARED, mode-aware)
├── Task tool: read all 4 log files
├── Agent Teams: read all 4 log files + lead coordination notes
├── Write draft plan using plan output template
├── AskUserQuestion approval gate (explore-then-commit)
├── Critical Evaluation Gate on user responses
└── Write final plan to user-specified location

Stage 5: Diagnostics (SHARED)
└── Write diagnostic YAML to logs/diagnostics/
```

### F# Pipeline Syntax

```fsharp
// Task tool mode
ProductOwner(topic, research?)
|> [Architect, EngDeliveryLead](po_output)
|> QACritic(all_prior_outputs)
|> Synthesis
|> ApprovalGate

// Agent Teams mode
ProductOwner(topic, research?)            // Task tool (solo)
|> AgentTeam[Architect, EngDeliveryLead, QACritic](po_output)  // Peer debate
|> Synthesis
|> ApprovalGate
```

### Mode Detection Logic (Pre-Flight)

```
1. Check $CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS env var
2. If env var is SET:
   - Offer user choice: "Agent Teams enhanced mode is available. Use Agent Teams or Task tool?"
   - Default to Task tool if user doesn't specify
3. If env var is NOT SET:
   - Use Task tool mode
   - If user explicitly requested Agent Teams: notify "Agent Teams requires CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1. Using Task tool mode."
```

## Success Criteria

### Functional (both modes)

1. Skill produces a structured implementation plan with phases, workpackages, milestones, and dependencies
2. All 4 scrum team roles contribute distinct, non-redundant perspectives visible in log artifacts
3. QA/Critic identifies at least one substantive gap not raised by other roles
4. YAML body is structurally compatible with CLEAR's MasterPlan interface
5. Passes `/anthropic-validator` with 0 critical, 0 high

### Task tool mode

6. SA2 compliance: all output in `logs/plan-creation/{slug}/`
7. Follows bulwark-brainstorm patterns (4-part prompting, diagnostics, error handling)

### Agent Teams mode

8. Graceful degradation when env var absent (user notification, Task tool fallback)
9. SA2 compliance: artifacts in logs/, mailbox for coordination only
10. Observable peer challenge in Agent Teams mode (teammate X challenges teammate Y)

### Portability

11. Skill runs in a fresh project (no Bulwark, no CLEAR) with no errors
12. Rsync to Essential Skills repo succeeds and skill remains functional

## Verification Plan

### Manual Test Protocol (Session 3)

**Task Tool Mode:**
1. Invoke skill with real topic and research file, env var NOT set
2. Verify PO spawns first, explores codebase autonomously
3. Verify Architect + Eng+Delivery Lead spawn in parallel
4. Verify QA/Critic spawns last with all prior outputs
5. Verify all 4 log files exist in `logs/plan-creation/{slug}/`
6. Verify plan output has Markdown preamble + YAML body
7. Verify YAML body has phases, workpackages, milestones, dependencies
8. Verify diagnostic YAML written

**Agent Teams Mode:**
1. Set `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`
2. Invoke skill, select Agent Teams when offered
3. Verify delegate mode active (lead coordinates only)
4. Verify 3 teammates spawn with correct roles
5. Verify peer challenge occurs via mailbox
6. Verify all 4 SA2-compliant log files exist
7. Verify synthesis incorporates debate insights
8. Verify diagnostic YAML captures AT-specific metrics
9. Verify no orphaned processes after completion

**Cross-Mode Comparison (if time permits):**
- Same topic through both modes, compare output quality and token cost

**Validation:**
- Run `/anthropic-validator` — 0 critical, 0 high
- Portability test in fresh project directory

## Kill Criteria (from Critical Analyst)

1. Comparative test shows no quality difference between Agent Teams and Task tool → descope Agent Teams mode
2. 3+ of 4 agents produce overlapping analysis → reduce roles further
3. Agent Teams fails to produce all 4 SA2-compliant log files → kill Agent Teams mode
4. Agent Teams experimental flag removed or API changes before Session 2 → defer Agent Teams indefinitely
5. Total implementation exceeds 3 sessions → ship whatever works at Session 3 boundary
