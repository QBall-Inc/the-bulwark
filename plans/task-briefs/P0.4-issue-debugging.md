## 1. Functional Requirements

### 1.1 Core Purpose

Provide systematic methodology for issue debugging that prevents the "first error fix and declare done" anti-pattern. This skill ensures:

- **Root cause analysis** before fixing (not just symptoms)
- **Impact/dependency mapping** to understand affected areas
- **Validation planning** with tiered test priorities
- **Confidence assessment** for fix reliability
- **Escalation triggers** for manual testing when needed

**Key Problem Addressed**: AI (and developers) often fix the first visible error and declare victory, without verifying end-user functionality actually works. This skill encodes systematic debugging methodology.

### 1.2 Content Requirements

| Requirement | Description |
|-------------|-------------|
| FR-1 | Root cause analysis methodology (5 Whys, hypothesis-driven) |
| FR-2 | Impact/dependency analysis procedure (upstream/downstream mapping) |
| FR-3 | Debug report format (separate file at `logs/debug-reports/`) |
| FR-4 | Tiered validation plan (P1 must, P2 should, P3 nice-to-have) |
| FR-5 | Confidence rubric (high/medium/low criteria) |
| FR-6 | Escalation triggers for manual testing |
| FR-7 | Debug journey logging (for medium/high complexity) |
| FR-8 | Anti-patterns checklist (what NOT to do) |

### 1.3 User Stories

- As the **Orchestrator**, I need systematic debugging methodology so fixes actually solve the user's problem
- As an **IssueAnalyzer agent**, I need a structured format for debug reports that downstream stages can consume
- As a **FixValidator agent**, I need a validation plan with tiered priorities so I don't run full regression
- As a **Developer**, I need clear escalation triggers so I know when manual testing is required

### 3.2 Debug Report Schema

**Location**: `logs/debug-reports/{issue-id}-{timestamp}.yaml`

```yaml
debug_report:
  # Metadata
  metadata:
    issue_id: "{id}"
    timestamp: "{ISO-8601}"
    analyzer: "bulwark-issue-analyzer"
    complexity: low | medium | high

  # Analysis Section
  analysis:
    symptom: "{user-visible problem}"
    root_cause: "{underlying reason - not just symptom}"
    fix_approach: "{recommended approach}"

  # Impact Analysis Section
  impact_analysis:
    affected_files:
      - "{file path}"
    upstream_dependencies:
      - "{what calls this code}"
    downstream_effects:
      - "{what this code affects}"
    risk_scope: isolated | medium | broad

  # Validation Plan Section
  validation_plan:
    tests_to_execute:
      - path: "{test file}"
        reason: "{why this test}"
        priority: 1  # P1=must, P2=should, P3=nice-to-have
    functionalities_to_validate:
      - "{user-level verification item}"

  # Confidence Criteria Section
  confidence_criteria:
    high:
      - "All P1-P2 tests pass"
      - "No regression in existing tests"
    medium:
      - "P1 tests pass, some P2-P3 skipped"
    low:
      - "Tests cannot reliably validate"
      - "Manual testing required"

  # Debug Journey Section (required for medium/high complexity)
  debug_journey:
    hypotheses_tested:
      - hypothesis: "{what was suspected}"
        result: confirmed | rejected
        evidence: "{what proved/disproved it}"
```

### 3.3 Root Cause Analysis Methodology

#### 3.3.1 The 5 Whys Method

```
Problem: Login fails with 500 error
├─ Why? → API returns null for user profile
├─ Why? → Database query found no records
├─ Why? → Migration didn't run on deploy
├─ Why? → Deployment script skipped migration step
└─ Why? → Environment variable missing

Root Cause: Missing environment variable (not the 500 error)
```

**Key Principle**: Focus on HOW and WHY the defect occurred, not just WHERE.

#### 3.3.2 Hypothesis-Driven Debugging

| Step | Action |
|------|--------|
| **Observe** | Gather data: errors, logs, stack traces, reproduction steps |
| **Hypothesize** | Form a **falsifiable** hypothesis about the cause |
| **Experiment** | Design tests to prove/disprove hypothesis |
| **Conclude** | If confirmed, proceed. If rejected, form new hypothesis |
| **Document** | Write down hypotheses tested (prevents circular investigation) |

#### 3.3.3 Binary Search / Bisection

For isolating bugs in time or space:
1. Place checkpoint at midpoint
2. Determine if bug is before or after
3. Repeat on remaining half
4. Continue until exact location found

**Efficiency**: O(log n) instead of O(n) - ~7 tests for 100 commits.

### 3.4 Impact / Dependency Analysis

#### 3.4.1 Mapping Procedure

```
Affected Code: src/auth/login.ts

Upstream (What calls this?):
├─ src/api/auth-routes.ts → POST /login
├─ src/middleware/session.ts → validateSession()
└─ src/pages/login.tsx → handleSubmit()

Downstream (What does this affect?):
├─ User dashboard → fetches profile on load
├─ Settings page → assumes profile exists
└─ API responses → include user context
```

#### 3.4.2 Risk Scope Assessment

| Scope | Definition | Validation Approach |
|-------|------------|---------------------|
| **Isolated** | Self-contained, no affected areas | Unit tests only |
| **Medium** | 1-5 affected areas | Integration tests |
| **Broad** | >5 affected areas, cross-cutting | Integration + E2E + manual |

### 3.5 Tiered Validation Plan

#### 3.5.1 Priority Definitions

| Priority | Definition | Action |
|----------|------------|--------|
| **P1 (Must)** | Direct tests of affected function | Always run |
| **P2 (Should)** | Integration tests for upstream/downstream | Run if time/tokens allow |
| **P3 (Nice-to-have)** | E2E tests, edge cases | Run if available |

#### 3.5.2 Selection Criteria

```yaml
# P1: Tests that directly cover the bug
- Tests in the same file/module as the fix
- Tests named after the affected function
- Tests that were failing before the fix

# P2: Tests that cover integration points
- Tests for upstream callers
- Tests for downstream consumers
- Tests for related functionality

# P3: Broader coverage
- E2E tests that include the affected flow
- Performance tests
- Edge case tests
```

### 3.6 Confidence Rubric

#### 3.6.1 Assessment Criteria

| Level | Criteria |
|-------|----------|
| **High** | All P1-P2 tests pass, no regression, at least one functionality verified |
| **Medium** | P1 tests pass, some P2-P3 skipped or not applicable |
| **Low** | Tests cannot reliably validate, broad risk scope with untested paths |

#### 3.6.2 Escalation Triggers

Manual testing is **required** when ANY of:
- Confidence level is `low`
- Risk scope is `broad` AND confidence is not `high`
- Any functionality cannot be validated via automated tests
- Security-related fix without security test coverage

**Escalation Message Format**:
```
Orchestrator should inform user: Manual testing required for:
- [functionality 1]
- [functionality 2]
Reason: [why automated validation insufficient]
```

### 3.7 Debug Journey Logging

#### 3.7.1 When Required

| Complexity | Debug Journey Required? |
|------------|------------------------|
| Low | Optional (skip for simple issues) |
| Medium | **Required** |
| High | **Required** |

#### 3.7.2 Format

```yaml
debug_journey:
  started_at: "{timestamp}"
  completed_at: "{timestamp}"

  hypotheses_tested:
    - hypothesis: "Database connection timeout causing failure"
      tested_at: "{timestamp}"
      method: "Checked DB logs and connection pool stats"
      result: rejected
      evidence: "DB logs show successful queries, pool not exhausted"

    - hypothesis: "Null profile object when user has no data"
      tested_at: "{timestamp}"
      method: "Added logging at profile access point"
      result: confirmed
      evidence: "Stack trace shows NPE at profile.name access"

  investigation_path:
    - "Started with error log analysis"
    - "Ruled out infrastructure issues"
    - "Narrowed to application code"
    - "Identified profile access as failure point"
```

### 3.8 Anti-Patterns Checklist

#### 3.8.1 What NOT to Do

| Anti-Pattern | Description | Correct Approach |
|--------------|-------------|------------------|
| **Shotgun Debugging** | Making random changes hoping bug disappears | Hypothesis-driven approach |
| **Fix Without Verify** | Declaring "fixed" without running tests | Always verify at code + user level |
| **Symptom Fixing** | Adding workarounds instead of fixing root cause | Use 5 Whys to find root cause |
| **Blind AI Trust** | Accepting AI suggestions without verification | Test the scenario that was failing |
| **Full Regression** | Running 4000+ tests after every fix | Use tiered validation plan |
| **Circular Investigation** | Repeating already-tested hypotheses | Document debug journey |

#### 3.8.2 Verification Checklist

Before declaring a fix complete:

- [ ] Root cause identified (not just symptom)
- [ ] Impact analysis completed
- [ ] Fix implemented
- [ ] P1 tests pass
- [ ] P2 tests pass (or documented why skipped)
- [ ] Confidence level assessed
- [ ] Manual testing escalated if required
- [ ] Debug journey documented (if medium/high complexity)

## 9. Integration Points

### 9.1 Pipeline Integration

This skill is consumed by the Fix Validation pipeline:

```fsharp
IssueAnalyzer (uses issue-debugging skill)
|> FixWriter (reads debug report)
|> TestWriter (reads validation plan)
|> FixValidator (executes validation plan, uses confidence rubric)
|> CodeReviewer (reviews all)
```

### 9.2 Agent Integration

| Agent | How It Uses This Skill |
|-------|------------------------|
| `bulwark-issue-analyzer` (P1.2) | Loads via `skills: issue-debugging`, follows methodology, produces debug report |
| `bulwark-fix-validator` (P1.3) | Loads via `skills: issue-debugging`, reads debug report, executes validation plan, assesses confidence |

### 9.3 Artifact Flow

```
IssueAnalyzer
    └─> logs/debug-reports/{issue-id}.yaml  (debug report)
    └─> logs/issue-analyzer-{timestamp}.yaml (standard log with summary)
              │
              ├─> Orchestrator (reads summary, gets debug report path)
              │
              ├─> FixValidator (reads debug report, executes validation plan)
              │         └─> validation_results with confidence level
              │
              └─> CodeReviewer (reviews all, makes approval decision)
```
