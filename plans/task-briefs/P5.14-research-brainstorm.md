# P5.14: Research & Brainstorm Skills (Production-Grade)

## Context

Session 58 created a draft `skills/research-brainstorm/` skill with 3 reference docs (phase-1-viewpoints.md, phase-2-roles.md, synthesis-protocol.md). The draft has correct methodology but non-standard conventions and is a single monolithic skill.

Session 59 analysis determined the skill must be split into two independent skills — `bulwark-research` and `bulwark-brainstorm` — primarily due to token budget constraints (a single session cannot run 10 agents + 2 syntheses within 200K context). The split also satisfies CS1 (single responsibility) and enables independent invocation.

### Design Decisions (Session 59)

| Decision | Rationale |
|----------|-----------|
| Split into two skills | Token budget: 5 agents + synthesis already consumes most of 200K |
| Custom output templates per skill | subagent-output-templating is code-agent oriented; these are documentation agents |
| SME explores codebase autonomously | Hardcoding architecture.md/tasks.yaml makes skill Bulwark-specific, not portable |
| AskUserQuestion before spawning | Prevents wasting 5 agent spawns on under-specified problem |
| Synthesis = template + enforcement, not reasoning logic | Reasoning is inherent to orchestrator; enforce completeness and provide structure |
| All Phase 2 agents are Opus | Prescriptive, opinionated analysis requires deep reasoning; quality gap vs Sonnet is significant |

## Functional Requirements

### FR1: bulwark-research Skill

Structured multi-viewpoint research on a given topic. Spawns 5 Sonnet sub-agents in parallel, each analyzing from a distinct analytical viewpoint, then synthesizes.

#### FR1.1: Skill Structure

```
skills/bulwark-research/
├── SKILL.md                              (main skill document)
├── references/
│   ├── viewpoint-direct-investigation.md (viewpoint 1 definition + prompt)
│   ├── viewpoint-practitioner.md         (viewpoint 2 definition + prompt)
│   ├── viewpoint-contrarian.md           (viewpoint 3 definition + prompt)
│   ├── viewpoint-first-principles.md     (viewpoint 4 definition + prompt)
│   └── viewpoint-prior-art.md           (viewpoint 5 definition + prompt)
└── templates/
    ├── viewpoint-output.md               (output template for each viewpoint agent)
    └── synthesis-output.md               (output template for per-topic synthesis)
```

#### FR1.2: Viewpoint Agents

Five parallel Sonnet sub-agents, each with a distinct analytical lens:

| # | Viewpoint | Core Question |
|---|-----------|---------------|
| 1 | Direct Investigation | What is this? How does it work? State of the art? |
| 2 | Practitioner Perspective | How do teams use this in production? What works? |
| 3 | Contrarian Angle | What failure modes do most people overlook? |
| 4 | First Principles | What core problem does this solve? Minimal viable version? |
| 5 | Prior Art / Historical | What similar patterns exist? Lessons from predecessors? |

Each reference doc contains: viewpoint definition, focus areas, prompt template using 4-part format (GOAL/CONSTRAINTS/CONTEXT/OUTPUT), and the output template path.

#### FR1.3: Output Conventions

Agent outputs written to `logs/research/{topic-slug}/`:

```
logs/research/{topic-slug}/
  01-direct-investigation.md
  02-practitioner.md
  03-contrarian.md
  04-first-principles.md
  05-prior-art.md
  synthesis.md
```

#### FR1.4: Invocation

```
/bulwark-research <topic-or-prompt> [--context <file>]
/bulwark-research --doc <path-to-document>
```

- `<topic-or-prompt>`: Free-text topic description or problem statement
- `--context <file>`: Additional context file to provide to all agents
- `--doc <path>`: Use a document as the topic source instead of free text

### FR2: bulwark-brainstorm Skill

Role-based brainstorming on a given topic. Spawns 5 Opus sub-agents in a sequenced pipeline (SME first, 3 parallel, Critic last), then synthesizes.

#### FR2.1: Skill Structure

```
skills/bulwark-brainstorm/
├── SKILL.md                              (main skill document)
├── references/
│   ├── role-project-sme.md              (role 1 definition + prompt)
│   ├── role-product-manager.md          (role 2 definition + prompt)
│   ├── role-technical-architect.md      (role 3 definition + prompt)
│   ├── role-development-lead.md         (role 4 definition + prompt)
│   └── role-critical-analyst.md         (role 5 definition + prompt)
└── templates/
    ├── role-output.md                    (output template for each role agent)
    ├── critic-output.md                  (output template for Critical Analyst — has verdict)
    └── synthesis-output.md               (output template for per-topic synthesis)
```

#### FR2.2: Role Agents

Five Opus sub-agents in sequenced execution:

| # | Role | Execution Order | Input |
|---|------|----------------|-------|
| 1 | Project SME | First (solo) | Problem statement + autonomous codebase exploration |
| 2 | Senior Product Manager | Second (parallel) | Problem statement + SME output |
| 3 | Senior Technical Architect | Second (parallel) | Problem statement + SME output |
| 4 | Senior Development Lead | Second (parallel) | Problem statement + SME output |
| 5 | Critical Analyst | Last (solo) | Problem statement + ALL prior outputs |

Each reference doc contains: role definition, focus areas, prompt template using 4-part format, and the output template path.

#### FR2.3: Project SME Context Gathering

The SME agent MUST NOT receive hardcoded project document paths. Instead:

- SME receives the problem statement and (optionally) Phase 1 research synthesis
- SME is spawned as `general-purpose` subagent type with Opus model
- SME autonomously explores the codebase using Glob, Grep, Read
- Skill instructs: "Identify the MINIMUM files needed to answer: What exists relevant to this topic? Where are the integration points? What must not be disrupted? Do NOT attempt to read the entire codebase."
- SME output documents which files it read and why

This makes the skill portable across any project, not just The Bulwark.

#### FR2.4: Output Conventions

Agent outputs written to `logs/brainstorm/{topic-slug}/`:

```
logs/brainstorm/{topic-slug}/
  01-project-sme.md
  02-product-manager.md
  03-technical-architect.md
  04-development-lead.md
  05-critical-analyst.md
  synthesis.md
```

#### FR2.5: Invocation

```
/bulwark-brainstorm <topic-or-prompt> [--research <synthesis-file>]
/bulwark-brainstorm --doc <path-to-document> [--research <synthesis-file>]
```

- `<topic-or-prompt>`: Free-text topic description or problem statement
- `--doc <path>`: Use a document as the topic source
- `--research <synthesis-file>`: Path to Phase 1 research synthesis (from bulwark-research). If provided, agents receive this as additional input.

### FR3: Cross-Skill Conventions (Both Skills)

#### FR3.1: AskUserQuestion Interviews

Both skills MUST interview the user at two points:

1. **Before spawning agents** — After reading the problem statement/document but before constructing prompts. If the problem statement is ambiguous, under-specified, or could benefit from scope boundaries, interview the user iteratively:
   - Ask 2-3 clarifying questions per round using AskUserQuestion
   - Assess whether the answers provide sufficient clarity to construct high-quality prompts
   - If not, ask up to 3 more questions in a follow-up round
   - Repeat until clarity is achieved (no hard cap on rounds, but each round is 2-3 questions max)
   - If the problem statement is clear and well-scoped from the start, skip this step and note in diagnostics: "Pre-flight interview: skipped (problem statement sufficient)"

2. **After synthesis** — Present key findings and any open questions to the user. Same iterative pattern: 2-3 questions per round, repeat if the user's answers surface additional ambiguities or decisions.

**Key constraint**: Limit each AskUserQuestion invocation to 2-3 questions (do not overwhelm the user with a wall of questions). Iterate in rounds instead.

#### FR3.2: Token Budget Management

Both skills are token-heavy. The orchestration protocol MUST include proactive token checking:

| Checkpoint | Threshold | Action |
|------------|-----------|--------|
| After constructing all prompts | >30% consumed | Warn user: "5 agents will consume significant context" |
| After reading each agent output | Running tally | If approaching 55% after reading 3 of 5, checkpoint with user |
| After synthesis | Must be <65% | Leave room for session closing per SR2 |
| Synthesis complete | >65% | Immediate handoff, do not start additional work |

If token budget is insufficient to complete all 5 agents + synthesis, the skill MUST inform the user and suggest splitting across sessions (e.g., "3 agents this session, 2 + synthesis next session").

#### FR3.3: Synthesis Protocol

Synthesis is performed by the orchestrator (main context), not a sub-agent. Requirements:

**Enforcement**:
- Orchestrator MUST read ALL 5 agent output files before writing synthesis
- If any output file is missing or empty, re-spawn that agent (max 1 retry)
- If agent fails twice, document the gap in synthesis under "Incomplete Coverage" section
- Do NOT write synthesis until all available outputs have been read

**Template**: Each skill has its own `templates/synthesis-output.md` with the target document structure. The orchestrator uses this template as the skeleton.

**Content**: The synthesis document structure captures:
- Key Findings (Convergent) — findings from 2+ agents
- Tensions and Trade-offs — where agents disagree
- Unique Insights — findings from a single agent
- Confidence Map — tabular mapping of findings to supporting agents
- Open Questions — unresolved items for user
- Implications — what this means for next steps

#### FR3.4: Error Handling

| Scenario | Action |
|----------|--------|
| Agent returns empty output | Re-spawn once. If still empty, document gap. |
| Agent returns truncated output | Accept as-is, note in diagnostics. |
| Agent fails to spawn | Re-spawn once. If still fails, reduce to 4 agents, document. |
| Token budget exceeded mid-session | Stop spawning, synthesize from available outputs, note incomplete. |
| User-provided document unreadable | AskUserQuestion for alternative source. |

#### FR3.5: Diagnostic Output

Both skills MUST write diagnostic output after every invocation.

Write to: `logs/diagnostics/{skill-name}-{YYYYMMDD-HHMMSS}.yaml`

```yaml
diagnostic:
  skill: bulwark-research  # or bulwark-brainstorm
  timestamp: 2026-02-15T12:00:00Z
  invocation:
    topic: "agent teams"
    source: prompt | document
    context_file: null | path
    research_file: null | path  # brainstorm only
  agents:
    total_spawned: 5
    successful: 5
    retried: 0
    failed: 0
    agents_list:
      - name: direct-investigation
        model: sonnet
        status: success
        output_file: logs/research/agent-teams/01-direct-investigation.md
  pre_flight_interview: skipped | completed
  post_synthesis_interview: completed
  synthesis_file: logs/research/agent-teams/synthesis.md
  token_checkpoints:
    pre_spawn: "25%"
    post_agents: "52%"
    post_synthesis: "61%"
```

## Technical Design

### Model Selection

| Phase | Agent | Subagent Type | Model | Rationale |
|-------|-------|--------------|-------|-----------|
| Research | 5 viewpoints | general-purpose | Sonnet | Research/analysis — Sonnet's sweet spot (OR2) |
| Research | Synthesis | Orchestrator | Opus (main) | Not a sub-agent |
| Brainstorm | Project SME | general-purpose | Opus | Needs codebase exploration + deep reasoning |
| Brainstorm | PM, Architect, Dev Lead | general-purpose | Opus | Prescriptive analysis, trade-off reasoning |
| Brainstorm | Critical Analyst | general-purpose | Opus | Heaviest reasoning: synthesizes 4 outputs |
| Brainstorm | Synthesis | Orchestrator | Opus (main) | Not a sub-agent |

### Skill Conventions (Aligned with code-review)

Both skills follow the code-review structural pattern:

| Section | Purpose |
|---------|---------|
| When to Use / When Not to Use | Trigger patterns and exclusions |
| Dependencies | Required files, optional files, when to load |
| Usage | Invocation syntax with arguments and flags |
| Stages | Numbered stages with names (like code-review's 3 phases) |
| Sections (per-agent) | Like code-review's Security/TypeSafety/etc — each agent is a section |
| Diagnostic Output | REQUIRED, writes YAML to logs/diagnostics/ |
| Completion Checklist | Checkboxes at end of skill |

### Stage Breakdown

**bulwark-research stages:**

```
Stage 1: Pre-Flight
├── Read problem statement / document
├── AskUserQuestion if ambiguous (2-3 questions max)
├── Create output directory: logs/research/{topic-slug}/
├── Load subagent-prompting skill
└── Token budget check

Stage 2: Viewpoint Analysis (5 Sonnet, parallel)
├── Load references/viewpoint-*.md for each agent
├── Construct 5 prompts using 4-part template
├── Include templates/viewpoint-output.md in each prompt
├── Spawn all 5 agents in parallel via Task tool
└── Token budget check after all complete

Stage 3: Synthesis
├── Read ALL 5 agent outputs (MANDATORY — do not skip any)
├── If any missing → retry once (FR3.4)
├── Write synthesis using templates/synthesis-output.md
├── AskUserQuestion for user on open questions
└── Token budget check

Stage 4: Diagnostics (REQUIRED)
├── Write diagnostic YAML to logs/diagnostics/
└── Completion checklist verification
```

**bulwark-brainstorm stages:**

```
Stage 1: Pre-Flight
├── Read problem statement / document
├── Load research synthesis if --research provided
├── AskUserQuestion if ambiguous (2-3 questions max)
├── Create output directory: logs/brainstorm/{topic-slug}/
├── Load subagent-prompting skill
└── Token budget check

Stage 2: Project SME (Opus, sequential — first)
├── Load references/role-project-sme.md
├── Construct prompt with problem statement + research synthesis
├── Spawn general-purpose Opus agent with codebase exploration tools
├── Read SME output
└── Token budget check

Stage 3: Role Analysis (3 Opus, parallel)
├── Load references/role-{pm,architect,dev-lead}.md
├── Construct 3 prompts with problem statement + SME output
├── Include templates/role-output.md in each prompt
├── Spawn all 3 agents in parallel via Task tool
└── Token budget check

Stage 4: Critical Analyst (Opus, sequential — last)
├── Load references/role-critical-analyst.md
├── Construct prompt with problem statement + ALL prior outputs (SME + PM + Architect + Dev Lead)
├── Include templates/critic-output.md in prompt
├── Spawn agent
└── Token budget check

Stage 5: Synthesis
├── Read ALL 5 agent outputs (MANDATORY — do not skip any)
├── If any missing → retry once (FR3.4)
├── Write synthesis using templates/synthesis-output.md
├── AskUserQuestion for user on open questions
└── Token budget check

Stage 6: Diagnostics (REQUIRED)
├── Write diagnostic YAML to logs/diagnostics/
└── Completion checklist verification
```

### Output Templates (Not subagent-output-templating)

These skills use custom output templates because their sub-agents produce documentation, not code changes. The templates live inside each skill's `templates/` directory.

**viewpoint-output.md** (research agents):

```markdown
---
viewpoint: {viewpoint-name}
topic: {topic}
confidence_summary:
  high: {count}
  medium: {count}
  low: {count}
key_findings:
  - {finding 1}
  - {finding 2}
  - {finding 3}
---

# {Topic} — {Viewpoint Name}

## Summary
{2-3 sentence overview of findings from this viewpoint}

## Detailed Analysis

### {Focus Area 1}
{Analysis with evidence and confidence levels}

### {Focus Area 2}
{Analysis with evidence and confidence levels}

...

## Confidence Notes
{Explain any LOW confidence findings and what would increase confidence}
```

**role-output.md** (brainstorm agents — PM, Architect, Dev Lead):

```markdown
---
role: {role-name}
topic: {topic}
recommendation: proceed | modify | defer | kill
key_findings:
  - {finding 1}
  - {finding 2}
  - {finding 3}
---

# {Topic} — {Role Name}

## Summary
{2-3 sentence overview from this role's perspective}

## Detailed Analysis

### {Focus Area 1}
{Prescriptive analysis — "Do X", not "Consider X or Y"}

### {Focus Area 2}
{Prescriptive analysis}

...

## Recommendation
{Clear recommendation with justification}
```

**critic-output.md** (Critical Analyst — has verdict section):

```markdown
---
role: critical-analyst
topic: {topic}
verdict: proceed | modify | defer | kill
verdict_confidence: high | medium | low
conditions:
  - {condition for verdict to hold}
key_challenges:
  - {challenge 1}
  - {challenge 2}
---

# {Topic} — Critical Analysis

## Cost-Benefit Assessment
{Is the investment justified?}

## Assumption Challenges
{What are we assuming that might be wrong?}

## Gaps in Proposals
{What has been overlooked?}

## Simpler Alternatives
{Could less ambitious approaches work?}

## Kill Criteria
{Under what conditions should this be abandoned?}

## Verdict
{proceed / modify / defer / kill — with specific conditions}
```
