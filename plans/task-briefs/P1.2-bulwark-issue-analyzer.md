## 1. Overview

### 1.1 Purpose

Create a custom sub-agent that analyzes issues to identify root cause, map impact, and produce a debug report with tiered validation plan. This agent is a **wrapper** around the `issue-debugging` skill, providing both:

1. **User-invocable entry point** - Users can directly invoke `/bulwark-issue-analyzer [path]` for ad-hoc issue analysis
2. **Pipeline stage** - Orchestrator spawns as Stage 1 in Fix Validation pipeline

### 1.2 Issue Types Supported

The agent handles issues in **both production code and test code**:

| Issue Type | Example | Root Cause Location |
|------------|---------|---------------------|
| **Production bugs** | "Login fails with 500 error" | Production code |
| **Test failures** | "Tests are failing", "CI broken" | Could be test code or production code |
| **Test code bugs** | "Flaky test", "Test has wrong assertion" | Test code itself |
| **Infrastructure** | "Build fails", "Migration didn't run" | Config, scripts, environment |

The methodology (5 Whys, hypothesis-driven) works regardless of where the bug resides.

### 1.3 Pipeline Context

```fsharp
// Fix Validation Pipeline - Stage 1
IssueAnalyzer (bulwark-issue-analyzer, produces debug_report)  // <-- THIS AGENT
|> FixWriter (orchestrator implements fix)
|> TestWriter (orchestrator writes tests)
|> FixValidator (bulwark-fix-validator, validates against debug_report)
|> CodeReviewer (reviews all, approves/rejects)
|> LOOP(max=3)
```

### 1.4 Why a Wrapper Agent?

| Reason | Explanation |
|--------|-------------|
| **Single-stage operation** | No internal sub-agent spawning needed (unlike test-audit) |
| **Isolated context** | Bias avoidance - analyzes issue without having generated the code |
| **Encapsulated behavior** | Agent definition specifies skills, tools, output format |
| **Clean interface** | Orchestrator invokes via `Task(subagent_type="bulwark-issue-analyzer")` |

### 2.2 Argument Handling

**Note**: `$ARGUMENTS` support for custom sub-agents is **to be validated**. It works for skills but may not work for agents invoked via Task tool. If it doesn't work, the agent should read the path from CONTEXT instead.

When invoked via slash command, the agent receives arguments:

| Invocation | `$ARGUMENTS` / `$1` Value |
|------------|---------------------------|
| `/bulwark-issue-analyzer tests/fixtures/bug-app/` | `tests/fixtures/bug-app/` (if supported) |
| `/bulwark-issue-analyzer` (no arg) | Empty - agent should ask or infer from context |
| Pipeline invocation | Issue details passed in CONTEXT (always works) |

**Argument processing**:
- If path provided via `$ARGUMENTS`: Start investigation at that path
- If `$ARGUMENTS` not available: Read path from CONTEXT or prompt user
- Path can be file (specific bug location) or directory (general area to investigate)

**Fallback**: If `$ARGUMENTS` doesn't work for custom sub-agents, the orchestrator passes the path in the CONTEXT section of the 4-part prompt.

### 2.4 Tool Usage Constraints (Instruction-Based)

```markdown
## Tool Usage Constraints

### Write
- **Allowed**: `logs/debug-reports/`, `logs/diagnostics/`
- **Forbidden**: Source files, test files, config files, any file outside `logs/`

### Bash
- **Allowed**:
  - Read-only git commands (`git log`, `git blame`, `git diff`, `git show`)
  - Test execution for reproduction (`just test`, `npm test`, etc.)
  - File inspection (`ls`, `wc`, `file`)
  - Process inspection (`ps`, `lsof` for port checks)
- **Forbidden**:
  - Destructive commands (`rm`, `rmdir`, `mv`, `cp` to overwrite)
  - File modification (`sed -i`, `awk` with output redirect, `truncate`)
  - Git modifications (`git commit`, `git push`, `git reset`, `git checkout`)
  - Package installation (`npm install`, `pip install`)

### General
- **NEVER** modify source code, test files, or config files
- Analysis only - fixes are done by the orchestrator in subsequent pipeline stages
```

## 3. Output Specification

### 3.1 Debug Report

**Location**: `logs/debug-reports/{issue-id}-{YYYYMMDD-HHMMSS}.yaml`

**Schema** (from fix-validation.md):

```yaml
debug_report:
  metadata:
    issue_id: "{from CONTEXT or generated}"
    timestamp: "{ISO-8601}"
    analyzer: bulwark-issue-analyzer

  analysis:
    symptom: "{observable problem}"
    root_cause: "{underlying reason}"
    complexity: low | medium | high
    fix_approach: "{recommended fix direction}"

  impact_analysis:
    affected_files:
      - "{path}"
    upstream_dependencies:
      - "{what calls the affected code}"
    downstream_effects:
      - "{what the affected code impacts}"
    risk_scope: isolated | medium | broad

  validation_plan:
    tests_to_execute:
      - path: "{test file}"
        reason: "{why this test}"
        priority: 1  # P1=must, P2=should, P3=nice-to-have
    functionalities_to_validate:
      - "{user-visible functionality to verify}"

  confidence_criteria:
    high:
      - "{conditions for high confidence}"
    medium:
      - "{conditions for medium confidence}"
    low:
      - "{conditions for low confidence}"

  debug_journey:  # Required for medium/high complexity
    hypotheses_tested:
      - hypothesis: "{what was suspected}"
        result: confirmed | rejected
        evidence: "{supporting evidence}"
```

### 3.2 Diagnostics

**Location**: `logs/diagnostics/bulwark-issue-analyzer-{YYYYMMDD-HHMMSS}.yaml`

**Schema** (from subagent-output-templating):

## 4. Agent Content Structure

```markdown
# Bulwark Issue Analyzer

## Identity

You are an issue analysis specialist in the Bulwark quality system. Your role is to investigate bugs and issues to understand their root cause, map their impact, and produce a debug report that guides the fix implementation.

## Mission

**DO**:
- Analyze issues to identify root cause (not just symptoms)
- Map impact across upstream and downstream dependencies
- Produce structured debug report with tiered validation plan
- Document your debug journey (hypotheses tested, evidence gathered)
- Reproduce issues via test execution when possible

**DO NOT**:
- Modify any source code, test files, or config files
- Implement fixes (that's the orchestrator's job)
- Skip the validation plan (FixValidator depends on it)
- Write to any location outside `logs/`

## Arguments

This agent accepts an optional path argument:

- **With path** (`/bulwark-issue-analyzer path/to/code`): Start investigation at specified path
- **Without path**: Check CONTEXT for issue details, or ask user for issue description

## Protocol

### Step 0: Parse Arguments

1. Check if `$ARGUMENTS` contains a path
2. If path provided: Use as starting point for investigation
3. If no path: Look for issue details in CONTEXT, or prompt user

### Step 1: Understand the Issue

Read the issue description from CONTEXT or user prompt. Identify:
- Observable symptom
- Error messages / stack traces
- Reproduction steps (if available)

### Step 2: Investigate Root Cause

1. Locate relevant code using Grep/Glob
2. Read affected files
3. Trace execution path
4. Form hypotheses and test them
5. Use Bash for git history, test reproduction

### Step 3: Map Impact

Identify:
- Affected files (direct)
- Upstream dependencies (what calls this code)
- Downstream effects (what this code impacts)
- Risk scope (isolated / medium / broad)

### Step 4: Create Validation Plan

Tier tests by priority:
- **P1 (must)**: Direct tests of affected functionality
- **P2 (should)**: Integration tests of upstream callers
- **P3 (nice-to-have)**: E2E tests, edge cases

List functionalities that need manual validation if tests can't cover.

### Step 5: Define Confidence Criteria

Specify what constitutes high/medium/low confidence that the fix worked.

### Step 6: Write Outputs

1. Write debug report to `logs/debug-reports/{issue-id}-{YYYYMMDD-HHMMSS}.yaml`
2. Write diagnostics to `logs/diagnostics/bulwark-issue-analyzer-{YYYYMMDD-HHMMSS}.yaml`
3. Return summary to orchestrator (include debug report path)

## Tool Usage Constraints

[As specified in Section 2.3]

## Output Format

[Reference subagent-output-templating skill for YAML schema]

## Related Skills

- `issue-debugging` - Core methodology (loaded via frontmatter)
- `subagent-output-templating` - Output format (loaded via frontmatter)
```

### 6.3 Test Fixtures

#### Directory Structure

```
tests/fixtures/
├── issue-analyzer/              # Code with bugs (passed to agent)
│   ├── production-bug/
│   │   ├── src/
│   │   │   └── auth.ts          # Production code with bug
│   │   └── tests/
│   │       └── auth.test.ts     # Passing tests (bug not caught)
│   │
│   └── test-bug/
│       ├── src/
│       │   └── parser.ts        # Working production code
│       └── tests/
│           └── parser.test.ts   # Test with flaky assertion
│
└── issue-analyzer-expected/     # Expected results (NOT passed to agent)
    ├── production-bug.yaml      # Expected debug report
    └── test-bug.yaml            # Expected debug report
```

#### Critical: No Bias in Fixtures

**IMPORTANT**: Test fixture files must contain **ONLY the code pattern** - NO comments explaining what the bug is or where it is located.

| Correct | Incorrect |
|---------|-----------|
| `if (user.profile) { ... }` | `// BUG: Missing null check here` |
| Natural code that happens to have a bug | Comments explaining the violation |
| Realistic production/test code | Hints about expected outcome |

This ensures:
- Agent detects issues through actual analysis
- No confirmation bias from reading explanations
- Realistic simulation of production debugging

Expected results are stored in `issue-analyzer-expected/` which is:
- **NEVER passed to the agent**
- Used only for human validation after analysis completes
- Contains expected debug report fields (root cause, complexity, etc.)

#### Fixture Descriptions (for tester reference only)

**Production Bug** (`production-bug/`):
- `auth.ts`: Has null pointer issue when user has no profile
- `auth.test.ts`: Tests pass because they don't cover the null case
- Expected: Root cause = null check missing, complexity = low/medium

**Test Bug** (`test-bug/`):
- `parser.ts`: Working production code
- `parser.test.ts`: Has flaky timing-dependent assertion
- Expected: Root cause = test code issue (race condition), complexity = medium
