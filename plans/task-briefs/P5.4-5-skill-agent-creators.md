# P5.4/P5.5: Skill-Creator and Agent-Creator Skills

## Context

Session 64 research (5 Sonnet agents, `/bulwark-research`) produced comprehensive findings on meta-skill design patterns. The research covers skill complexity classification, context fork decisions, sub-agent architecture selection, and meta-skill generation workflows. The original bundled brief with P5.3 was split in Session 63 — P5.3 has its own standalone brief.

### Research Decisions (Session 64)

| Decision | Rationale |
|----------|-----------|
| Adopt and adapt Anthropic's skill-creator | Official skill-creator provides foundation (6-step process, degrees-of-freedom framework). Bulwark adds SA1-SA6 compliance, multi-agent pattern generation, anthropic-validator, content guidance. |
| Adaptive interview depth | 5 core questions + 3-5 follow-up for complex skills. Simple skills stay fast; complex skills get thorough classification. Aligns with research finding: "the interview IS the classification." |
| 5 skill-type templates (full taxonomy) | Simple, Reference-heavy, Pipeline, Script-driven, Research/Brainstorm. Covers full Bulwark corpus. DSL narrowness lesson: specialize per type. |
| Structure + content guidance | Include content scaffolding (good/bad examples, description patterns, pitfalls). Addresses Contrarian finding that structure-only automation is low-value. |
| Each skill has its own references/ | Per Anthropic conventions. Shared decision-framework content duplicated in each skill's `references/decision-framework.md`. |
| Post-generation summary (not inline comments) | After generating, present architectural decisions to user. Generated files stay clean. |
| General-purpose | Both skills usable by any Claude Code user. Sync to standalone repo. |
| Generate-and-customize contract | Output is explicitly a scaffold, not production-ready. Research convergence across 3 viewpoints. |

## P5.4: Skill-Creator

### Functional Requirements

#### FR1: Skill Structure

```
skills/skill-creator/
├── SKILL.md                              (main skill document)
├── references/
│   ├── decision-framework.md             (shared interview + classification logic)
│   ├── template-simple.md                (vanilla SKILL.md template)
│   ├── template-reference-heavy.md       (SKILL.md + references/ template)
│   ├── template-pipeline.md              (multi-stage orchestration template)
│   ├── template-script-driven.md         (SKILL.md + scripts/ template)
│   ├── template-research.md              (parallel/sequential multi-agent template)
│   └── content-guidance.md               (instruction writing patterns, description rules, pitfalls)
└── templates/
    └── diagnostic-output.yaml            (standard diagnostic schema)
```

#### FR2: Invocation

```
/skill-creator <description-or-name>
/skill-creator --doc <requirements-document>
```

**Arguments:**
- `<description-or-name>` — Free-text description of desired skill or skill name
- `--doc <path>` — Requirements document to use as input instead of free text

#### FR3: Pipeline Topology

```fsharp
// skill-creator pipeline
PreFlight(args)                              // Stage 0: Orchestrator — parse input, adaptive interview
|> Classify(interview_answers)               // Stage 1: Orchestrator — three independent decisions
|> Generate(classification, template, examples) // Stage 2: Sonnet sub-agent — produce skill files
|> Validate(generated_output)                // Stage 3: Orchestrator — run anthropic-validator
|> Refine(validator_findings)                // Stage 4: Sonnet sub-agent (conditional, max 2 retries)
|> Present(final_output, decisions_summary)  // Stage 5: Orchestrator — scaffold + post-generation summary
```

#### FR4: Stage Definitions

##### Stage 0: Pre-Flight (Orchestrator)

```
Stage 0: Pre-Flight
├── Parse arguments (description or --doc)
├── Load references/decision-framework.md
├── Load references/content-guidance.md
├── Load subagent-prompting skill
├── AskUserQuestion: 5 core questions
│   ├── Q1: What does this skill do? (concrete examples of invocation)
│   ├── Q2: Does it need conversation history, or can it run in isolation?
│   ├── Q3: Does it orchestrate multiple distinct operations or analyses?
│   ├── Q4: How much reference content does it need (domain-specific knowledge)?
│   ├── Q5: Does it produce structured output that must match a specific format?
│   └── If complexity detected: 3-5 follow-up questions
│       ├── Q6: Do the operations depend on each other's output?
│       ├── Q7: Do workers need to communicate with each other?
│       ├── Q8: What error handling patterns are needed?
│       └── Q9-Q10: Context-specific follow-ups based on Q1-Q5 answers
├── Create output directory for generated skill
└── Token budget check (warn if >30% consumed)
```

##### Stage 1: Classify (Orchestrator)

Three independent decisions from interview answers:

**Decision A — Context:**
```
Needs conversation history? → inline (no fork)
Multi-step, noise-prone, isolated task? → context: fork
Simple guideline/knowledge layer? → inline (no fork, warn if fork requested)
```

**Decision B — Sub-agent pattern:**
```
Single-step or latency-sensitive? → no sub-agents
Multiple dependent passes? → sequential Task tool sub-agents
Multiple independent analyses? → parallel Task tool sub-agents
Workers must communicate directly? → Agent Teams (with experimental warning)
```

**Decision C — Supporting files:**
```
Content < 500 lines, no domain-specific references? → vanilla SKILL.md
Domain-specific reference data? → add references/
Structured output format required? → add templates/
Deterministic code execution needed? → add scripts/
```

**Output:** Classified skill type (1 of 5 templates) + architectural decisions.

##### Stage 2: Generate (Sonnet sub-agent)

```
Stage 2: Generate
├── Construct prompt using 4-part template (GOAL/CONSTRAINTS/CONTEXT/OUTPUT)
│   ├── GOAL: Generate a complete skill matching the classification
│   ├── CONSTRAINTS: Follow Anthropic skill conventions, single-line description,
│   │   "when to use" trigger framing, no unnecessary files
│   ├── CONTEXT: Classification from Stage 1, selected template from references/,
│   │   content-guidance.md, user's concrete examples from interview,
│   │   existing skill examples from codebase (agent reads autonomously)
│   └── OUTPUT: Generated files written to target skill directory
├── Spawn general-purpose Sonnet agent
│   ├── Agent reads the appropriate template-*.md reference
│   ├── Agent reads content-guidance.md for instruction writing patterns
│   ├── Agent reads 1-2 existing skills of the same type for structural reference
│   ├── Agent generates SKILL.md + supporting files per classification
│   └── Agent ensures description is single-line, trigger-specific
└── Read generated output
```

##### Stage 3: Validate (Orchestrator)

```
Stage 3: Validate
├── Run /anthropic-validator on generated skill
├── Check for critical/high findings
│   ├── If 0 critical, 0 high → proceed to Stage 5
│   └── If findings → proceed to Stage 4 (refine)
├── Check description is single-line (YAML frontmatter validation)
└── Check no unnecessary files generated (no README, CHANGELOG, etc.)
```

##### Stage 4: Refine (Sonnet sub-agent, conditional)

```
Stage 4: Refine (max 2 retries)
├── Construct prompt with validator findings in CONTEXT
├── Spawn general-purpose Sonnet agent
│   ├── Agent reads validator findings
│   ├── Agent reads current generated files
│   └── Agent fixes issues identified by validator
├── Re-run validation (Stage 3)
└── If still failing after 2 retries → present with caveats
```

##### Stage 5: Present (Orchestrator)

```
Stage 5: Present
├── Post-generation summary to user:
│   ├── "Generated skill at: {path}"
│   ├── "Architectural decisions:"
│   │   ├── "Context: {fork/inline} because {reason}"
│   │   ├── "Sub-agents: {none/sequential/parallel/Agent Teams} because {reason}"
│   │   └── "Supporting files: {list} because {reason}"
│   ├── "Skill type: {template used}"
│   ├── "Validation: {pass/fail with details}"
│   └── "Next steps: customize instructions, test activation, iterate"
├── Write diagnostic YAML
└── Completion checklist
```

##### Stage 6: Diagnostics (REQUIRED)

Write to `logs/diagnostics/skill-creator-{YYYYMMDD-HHMMSS}.yaml`

#### FR5: Content Guidance

The `references/content-guidance.md` file must include:

1. **Description field rules**: Single-line, "when to use" framing, trigger-specific language, avoid vague descriptions
2. **Instruction writing patterns**: Good/bad examples of skill instructions. What makes Claude follow instructions vs. ignore them (DEF-P4-005 binding language pattern)
3. **Common pitfalls**: Multi-line YAML breaks discovery, guidelines-only skills become no-ops with fork, over-elaborate output, emoji-heavy content
4. **Activation tuning**: How to improve activation rates (detailed triggers, explicit invocation patterns)

#### FR6: Error Handling

| Scenario | Action |
|----------|--------|
| Generator sub-agent returns empty output | Re-spawn once. If still empty, STOP. |
| anthropic-validator finds critical issues | Stage 4 retry (max 2). |
| Interview answers are ambiguous | Ask follow-up questions (adaptive depth). |
| User requests Agent Teams | Include experimental warning + env var gating. |
| Token budget exceeded | Stop, present partial output with explanation. |

## P5.5: Agent-Creator

### Functional Requirements

#### FR1: Skill Structure

```
skills/agent-creator/
├── SKILL.md                              (main skill document)
├── references/
│   ├── decision-framework.md             (shared interview + classification logic)
│   ├── agent-conventions.md              (system-prompt register, Stop hooks, permissions)
│   ├── template-single-agent.md          (single-purpose agent template)
│   ├── template-pipeline-agent.md        (multi-stage orchestrator agent template)
│   ├── template-teams-agent.md           (Agent Teams lead + teammates template)
│   └── content-guidance.md               (same as skill-creator, plus agent-specific guidance)
└── templates/
    └── diagnostic-output.yaml            (standard diagnostic schema)
```

#### FR2: Invocation

```
/agent-creator <description-or-name>
/agent-creator --doc <requirements-document>
```

#### FR3: Pipeline Topology

Same as skill-creator (Stage 0-6) with agent-specific classification and generation logic.

#### FR4: Agent-Specific Stage Modifications

##### Stage 0 Additions

Additional interview questions for agents:
- Q-A1: What is this agent's identity/mission? (system-prompt content)
- Q-A2: What tools does it need access to?
- Q-A3: Does it need to emit diagnostic output?
- Q-A4: Should it have restricted permissions (permissionMode)?

##### Stage 1 Agent-Specific Classification

**Decision A — Context (always fork for agents):**
Agents in `.claude/agents/` inherently fork. No decision needed — this is the format.

**Decision B — Agent architecture:**
```
Single focused task? → single-agent (template-single-agent.md)
Multi-step workflow with stages? → pipeline orchestrator (template-pipeline-agent.md)
Workers need inter-agent communication? → Agent Teams (template-teams-agent.md, experimental)
```

**Decision C — Supporting configuration:**
```
Always generate: Stop hook for diagnostic output
If sub-agents used: SA1 4-part template in agent body, subagent-prompting in skills: dependency
If restricted: permissionMode in frontmatter, tools: list
```

##### Stage 2 Agent-Specific Generation

The generator sub-agent must:
1. Write in the **system-prompt register** (WHO the agent IS, not WHAT to DO)
2. Include Stop hook in frontmatter for diagnostic output
3. Include `skills:` dependencies (at minimum: `subagent-prompting`)
4. Include a "Permissions Setup" section documenting what the user must configure in `.claude/settings.json` (no automated solution per GitHub #10093)
5. Include GOAL/CONSTRAINTS/CONTEXT/OUTPUT template in the agent's protocol section
6. Place output in `.claude/agents/` (not `skills/`)

##### Stage 5 Agent-Specific Presentation

Post-generation summary includes:
- Agent file location (`.claude/agents/{name}.md`)
- Architecture decision (single/pipeline/teams)
- Stop hook command generated
- Permissions the user must configure manually
- How to invoke: `Task(subagent_type="{name}")` or via `/agents` menu

#### FR5: System-Prompt vs. Task-Instruction Register

This is the critical distinction identified in research. The agent body must read as identity and mission:

**Good (system-prompt register):**
```markdown
You are a code security reviewer. Your mission is to identify vulnerabilities
in code changes. You focus on OWASP top 10, injection attacks, and auth flaws.

When reviewing code, you:
1. Read all changed files
2. Classify each change by risk level
3. Report findings in structured format
```

**Bad (task-instruction register):**
```markdown
## Steps
1. Read the files provided
2. Look for security issues
3. Write a report

## Output
Write findings to logs/security-review.md
```

The first reads as WHO THE AGENT IS. The second reads as WHAT TO DO (which belongs in skill instructions, not agent definitions).

## Technical Design (Shared)

### Model Selection

| Stage | Agent | Model | Rationale |
|-------|-------|-------|-----------|
| 0 (Pre-Flight) | Orchestrator | — | Interview, classification — direct |
| 1 (Classify) | Orchestrator | — | Decision tree — direct |
| 2 (Generate) | Generator | Sonnet | Analysis/generation task |
| 3 (Validate) | Orchestrator | — | anthropic-validator invocation |
| 4 (Refine) | Refiner | Sonnet | Fix validation issues |
| 5 (Present) | Orchestrator | — | Summary — direct |

## Implementation Plan

### Build Order

**Step 1: Shared References (~15% of session)**
- Create `references/decision-framework.md` — interview questions, three-decision classification logic, skill-type mapping
- Create `references/content-guidance.md` — description rules, instruction writing patterns, common pitfalls, activation tuning guidance

**Step 2: Skill-Creator Templates (~20% of session)**
- Create `references/template-simple.md` — vanilla SKILL.md structure
- Create `references/template-reference-heavy.md` — with references/ directory
- Create `references/template-pipeline.md` — multi-stage with references/ + templates/
- Create `references/template-script-driven.md` — with scripts/ directory
- Create `references/template-research.md` — parallel/sequential multi-agent

**Step 3: Agent-Creator References (~10% of session)**
- Create `references/agent-conventions.md` — system-prompt register, Stop hooks, permissions setup, SA1 compliance
- Create `references/template-single-agent.md`
- Create `references/template-pipeline-agent.md`
- Create `references/template-teams-agent.md`

**Step 4: SKILL.md Files (~20% of session)**
- Create `skills/skill-creator/SKILL.md` — Stage 0-6, interview logic, classification, error handling
- Create `skills/agent-creator/SKILL.md` — Stage 0-6, agent-specific modifications

**Step 5: Sync and Validate (~10% of session)**
- Copy to `.claude/skills/skill-creator/` and `.claude/skills/agent-creator/` (dogfood)
- Run `/anthropic-validator` on both skills
- Update `scripts/sync-essential-skills.sh` to include both skills

### Effort Estimation

**Total: 1-2 sessions (estimated 40-60K tokens)**

The shared references reduce duplication. The 5 templates are the bulk of the authoring effort. If token-constrained, split: Session A = shared refs + skill-creator, Session B = agent-creator + validation.

## Risks and Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| Generated output quality too low (30-50% rework) | HIGH | Content guidance, good/bad examples, anthropic-validator gate. Explicit scaffold contract. |
| Decision tree classifies incorrectly | MEDIUM | Conversational elicitation (ask user, don't infer). Three decisions are independent — partial correctness still valuable. |
| LLM non-determinism produces inconsistent output | MEDIUM | Validate-and-retry loop. Templates constrain structural variance. |
| 5 templates = significant authoring effort | MEDIUM | Templates are reference docs (50-100 lines each), not implementations. Shared content-guidance reduces duplication. |
| Agent Teams template becomes stale (experimental API) | LOW | Template includes experimental warning. Updated when Agent Teams stabilizes. |
| Anthropic skill-creator diverges from our fork | LOW | Periodic sync via continuous-feedback skill (P5.3). |
