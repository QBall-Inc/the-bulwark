## 1. Functional Requirements

### 1.1 Core Purpose

Provide pre-defined F# pipe workflows for common multi-agent scenarios with **deterministic invocation** and **hard validation**. This skill ensures that pipeline orchestration follows consistent patterns.

**Key Characteristics**:
- Loaded deterministically via global PreToolUse hook on Task
- Validates pipeline patterns (hard enforcement - can deny)
- Tracks pipeline progress via SubagentStart/SubagentStop
- Single-agent tasks allowed silently (no validation needed)

### 1.2 Content Requirements

| Requirement | Description |
|-------------|-------------|
| FR-1 | Code Review Pipeline defined |
| FR-2 | Fix Validation Pipeline defined |
| FR-3 | Test Audit Pipeline defined |
| FR-4 | New Feature Pipeline defined |
| FR-5 | Validation rules for pipeline correctness |
| FR-6 | Model selection guidance (reference P0.1 + extend) |
| FR-7 | Single-agent task bypass logic |

### 1.3 User Stories

- As the **Orchestrator**, I need pipeline templates to ensure consistent multi-agent workflows
- As a **Quality Gate**, I need to block incorrect pipeline usage before execution
- As a **Developer**, I need clear guidance on which pipeline to use for which task

### 2.2 Frontmatter Specification

```yaml
---
name: pipeline-templates
description: Pre-defined F# pipe workflows for multi-agent orchestration. Provides code review, fix validation, test audit, and new feature pipelines with hard validation. Loaded automatically before Task invocations.
user-invocable: false
---
```

**Field Rationale**:
- `user-invocable: false`: Internal skill, loaded by hooks
- No `agent:` field: Knowledge skill, runs in main context
- No `context: fork`: Shares main context for orchestration
- No `hooks:` in skill: Hooks are global (see Section 2.4)

### 2.3 Hook Configuration (Development)

Hooks are configured in `.claude/settings.json` for development and testing.
Plugin deployment (P3.4) will migrate these to `hooks/hooks.json`.

**Why not skill-scoped hooks?** Skill-scoped hooks only fire when the skill is already loaded. We need global hooks to ensure deterministic loading BEFORE Task invocation.

**Development configuration (`.claude/settings.json`):**

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Task",
        "command": "${CLAUDE_PROJECT_DIR}/scripts/hooks/validate-pipeline.sh"
      }
    ],
    "SubagentStart": [
      {
        "command": "${CLAUDE_PROJECT_DIR}/scripts/hooks/track-pipeline-start.sh"
      }
    ],
    "SubagentStop": [
      {
        "command": "${CLAUDE_PROJECT_DIR}/scripts/hooks/track-pipeline-stop.sh"
      }
    ]
  }
}
```

### 2.4 Hook Scripts

#### 2.4.1 validate-pipeline.sh (PreToolUse)

```bash
#!/bin/bash
# Validates Task tool usage against pipeline patterns
# Input: JSON with tool_name, tool_input
# Output: JSON with permissionDecision, reason, systemMessage

# Read input from stdin
INPUT=$(cat)

# Parse task details
SUBAGENT_TYPE=$(echo "$INPUT" | jq -r '.tool_input.subagent_type // "unknown"')
DESCRIPTION=$(echo "$INPUT" | jq -r '.tool_input.description // ""')

# Check if this is a simple single-agent task (allow silently)
# Simple tasks: exploration, quick lookups, file reads
SIMPLE_PATTERNS="explore|search|find|lookup|read|fetch"
if echo "$DESCRIPTION" | grep -qiE "$SIMPLE_PATTERNS"; then
  echo '{"permissionDecision": "allow"}'
  exit 0
fi

# For complex multi-agent work, inject pipeline guidance
cat << 'EOF'
{
  "permissionDecision": "allow",
  "systemMessage": "Pipeline-templates skill loaded. Before invoking sub-agents for complex work, ensure you are following an appropriate pipeline pattern. Reference: skills/pipeline-templates/SKILL.md"
}
EOF
```

#### 2.4.2 track-pipeline-start.sh (SubagentStart)

```bash
#!/bin/bash
# Logs pipeline stage start
# Input: JSON with session_id, agent_id, subagent_type, agent_transcript_path

INPUT=$(cat)
TIMESTAMP=$(date -u +"%Y-%m-%dT%H:%M:%SZ")
AGENT_ID=$(echo "$INPUT" | jq -r '.agent_id // "unknown"')
SUBAGENT_TYPE=$(echo "$INPUT" | jq -r '.subagent_type // "unknown"')

# Log to pipeline tracking file
echo "[$TIMESTAMP] SubagentStart: $AGENT_ID ($SUBAGENT_TYPE)" >> "${CLAUDE_PROJECT_DIR}/logs/pipeline-tracking.log"
```

#### 2.4.3 track-pipeline-stop.sh (SubagentStop)

```bash
#!/bin/bash
# Logs pipeline stage completion and verifies output
# Input: JSON with stop event details

INPUT=$(cat)
TIMESTAMP=$(date -u +"%Y-%m-%dT%H:%M:%SZ")

# Log completion
echo "[$TIMESTAMP] SubagentStop: $(echo "$INPUT" | jq -r '.agent_id // "unknown"')" >> "${CLAUDE_PROJECT_DIR}/logs/pipeline-tracking.log"
```

### 3.3 Pipeline Reference Files

Each pipeline is defined in its own reference file for progressive disclosure.

| File | Pipeline | Purpose | Key Pattern |
|------|----------|---------|-------------|
| `code-review.md` | Code Review | Review code before merge | Audit → Audit → conditional Fix |
| `fix-validation.md` | Fix Validation | Fix issues and validate | Analyze → Fix → Review → loop |
| `test-audit.md` | Test Audit | Audit test quality | Classify → conditional Rewrite |
| `new-feature.md` | New Feature | Implement new functionality | Research → Write → Test → Review |
| `research-planning.md` | Research & Planning | Research and plan before work | Lookup → Synthesize → Review → loop(min=3) |
| `test-execution-fix.md` | Test Execution & Fix | Run tests, fix failures | Execute → Analyze → Fix → loop |

#### 3.3.1 Research & Planning Pipeline (NEW)

```fsharp
// Purpose: Research and plan before implementation
// Trigger: Complex task requiring planning
// Constraint: Minimum 3 iterations

Researcher (web search, file search)     // Haiku - lookups
|> Orchestrator (synthesize plan draft)  // Main thread
|> PlanReviewer (critical review)        // Sonnet - analysis
|> Orchestrator (apply feedback)         // Main thread
|> LOOP(min=3)
```

**Key Constraints**:
- Minimum 3 iterations to ensure plan quality
- Reviewer checks against requirements/PRD/conversation context
- Orchestrator synthesizes, sub-agents research and review

#### 3.3.2 Test Execution & Fix Pipeline (NEW)

```fsharp
// Purpose: Run tests, analyze failures, fix
// Trigger: After code changes, CI failures
// Constraint: Max iterations to prevent infinite loops

TestRunner (execute tests)              // Haiku - execution
|> (if failures > 0
    then FailureAnalyzer                // Sonnet - analyze
    else Done)
|> FixWriter (apply fixes)              // Opus - write fixes
|> TestRunner (re-execute)              // Haiku - verify
|> (if failures > 0 && iterations < max
    then FailureAnalyzer                // Loop
    else Done)
```

### 3.4 Model Selection

**Reference P0.1 (subagent-prompting)** for task-type based model selection:
- **Haiku**: Lookups & Execution (web fetch, run tests, lint)
- **Sonnet**: Review & Analysis (code review, audits, failure analysis)
- **Opus**: Write & Fix (write code, write tests, apply fixes)

### 3.5 Validation Rules

**Valid Pipeline Invocation**:
1. ✅ Uses defined pipeline template
2. ✅ Specifies model for each stage
3. ✅ Includes 4-part prompt for each Task
4. ✅ Reads previous stage output before next stage

**Invalid (will warn/block)**:
1. ❌ Ad-hoc multi-agent with no pipeline
2. ❌ Missing model specification
3. ❌ Skipping stages without justification
4. ❌ Using Opus for simple tasks

### 3.5 Single-Agent Task Bypass

Tasks matching these patterns bypass pipeline validation:
- `explore`, `search`, `find` - Quick lookups
- `read`, `fetch` - File/web reads
- `lookup`, `check` - Simple queries

These are allowed silently without loading pipeline-templates.

## 12. Session 3 Design Revisions

**Date**: 2026-01-12
**Reason**: Research revealed PreToolUse hook limitations

### 12.1 Problem Identified

Original design used PreToolUse on Task to validate/inject pipeline guidance. Research found:
1. `systemMessage` field does not exist in hook output
2. PreToolUse `reason` with `"approve"` goes to user, not Claude
3. Cannot inject guidance that Claude acts on via PreToolUse

### 12.2 Revised Design

| Aspect | Original | Revised |
|--------|----------|---------|
| Hook Event | PreToolUse on Task | PostToolUse on Write\|Edit |
| Trigger | Before sub-agent spawn | After code change |
| Detection | Task description pattern | File type + change size |
| Injection | `systemMessage` (invalid) | `additionalContext` |
| Enforcement | Blocking validation | Non-blocking suggestion |

### 12.3 New Hook: suggest-pipeline.sh

Replaces `validate-pipeline.sh`. Fires after Write/Edit:
- Detects file type (code, test, config, doc, script, data)
- Calculates change size (lines changed)
- Compares against threshold per file type
- If significant: injects `additionalContext` suggesting pipeline

### 12.4 File Type Thresholds

| File Type | Threshold | Recommended Pipeline |
|-----------|-----------|---------------------|
| Code | < 5 lines | Code Review |
| Tests | < 10 lines | Test Audit |
| Config | < 3 lines | Code Review (security) |
| Documentation | <= 10 lines | Light review |
| Scripts | < 3 lines | Code Review (security) |
| Data | Any change | Manual review |

### 12.5 Updated Deliverables

| Original | Revised |
|----------|---------|
| `validate-pipeline.sh` (PreToolUse) | `suggest-pipeline.sh` (PostToolUse) |
| Hard validation | Soft suggestion |
| SKILL.md with validation rules | SKILL.md with file-type mapping |

### 12.6 Key Learning

PostToolUse hooks can inject `additionalContext` that Claude considers, but it's advisory not directive. The pipeline suggestion is a recommendation, not enforcement. True enforcement would require blocking (exit code 2), which is not appropriate for this use case.
