## 1. Overview

### 1.1 Purpose

Create a custom sub-agent that validates fixes against the debug report produced by `bulwark-issue-analyzer`. This agent executes the tiered validation plan, assesses confidence, and determines if the fix is ready for code review.

**Invocation**: Via Task tool (agents don't appear in `/` menu)

```
Task(subagent_type="bulwark-fix-validator", prompt="...")
```

### 1.2 Pipeline Context

```fsharp
// Fix Validation Pipeline - Stage 4
IssueAnalyzer (bulwark-issue-analyzer, produces debug_report)
|> FixWriter (orchestrator implements fix)
|> (if !tests_cover_scenario then TestWriter else Skip)  // Conditional: only if tests don't already exist
|> FixValidator (bulwark-fix-validator, validates against debug_report)  // <-- THIS AGENT
|> CodeReviewer (reviews all, approves/rejects)
|> (if !approved then IssueAnalyzer else Done)
|> LOOP(max=3)
```

**TestWriter Condition**: The orchestrator evaluates whether existing tests cover the bug scenario using the debug report's `validation_plan.tests_to_execute` and `recommendation.new_tests_needed` fields. TestWriter is skipped if adequate test coverage already exists.

### 1.3 Why a Wrapper Agent?

| Reason | Explanation |
|--------|-------------|
| **Isolated context** | Validates fix without bias from having written it |
| **Debug report driven** | Uses validation plan and confidence criteria from IssueAnalyzer |
| **Encapsulated behavior** | Agent definition specifies skills, tools, output format |
| **Clean interface** | Orchestrator invokes via Task tool with fix details in CONTEXT |

### 1.4 Key Insight from P1.2 Testing (Session 14)

During P1.2-3 testing, the orchestrator improvised a fix validator that:
- Generated executable JS scripts to run tests
- Executed tiered validation (P1 → P2 → P3)
- Analyzed fix implementation (before/after, components, edge cases)
- Produced comprehensive validation report with confidence assessment
- Wrote results to `tmp/validation-results-{issue-id}.txt`

This task brief incorporates those patterns.

## 3. Protocol

### Step 1: Read Debug Report

Parse the debug report YAML to extract:
- `validation_plan.tests_to_execute` - Tiered test list (P1/P2/P3)
- `validation_plan.functionalities_to_validate` - User-visible behaviors
- `confidence_criteria` - High/medium/low rubrics
- `analysis.root_cause` - What the fix should address
- `analysis.fix_approach` - Expected fix direction

### Step 2: Execute Tiered Tests

Run tests in priority order, stopping if blockers found:

| Priority | Action | Stop Condition |
|----------|--------|----------------|
| **P1 (must)** | Run all P1 tests | Any failure → FAIL |
| **P2 (should)** | Run P2 if P1 passes | Failures noted, continue |
| **P3 (nice-to-have)** | Run P3 if time permits | Failures noted, continue |

**Test Execution Methods** (in order of preference):
1. Native test runner (`just test`, `npm test`, `pytest`)
2. Direct file execution (`node`, `ts-node`, `python`)
3. Generate validation scripts when native runners fail (e.g., missing dependencies, compilation errors) - see Section 6.3 for examples
4. Manual logic validation (last resort)

### Step 3: Validate Functionalities

For each item in `functionalities_to_validate`:
- Check if tests cover the functionality
- Trace code path to verify fix addresses it
- Note any gaps requiring manual validation

### Step 3.5: Call Site Analysis

Identify impact of the fix beyond direct test coverage:

1. **Find modified functions**: List all functions/methods changed by the fix
2. **Search for call sites**: Use Grep to find all callers of modified functions
   ```bash
   # Example: Find all callers of the modified function
   grep -rn "functionName(" src/ --include="*.ts"
   ```
3. **Assess coverage**: For each call site:
   - Is the caller covered by P1/P2 tests?
   - Does the fix change behavior for this caller?
   - Flag as risk if not covered by tests
4. **Document gaps**: List uncovered call sites in validation report

**Note**: See Future Enhancements (Section 12) for LSP-based call site analysis.

### Step 4: Analyze Fix Implementation

Examine the fix applied:

| Check | Description |
|-------|-------------|
| **Root cause addressed** | Does fix target the issue identified in debug report? |
| **Minimal change** | Is fix surgical or does it touch unrelated code? |
| **Edge cases** | Are boundary conditions handled? |
| **Type safety** | Does fix align with type system? |
| **No regressions** | Do existing tests still pass? |
| **Call site coverage** | Are all call sites from Step 3.5 covered by tests or flagged as risks? |

### Step 5: Assess Confidence

Map results to confidence criteria from debug report:

| Level | Criteria (typical) |
|-------|-------------------|
| **HIGH** | All P1 tests pass, root cause clearly addressed, no regressions, new test covers bug scenario |
| **MEDIUM** | P1 tests pass, some P2 fail or skipped, minor uncertainty remains |
| **LOW** | Tests pass but root cause unclear, or unable to fully verify, or edge cases not covered |

### Step 6: Write Outputs

1. Write validation report to `logs/validations/fix-validation-{issue-id}-{YYYYMMDD-HHMMSS}.yaml`
2. Write diagnostics to `logs/diagnostics/bulwark-fix-validator-{YYYYMMDD-HHMMSS}.yaml`
3. Return summary to orchestrator (include validation report path and confidence level)

## 6. Test Execution Strategies

### 6.1 Native Test Runner (Preferred)

```bash
# Detect and use project's test runner
just test                           # If justfile exists
npm test                            # If package.json with test script
pytest                              # If pytest.ini or conftest.py
go test ./...                       # If go.mod exists
```

### 6.2 Direct Execution

```bash
# Run specific test file directly
npx ts-node tests/auth.test.ts      # TypeScript
node tests/auth.test.js             # JavaScript
python -m pytest tests/test_auth.py # Python
```

### 6.3 Generated Validation Script

When native runners fail (e.g., missing dependencies), generate a minimal validation script:

```javascript
// tmp/validate-{issue-id}.js
const { AuthService } = require('./src/auth');

async function validate() {
  const auth = new AuthService();

  // Test 1: Register and login without profile
  await auth.register('test@example.com', 'password');
  const result = await auth.login('test@example.com', 'password');

  console.log('Test 1:', result.success ? 'PASS' : 'FAIL');
  console.log('Welcome message:', result.welcomeMessage);

  // Verify email fallback
  if (result.welcomeMessage.includes('test@example.com')) {
    console.log('Email fallback: PASS');
  } else {
    console.log('Email fallback: FAIL');
  }
}

validate().catch(console.error);
```

**Note**: Delete generated scripts after execution (security hygiene).

### 6.4 Manual Logic Validation

When execution isn't possible, validate by code inspection:
1. Trace execution path through fixed code
2. Verify fix addresses root cause identified in debug report
3. Check edge cases are handled
4. Confirm type system alignment
5. Note as "manual validation" in report

## 7. Confidence Mapping

### 7.1 From Debug Report

The debug report's `confidence_criteria` section defines what HIGH/MEDIUM/LOW mean for this specific fix. The validator must:

1. Read these criteria
2. Check each criterion
3. Map results to appropriate level

### 7.2 Default Criteria (if not specified)

| Level | Default Criteria |
|-------|-----------------|
| **HIGH** | All P1 tests pass, new test covers bug scenario, no regressions, fix is minimal |
| **MEDIUM** | P1 tests pass, some criteria uncertain, minor edge cases unclear |
| **LOW** | Tests pass but validation incomplete, or fix doesn't clearly address root cause |

### 7.3 Escalation Triggers

Escalate to manual testing when:
- Cannot execute tests (missing dependencies, compilation errors)
- Fix touches areas outside validation plan
- Edge cases require human judgment
- Security implications suspected

### 7.4 Complexity-Driven Validation Depth

Use the `complexity` and `risk_scope` from the debug report to calibrate validation effort:

| Complexity | Risk Scope | Validation Depth |
|------------|------------|------------------|
| **Low** | Isolated | P1 tests only, skip call site analysis, no human-readable report |
| **Low** | Medium | P1 tests + cursory P2, minimal call site check |
| **Medium** | Any | P1 + P2 tests, full call site analysis, human-readable report |
| **High** | Any | P1 + P2 + P3 (if available), exhaustive call site analysis, human-readable report, likely escalation |

**Token Budget Guidance**:

| Complexity | Expected Token Range | Rationale |
|------------|---------------------|-----------|
| Low | 3-8K | Simple fix, minimal verification needed |
| Medium | 8-15K | Standard fix, thorough verification |
| High | 15-25K | Complex fix, exhaustive verification |

**Important**: Do not "throw the kitchen sink at a typo." A low-complexity typo fix does not warrant full call site analysis or P2/P3 test execution.

## 12. Future Enhancements

### 12.1 LSP-Based Call Site Analysis

**Current approach (Section 3.5)**: Grep-based call site search
```bash
grep -rn "functionName(" src/ --include="*.ts"
```

**Future enhancement**: Use Language Server Protocol (LSP) for precise call site analysis.

**Benefits of LSP**:
- Accurate symbol resolution (not pattern matching)
- Handles aliased imports and re-exports
- Type-aware (finds interface implementations)
- Cross-file reference tracking

**Implementation path**:
1. Claude Code now supports LSP integration
2. Add LSP-based "Find All References" capability to call site analysis
3. Replace Grep-based approach for supported languages
4. Fall back to Grep for languages without LSP support

## 13. `/fix-bug` Skill (Pipeline Trigger)

### 13.1 Purpose

Provide explicit, deterministic invocation of the Fix Validation pipeline. Without this skill, users must use specific phrasing ("run the Fix Validation pipeline") for Claude to recognize and follow the pipeline stages.

**Problem observed (Session 16)**: Conversational prompts like "please investigate, fix and validate" caused Claude to skip IssueAnalyzer and fix directly. The explicit pipeline name was required.

### 13.2 Specification

```yaml
---
name: fix-bug
description: Run the Fix Validation pipeline to investigate, fix, and validate a bug
user-invocable: true
---
```

**Usage**:
```
/fix-bug path/to/code "description of issue"
```

**Arguments**:
- `$1`: Path to code with bug (required)
- `$2` or remaining `$ARGUMENTS`: Description of the issue (optional but recommended)

### 13.3 Skill Behavior

The skill instructs Claude to:

1. Load the `pipeline-templates` skill (specifically `references/fix-validation.md`)
2. Follow the Fix Validation pipeline stages in order:
   ```fsharp
   IssueAnalyzer (bulwark-issue-analyzer)
   |> FixWriter (orchestrator implements fix)
   |> (if !tests_cover_scenario then TestWriter else Skip)
   |> FixValidator (bulwark-fix-validator)
   |> CodeReviewer (reviews all)
   |> (if !approved then IssueAnalyzer else Done)
   |> LOOP(max=3)
   ```
3. Use `bulwark-issue-analyzer` for Stage 1
4. Use `bulwark-fix-validator` for Stage 4
5. Report progress at each stage

### 13.4 Success Criteria

| Criterion | Validation |
|-----------|------------|
| Skill appears in `/` menu | `user-invocable: true` |
| Accepts path argument | `$1` parsed correctly |
| Accepts description | `$ARGUMENTS` or `$2` parsed |
| Triggers IssueAnalyzer | Stage 1 executes |
| Follows full pipeline | All stages execute in order |
| Uses correct agents | bulwark-issue-analyzer, bulwark-fix-validator |

### 13.5 Manual Test

```

```

**Verification**:
- [ ] Skill loaded from `/` menu
