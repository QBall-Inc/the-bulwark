## 1. Functional Requirements

### 1.1 Core Purpose

Provide dynamic validation of Claude Code assets (skills, hooks, agents, commands, MCP servers) against official Anthropic standards. Unlike a static checklist, this skill:

1. **Fetches latest standards** from official Claude Code documentation
2. **Critically analyzes** assets against those standards
3. **Produces actionable reports** with specific violations and remediation

**Key Problem Addressed**: Claude Code evolves rapidly. Static checklists become outdated. This skill ensures validation always uses current standards.

### 1.2 Content Requirements

| Requirement | Description |
|-------------|-------------|
| FR-1 | Sectioned structure - one section per Claude Code capability |
| FR-2 | Dynamic doc fetching via `claude-code-guide` sub-agent |
| FR-3 | Critical analysis via `bulwark-standards-reviewer` custom agent |
| FR-4 | Fallback to embedded checklist if docs unavailable (with warning) |
| FR-5 | Structured YAML output to `logs/validations/` |
| FR-6 | Human-readable summary for user |
| FR-7 | Batch validation support (directory of assets) |
| FR-8 | Both file path input and context inference supported |

### 1.3 User Stories

- As a **developer**, I want to validate a skill I just created against Anthropic standards
- As an **orchestrator**, I want to validate all assets in a directory before release
- As a **reviewer**, I want structured reports showing exactly what violates standards
- As a **user with network issues**, I want validation to still work with fallback checklists

## 3. Technical Design

### 3.1 Section Structure

Each section follows this pattern:

```markdown
## {Capability} Validation

### Official Documentation
{Link to Anthropic docs}

### Validation Workflow

1. Identify {capability} asset(s) to validate
2. Spawn `claude-code-guide` agent with doc link
3. Receive current standards and supported features
4. Spawn `bulwark-standards-reviewer` with:
   - Asset content
   - Fetched standards
   - Validation criteria
5. Receive structured findings
6. Write YAML report to `logs/validations/`
7. Present summary to user

### Fallback Checklist
{Static checklist used if doc fetch fails}
```

### 3.2 Sections to Include

| Section | Doc Link | Key Validation Points |
|---------|----------|----------------------|
| **Skills** | /docs/claude-code/skills | Frontmatter fields, SKILL.md structure, user-invocable, agent field |
| **Hooks** | /docs/claude-code/hooks | Hook types, matcher patterns, once: true, command format |
| **Agents** | /docs/claude-code/sub-agents | context: fork, agent model, Stop hooks, output format |
| **Commands** | /docs/claude-code/commands | Command structure, naming, parameters |
| **MCP Servers** | /docs/claude-code/mcp | Server configuration, tool definitions |
| **Plugins** | /docs/claude-code/plugins | Flat structure, manifest format, registration |

### 3.3 Validation Workflow

```fsharp
// Pseudo-code for validation flow
DetermineAssetType (file or context)
|> SelectRelevantSection
|> SpawnClaudeCodeGuide (fetch latest standards)
|> (if fetch_failed then WarnAndUseFallback else UseLatestStandards)
|> SpawnStandardsReviewer (critical analysis)
|> WriteYamlReport (logs/validations/)
|> PresentSummary (human-readable)
```

### 3.4 Sub-Agent Prompts

#### claude-code-guide Agent

```
GOAL: Fetch current standards for {capability_type} from official Claude Code docs

CONSTRAINTS:
- Only use official Anthropic documentation
- Report what IS supported vs what is NOT
- Include any recent changes or deprecations

CONTEXT:
- Documentation URL: {doc_link}
- Capability being validated: {capability_type}

OUTPUT:
- Current supported fields/features
- Required vs optional elements
- Common pitfalls or mistakes
- Any version-specific notes
```

#### bulwark-standards-reviewer Agent

```
GOAL: Critically analyze {asset_path} against fetched standards

CONSTRAINTS:
- Be thorough - check every requirement
- Rate findings by severity (Critical/High/Medium/Low)
- Provide specific remediation for each finding
- Do NOT modify the asset, only report

CONTEXT:
- Asset to validate: {asset_content}
- Current standards: {fetched_standards}
- Asset type: {capability_type}

OUTPUT:
Write structured YAML to logs/validations/{asset-name}-{timestamp}.yaml:
- metadata (asset, timestamp, validator)
- findings[] (severity, rule, violation, remediation)
- summary (total findings by severity, pass/fail)
```

### 3.5 Output Format

#### YAML Report (logs/validations/)

```yaml
validation_report:
  metadata:
    asset: "{file_path}"
    asset_type: skill | hook | agent | command | mcp
    timestamp: "{ISO-8601}"
    validator: "bulwark-standards-reviewer"
    standards_source: fetched | fallback

  findings:
    - severity: critical | high | medium | low
      rule: "{standard being checked}"
      violation: "{what is wrong}"
      location: "{line or field}"
      remediation: "{how to fix}"

  summary:
    total_findings: 0
    critical: 0
    high: 0
    medium: 0
    low: 0
    verdict: pass | fail
    notes: "{any additional context}"
```

#### Human-Readable Summary

```
Validation: skills/my-skill/SKILL.md
Standards: Fetched from official docs (2026-01-16)
Verdict: FAIL (2 critical, 1 high)

Critical:
  - Missing required 'description' field in frontmatter
  - SKILL.md exceeds 500 line limit (612 lines)

High:
  - 'agent' field uses deprecated value 'gpt-4'

Remediation suggestions provided in:
  logs/validations/my-skill-2026-01-16T10-30-00.yaml
```

### 3.6 Batch Validation

When given a directory:

```
Input: skills/
Process:
  1. Glob all SKILL.md files
  2. For each, run single validation workflow
  3. Aggregate results
  4. Write summary report + individual reports
```

### 3.7 Fallback Behavior

```
If claude-code-guide fetch fails:
  1. Log warning: "Could not fetch latest standards, using embedded checklist"
  2. Set standards_source: fallback in report
  3. Use embedded checklist from references/
  4. Continue validation
  5. Include note in summary: "Validated against potentially outdated standards"
```

### 10.3 Resolved: Severity Threshold

**Decision**: Any Critical = Fail, but report ALL findings

**Rationale**:
- Critical findings are blockers (missing required fields, wrong structure)
- High/Medium/Low are quality improvements
- User needs full visibility to make informed decisions
- Comprehensive report lets user prioritize fixes

**Severity Definitions**:
| Severity | Definition | Examples |
|----------|------------|----------|
| **Critical** | Blocks functionality, violates required standards | Missing required frontmatter, wrong file location, unsupported field |
| **High** | Significant issue, should fix before release | Deprecated field value, exceeds recommended limits |
| **Medium** | Quality improvement, recommended | Missing optional but recommended fields |
| **Low** | Style/naming suggestions | Naming conventions, documentation suggestions |

**Verdict Logic**:
```
if any_critical_findings:
    verdict = "FAIL"
else:
    verdict = "PASS"
# Always list ALL findings regardless of verdict
```
