# P5.10-12: Test Audit Hardening

## Context

The test-audit skill currently relies on LLM heuristics for line counting, T3+ detection, and has no T4 detection. This produces imprecise results: negative effectiveness percentages, missed violations with generic variable names, and undetected skipped tests.

P5.10-12 replaces guessing with knowing — three AST scripts that read code structurally, integrated as a deterministic pre-processing step in the test-audit pipeline.

### Research Foundation

Eight research agents (3 domain + 5 viewpoint) produced findings documented in:
- `logs/p5.10-research-20260211-014713.yaml` — AST tooling comparison
- `logs/p5.11-research-20260211-064759.yaml` — Dual-mode architecture

## Functional Requirements

### FR1: Three AST Scripts (P5.10)

All scripts use **ts-morph** for AST parsing, run via **npx tsx**, and output **JSON to stdout**.

#### FR1.1: verification-counter.ts

Precise line counting replacing heuristic estimates.

**Input**: Path to test file(s)
**Output**: JSON with per-file metrics

```json
{
  "file": "tests/user.test.ts",
  "metrics": {
    "total_lines": 156,
    "test_logic_lines": 98,
    "assertion_lines": 42,
    "setup_lines": 56,
    "effectiveness_percent": 42.86,
    "framework_detected": "jest"
  },
  "breakdown": {
    "imports": 12,
    "comments": 18,
    "empty_lines": 28,
    "boilerplate": 14,
    "test_logic": 98
  }
}
```

**Counting rules**:
- **Exclude**: imports, comments, empty lines, test framework wrappers (`describe`, `it`, `beforeEach`, `afterEach`, `beforeAll`, `afterAll` — the wrapper call itself, not the body)
- **Include**: statements within test function bodies, mock setup, assertions, variable assignments

**Assertion recognition** (extensible registry):
- Jest/Vitest: `expect(...).toX(...)`
- node:assert: `assert(...)`, `assert.strictEqual(...)`, `assert.deepStrictEqual(...)`
- Mocha/Chai: `expect(...).to.equal(...)`, `should.equal(...)`

Start with Jest/Vitest coverage (95% of ecosystem). Registry pattern allows adding frameworks later.

#### FR1.2: skip-detector.ts

Deterministic T4 detection — finds disabled/focused test markers.

**Input**: Path to test file(s)
**Output**: JSON with marker locations

```json
{
  "file": "tests/user.test.ts",
  "markers": [
    {
      "type": "test.skip",
      "line": 42,
      "test_name": "should handle edge case",
      "severity": "medium",
      "rule": "T4"
    }
  ],
  "summary": {
    "skip_count": 3,
    "only_count": 1,
    "todo_count": 2
  }
}
```

**Patterns to detect**:
- `test.skip()`, `it.skip()`, `describe.skip()`
- `test.only()`, `it.only()`, `describe.only()`
- `test.todo()`, `it.todo()`
- `xdescribe()`, `xit()` (alternative skip syntax)
- `fdescribe()`, `fit()` (Jasmine focus syntax)

#### FR1.3: data-flow-analyzer.ts

T3+ broken integration chain detection via data flow tracing.

**Scope**: Single-file analysis (catches ~90% of T3+ violations based on known limitation analysis). Cross-file deferred to future enhancement.

**Input**: Path to test file(s)
**Output**: JSON with violation locations

```json
{
  "file": "tests/workflow.integration.ts",
  "violations": [
    {
      "line": 42,
      "type": "T3+",
      "confidence": "high",
      "variable": "orderData",
      "source": "object_literal",
      "message": "Variable 'orderData' is manually constructed — breaks integration chain",
      "suggestion": "Replace with factory function or upstream function output"
    }
  ]
}
```

**Algorithm**:
1. Identify test functions (CallExpressions matching `test`/`it`/`describe`)
2. Extract test body
3. Build variable assignment map within test body
4. For each variable used as function argument, trace to source:
   - Object/Array literal → FLAG as T3+ (manually constructed data)
   - CallExpression result → PASS (real function output)
   - Import from `*mock*` file → WARN for review
5. Report violations with confidence level

### FR2: Dual-Mode Architecture (P5.11)

Automatic mode selection based on file count.

| Mode | Trigger | Pipeline |
|------|---------|----------|
| **Deep** | ≤5 files | Stage 0 (AST) → Stage 2 (Detection on ALL files) → Stage 3 (Synthesis) |
| **Scale** | >5 files | Stage 0 (AST) → Stage 1 (Classification) → Stage 2 (Detection on flagged) → Stage 3 (Synthesis) |

**Deep Mode**: Skips classification, runs full Sonnet analysis on every file. Detection stage self-computes verification_lines and mock_indicators (normally provided by classification).

**Scale Mode**: Current pipeline with AST hints injected into classification and detection prompts.

**Threshold**: Default 5, configurable via `--threshold=N`. Safety cap: deep mode refuses >25 files.

**UX**: Display mode selection with rationale at audit start. Diagnostic log records mode, threshold, stages executed/skipped.

## Technical Design

### Tooling Stack

| Component | Choice | Rationale |
|-----------|--------|-----------|
| AST Parser | ts-morph (v27.x) | Type-aware analysis, ergonomic API, wraps TypeScript Compiler API |
| Runtime | npx tsx | Zero-config TS execution, no build step |
| Peer Dependency | typescript | Required by ts-morph |
| Invocation | Justfile recipes | Per V2 (use just for execution), abstracts runtime |

### Justfile Recipes

```just
# ============================================================
# Test Audit AST Recipes
# ============================================================

# Count verification lines in test file(s)
verify-count *path:
    @npx tsx skills/test-audit/scripts/verification-counter.ts {{path}}

# Detect skipped/focused/todo test markers
skip-detect *path:
    @npx tsx skills/test-audit/scripts/skip-detector.ts {{path}}

# Analyze data flow for T3+ broken integration chains
ast-analyze *path:
    @npx tsx skills/test-audit/scripts/data-flow-analyzer.ts {{path}}
```

These recipes:
- Abstract the tsx runtime (swap to compiled JS later without changing callers)
- Follow existing Justfile conventions (lowercase, verb-noun)
- Accept glob paths (`just verify-count tests/**/*.test.ts`)
- Must be added to scaffold Justfile templates (P6.2/P7.1)

### File Layout

All new files inside the test-audit skill (self-contained principle):

```
skills/test-audit/
├── SKILL.md                          (modified — Stage 0, dual-mode)
├── scripts/
│   ├── verification-counter.ts       (new)
│   ├── skip-detector.ts              (new)
│   ├── data-flow-analyzer.ts         (new)
│   └── package.json                  (new — ts-morph + tsx deps)
└── references/
    ├── stub-patterns.md              (new)
    └── false-positive-prevention.md  (new)

skills/mock-detection/
└── SKILL.md                          (modified — extended patterns)

Justfile                              (modified — 3 new recipes)
```

### Stage 0 Integration

Stage 0 is mode-agnostic — runs before mode selection, feeds both modes.

```
Stage 0: AST Pre-Processing
├── Input: Target path (from /test-audit invocation)
├── Execution:
│   just verify-count {target} > /tmp/claude/ast-verify-count.json
│   just skip-detect {target} > /tmp/claude/ast-skip-detect.json
│   just ast-analyze {target} > /tmp/claude/ast-data-flow.json
├── Output: Per-file AST metadata (verification_lines, skip_markers, data_flow_chains)
├── Duration: ~5-10s for 50 files (deterministic, no LLM cost)
└── Failure handling: If any script fails, log warning and continue with LLM-only analysis (graceful degradation)
```

AST output is injected into LLM prompts as structured metadata:
- Classification (Haiku) gets AST hints for better flagging
- Detection (Sonnet) gets precise line counts + T3+ leads
- Synthesis uses AST verification_lines for effectiveness calculation (replaces heuristic)

### Error Handling

Per viewpoint feedback, every script must:
1. Exit 0 on success with JSON to stdout
2. Exit 1 on failure with error JSON to stderr: `{ "error": "...", "file": "..." }`
3. Handle unparseable files gracefully (skip file, log warning, continue)
4. Not crash on edge cases (empty files, non-TS files, syntax errors)

The test-audit orchestrator treats Stage 0 as best-effort:
- If scripts succeed → use AST metadata (enhanced accuracy)
- If scripts fail → fall back to LLM-only analysis (current behavior)
- Never block the audit because AST scripts failed

### Pre-Flight Gate (DEF-P4-005 Prevention)

Stage 0 must include binding language in the SKILL.md update per DEF-P4-005. The orchestrator MUST run AST scripts before LLM stages — this is not optional or skippable.

### Prompt Schema (Session 48 Lesson)

LLM prompts that consume AST output MUST include the exact JSON schema they expect. Agents produce wrong output formats without literal schema in the prompt.

## Implementation Phases

Research unanimously recommended phased rollout with validation gates.

### Phase 1: Core Scripts (P5.10a + P5.10b)

**Scope**: verification-counter.ts + skip-detector.ts + Justfile recipes + Stage 0 skeleton

**Why first**: Lowest risk, immediate value, validates the ts-morph + tsx tooling stack before investing in the complex data-flow-analyzer.

**Deliverables**:
- `skills/test-audit/scripts/verification-counter.ts`
- `skills/test-audit/scripts/skip-detector.ts`
- `skills/test-audit/scripts/package.json` (ts-morph + tsx)
- Justfile: `verify-count` and `skip-detect` recipes
- Stage 0 integration in test-audit SKILL.md (verify-count + skip-detect only)

**Validation gate**: Run scripts on real test files. Verify:
- Precise line counts (compare to manual count)
- All skip/only/todo markers found (compare to grep)
- No crashes on edge cases (empty files, JS files, syntax errors)

### Phase 2: Data Flow Analyzer (P5.10c)

**Scope**: data-flow-analyzer.ts + Justfile recipe + Stage 0 completion

**Why second**: Most complex script, highest value. Needs validation before building dual-mode on top.

**Deliverables**:
- `skills/test-audit/scripts/data-flow-analyzer.ts`
- Justfile: `ast-analyze` recipe
- Stage 0 updated to include data-flow-analyzer

**Validation gate**: Run on test fixtures with known T3+ violations. Verify:
- Detects object literals used as function arguments in integration chains
- Does not flag legitimate test data (builders, factory functions)
- Confidence levels are meaningful

### Phase 3: Reference Docs (P5.12)

**Scope**: stub-patterns.md + false-positive-prevention.md + mock-detection update

**Why third**: Can parallel with Phase 2. Independent of AST scripts — these are LLM-consumed reference docs.

**Deliverables**:
- `skills/test-audit/references/stub-patterns.md`
- `skills/test-audit/references/false-positive-prevention.md`
- `skills/mock-detection/SKILL.md` (updated patterns)

**Validation gate**: Run test-audit on code with manual stubs (StubX, FakeX). Verify:
- Extended patterns detected
- False positives reduced (builders, supertest, MSW not flagged)

### Phase 4: Dual-Mode (P5.11)

**Scope**: Deep mode + Scale mode + mode selection + UX

**Why last**: Builds on proven AST scripts. Only worth the complexity after scripts are validated.

**Deliverables**:
- `skills/test-audit/SKILL.md` (full dual-mode orchestration)
- Deep mode detection prompt (self-computes classification metadata)
- Unified synthesis prompt (handles both modes)
- Mode selection UX (display + diagnostic logging)

**Validation gate**:
- Run on 3 files → verify deep mode selected, no classification stage
- Run on 50 files → verify scale mode selected, batching works
- Compare results between modes (should be equivalent for same files)

## Success Criteria

1. verification-counter produces exact line counts (no negative percentages)
2. skip-detector finds all .skip/.only/.todo markers deterministically
3. data-flow-analyzer detects T3+ violations without "mock" in variable names
4. All scripts run via Justfile recipes (`just verify-count`, `just skip-detect`, `just ast-analyze`)
5. Stage 0 output feeds LLM stages with structured metadata
6. Graceful degradation: AST failure doesn't block audit
7. Dual-mode correctly selects deep (≤5) vs scale (>5)
8. Mock-detection skill references new stub-patterns and false-positive-prevention docs
9. All scripts self-contained inside `skills/test-audit/scripts/`

## Risk Mitigations (From Viewpoint Analysis)

| Risk | Source | Mitigation |
|------|--------|-----------|
| ts-morph bundle size (1.47MB) | Viewpoint 1 | Acceptable for skills context; fallback to compiled JS if needed |

| "90%" single-file claim unvalidated | Viewpoint 3 | Validate in Phase 2 before building dual-mode on top |
| Self-contained skills principle | Viewpoint 5 | Scripts inside `skills/test-audit/scripts/`, not root |
| Pre-Flight Gate needed | Viewpoint 5 | Binding language in SKILL.md per DEF-P4-005 |
| Prompt needs exact schema | Viewpoint 5 | Literal JSON schema in Stage 1/2 CONTEXT sections |
| Allowlists = technical debt | Viewpoint 4 | Start minimal; expand only when false positives emerge in practice |
| No empirical baseline | Viewpoint 3 | Phase 1 validation establishes baseline before further investment |
