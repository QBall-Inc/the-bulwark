# P5.3: Continuous-Feedback Skill

## Context

Session 62 research established that P5.3 uses Task tool sub-agents (not Agent Teams, not Ralph Loops). Session 63 brainstorm (5 Opus agents) produced a full design with post-synthesis user decisions. The original bundled brief P5.3-5 was split — P5.4/P5.5 will have separate briefs after their research cycle (Session 64).

### Design Decisions (Session 63 Brainstorm)

| Decision | Rationale |
|----------|-----------|
| Full 4-agent pipeline directly (no v0) | User trusts existing multi-stage patterns. Critic's v0 approach adds build/test overhead without guaranteed value. |
| 3 specializations: test-audit + code-review + general | PM recommendation. Both skills have rich documented improvement histories. General covers everything else. |
| Pass-through schema (Collector preserves full content) | Maximum fidelity. Avoids the lossy intermediate format the Critic identified as second-highest risk. |
| Proposal-only output (no direct file mods) | User preference. Preserves trust, fits user-as-orchestrator pattern. Auto-apply deferred to v2. |
| Collector as sub-agent (not orchestrator) | Avoids 25-40% token consumption from raw file reading in orchestrator context. |
| LLM-classified routing (not keyword matching) | Collector assigns skill_relevance during collection. Keywords miss 30-40% of learnings (Critic finding). |
| No Critical Evaluation Gate | Proposal-only output doesn't need interactive synthesis. Saves tokens. |
| General-purpose (Claude Code ecosystem) | Assumes `.claude/` skills/agents and `sessions/` handoffs. Configurable input paths. |
| First test run targets test-audit | Richest improvement history. If pipeline works here, works for simpler skills. |
| Specialization references sync to standalone repo | Included as examples users can adapt for their own skills. |

## Functional Requirements

### FR1: Skill Structure

```
skills/continuous-feedback/
├── SKILL.md                              (main skill document)
├── references/
│   ├── collect-instructions.md          (parsing rules for session handoffs, MEMORY.md, custom paths)
│   ├── specialize-test-audit.md         (test-audit improvement patterns)
│   ├── specialize-code-review.md        (code-review improvement patterns)
│   └── specialize-general.md           (general skill improvement patterns)
└── templates/
    ├── collect-output.md                (pass-through learning item schema)
    ├── proposal-output.md               (change proposal format)
    └── diagnostic-output.yaml           (standard diagnostic schema)
```

### FR2: Pipeline Topology

```fsharp
// continuous-feedback pipeline
PreFlight(args, inputs)                    // Stage 0: Orchestrator
|> Collector(sessions, memory, custom)     // Stage 1: Sonnet, sequential
|> [Analyzer(test-audit), Analyzer(code-review), Analyzer(general)]  // Stage 2: Sonnet, parallel (dynamic 1-3)
|> Proposer(all_analyses, target_skills)   // Stage 3: Sonnet, sequential
|> Validate(proposal)                      // Stage 4: Orchestrator, no sub-agent
```

4 sub-agents total: 1 Collector + 1-3 Analyzers + 1 Proposer. All Sonnet (analysis/proposal work, not code writing).

### FR3: Invocation

```
/continuous-feedback <target-skill-or-path> [--sources <paths>] [--since <session-N>]
```

**Arguments:**
- `<target-skill-or-path>` — Target skill name (e.g., `test-audit`) or path to skill directory. If a directory of skills, analyze all detected skill types.
- `--sources <paths>` — Custom input source paths (files or folders). Overrides default input sources.
- `--since <session-N>` — Only collect learnings from session N onwards. Default: last 10 sessions.

**Default input sources** (when `--sources` not provided):
- `sessions/*.md` — Session handoff files (windowed by `--since`)
- Project MEMORY.md — Always read in full (curated summary)
- Agent memory files — When available (e.g., `.claude/agent-memory/*/MEMORY.md`)

### FR4: Stage Definitions

#### Stage 0: Pre-Flight (Orchestrator)

```
Stage 0: Pre-Flight
├── Parse arguments (target, --sources, --since)
├── Resolve input sources (default or custom)
├── Verify minimum input threshold: ≥5 session handoffs (Pre-Flight Gate)
├── Detect target skill type: read SKILL.md frontmatter if targeting a skill
├── Determine which specialization references to load (based on detected skill types in target project)
├── Create output directory: logs/continuous-feedback/{run-slug}/
├── Load subagent-prompting skill
├── AskUserQuestion if target is ambiguous or inputs are unclear
└── Token budget check (warn if >30% consumed)
```

**Pre-Flight Gate (from test-audit pattern):**
- MUST verify ≥5 session handoffs exist in the input scope
- MUST verify target path exists and contains readable files
- MUST NOT proceed if input threshold not met — inform user: "Insufficient input data. Need at least 5 session handoffs."

#### Stage 1: Collect (Sonnet sub-agent, sequential)

```
Stage 1: Collect
├── Load references/collect-instructions.md
├── Construct prompt using 4-part template (GOAL/CONSTRAINTS/CONTEXT/OUTPUT)
│   ├── GOAL: Parse input sources and extract learning items with source attribution
│   ├── CONTEXT: Input file paths, parsing rules from collect-instructions.md
│   └── OUTPUT: logs/continuous-feedback/{run-slug}/01-collect.md
├── Spawn general-purpose Sonnet agent
│   ├── Agent reads session handoffs (Learnings, Technical Decisions, Blockers sections)
│   ├── Agent reads MEMORY.md (Defects, Lessons Learned, Architecture Decisions, Framework Observations)
│   ├── Agent reads custom paths (if --sources provided)
│   ├── Agent uses Grep to locate section headers, Read with offsets for targeted extraction
│   └── Agent assigns skill_relevance tags via LLM classification (not keyword matching)
├── Read Collector output
└── Token budget check
```

**Pass-through schema:** The Collector groups and tags learning items but preserves near-raw content. Each item includes:

```yaml
- source: "session_45_20260208.md"
  section: "Learnings"
  category: "defect-pattern"  # defect-pattern | architecture-decision | framework-observation | workflow-improvement | tool-behavior
  skill_relevance: ["test-audit", "code-review"]  # LLM-classified, which skills this learning might improve
  content: |
    Full learning text preserved with surrounding context.
    Multiple sentences retained for downstream actionability.
    No lossy compression — Analyzers handle interpretation.
```

#### Stage 2: Analyze (1-3 Sonnet sub-agents, parallel)

```
Stage 2: Analyze
├── Read Collector output (01-collect.md)
├── Determine Analyzer count: 1 per detected skill type in skill_relevance, capped at 3
│   ├── If test-audit learnings found → spawn test-audit Analyzer with specialize-test-audit.md
│   ├── If code-review learnings found → spawn code-review Analyzer with specialize-code-review.md
│   └── Always spawn general Analyzer with specialize-general.md
├── Load corresponding references/specialize-*.md for each Analyzer
├── Construct prompts using 4-part template
│   ├── Each receives: filtered learning items (matching its specialization) + target skill's current files
│   └── Each writes to: logs/continuous-feedback/{run-slug}/02-analyze-{specialization}.md
├── Spawn all Analyzers in parallel (single message, multiple Task calls)
├── Read all Analyzer outputs
└── Token budget check (checkpoint if >55%)
```

**Dynamic spawning:** The number of Analyzers is data-driven. If collected learnings only have `skill_relevance` matching test-audit and general, only 2 Analyzers spawn. The general Analyzer always runs.

#### Stage 3: Act/Propose (Sonnet sub-agent, sequential)

```
Stage 3: Act
├── Load templates/proposal-output.md
├── Construct prompt using 4-part template
│   ├── GOAL: Synthesize analysis into concrete, copy-paste-ready change proposals
│   ├── CONTEXT: All Analyzer outputs + target skill's current files (agent reads these autonomously)
│   └── OUTPUT: logs/continuous-feedback/{run-slug}/03-proposal.md
├── Spawn general-purpose Sonnet agent
│   ├── Agent reads all Analyzer outputs
│   ├── Agent reads target skill files (current state) to avoid stale proposals
│   ├── Agent produces proposals with mandatory fields (see proposal-output.md)
│   └── Agent includes good/bad proposal examples in output (from prompt)
├── Read Proposer output
└── Token budget check
```

**Proposal output format** (each proposed change):

```markdown
## Proposed Change N: {Short descriptive title}

**Target**: {exact file path, e.g., `.claude/skills/test-audit/references/mock-detection-patterns.md`}
**Change type**: Add | Modify | Remove
**Section**: {target section within file, or "New section"}
**Priority**: High | Medium | Low
**Source learnings**: {session/memory references that drive this proposal}

### Proposed content:
{The actual text to add or the modified text. Copy-paste ready.}

### Rationale:
{Why this improves the skill, traced to specific learning items.}

### Validation:
{How to verify this change works — e.g., "Run /anthropic-validator", "Re-run test-audit on fixture X"}
```

**Critical quality requirement:** The Proposer MUST produce proposals specific enough that the user can copy-paste them. "Improve security coverage" is a failure. "Add the following SSRF detection pattern to `references/security-patterns.md` under the ## Server-Side Request Forgery section: [specific content]" is success.

#### Stage 4: Validate (Orchestrator, no sub-agent)

```
Stage 4: Validate
├── Read proposal document (03-proposal.md)
├── For each proposed change targeting a skill asset:
│   └── Note: "Run /anthropic-validator on {target} after applying this change"
├── For proposals targeting code files:
│   └── Note: "Run just typecheck && just lint && just test after applying"
├── Write validation notes to logs/continuous-feedback/{run-slug}/04-validation.md
└── This stage is informational — actual validation happens when user applies proposals
```

**Design note:** The Validate stage does NOT run validators on proposals (proposals are not applied yet). It annotates each proposal with the appropriate validation steps the user should run after applying.

#### Stage 5: Diagnostics (REQUIRED)

```
Stage 5: Diagnostics
├── Write diagnostic YAML to logs/diagnostics/continuous-feedback-{YYYYMMDD-HHMMSS}.yaml
└── Verify completion checklist
```

### FR5: Error Handling

| Scenario | Action |
|----------|--------|
| Collector returns empty output | Re-spawn once. If still empty, STOP — cannot proceed without collected learnings. |
| Collector returns truncated output | Accept as-is, note in diagnostics. |
| Analyzer returns empty output | Re-spawn once. If still empty, skip that specialization, document gap. |
| Proposer returns empty/vague output | Re-spawn once with reinforced specificity instructions. If still vague, document in diagnostics. |
| Token budget exceeded mid-pipeline | Stop spawning, write partial results, note incomplete in diagnostics. |
| No learnings match a specialization | Do not spawn that Analyzer. Document in diagnostics. |
| Fewer than 5 session handoffs | Pre-Flight Gate blocks. Inform user. |

### FR6: Dogfooding and Sync

- Copy to `.claude/skills/continuous-feedback/` for dogfooding
- Add to `scripts/sync-essential-skills.sh` for standalone repo sync
- Specialization references sync as examples (user decision)
- Rsync transforms: strip any Bulwark-specific content during sync

## Technical Design

### Model Selection

| Stage | Agent | Subagent Type | Model | Rationale |
|-------|-------|--------------|-------|-----------|
| 1 (Collect) | Collector | general-purpose | Sonnet | File parsing and classification — analysis task |
| 2 (Analyze) | 1-3 Analyzers | general-purpose | Sonnet | Pattern identification — analysis task |
| 3 (Act) | Proposer | general-purpose | Sonnet | Proposal generation — analysis/synthesis task |
| 4 (Validate) | — | Orchestrator | — | Direct tool invocation, no sub-agent |

All agents are Sonnet per OR2 (standard complexity — review and analysis). No Opus agents needed — this skill analyzes and proposes, it doesn't write implementation code.

### Skill Conventions

Follows the established multi-stage skill pattern:

| Convention | Source Pattern | How Applied |
|------------|--------------|-------------|
| Directory layout | bulwark-brainstorm | SKILL.md + references/ + templates/ |
| Pre-Flight Gate | test-audit | MUST/MUST NOT blocking checks |
| Numbered output files | bulwark-research | 01-collect.md, 02-analyze-*.md, 03-proposal.md |
| Parallel agent spawning | bulwark-research, bulwark-brainstorm Stage 3 | Single message with multiple Task calls |
| Diagnostic YAML | All pipeline skills | logs/diagnostics/{skill}-{timestamp}.yaml |
| 4-part prompt template | subagent-prompting | GOAL/CONSTRAINTS/CONTEXT/OUTPUT for all agents |
| SA2 compliance | CLAUDE.md | All outputs to logs/, orchestrator reads logs |

### Frontmatter

```yaml
---
name: continuous-feedback
description: Identifies improvement targets from accumulated session learnings and proposes concrete skill/agent modifications. General-purpose pipeline for any Claude Code project.
user-invocable: true
skills:
  - subagent-prompting
---
```

## Implementation Plan

### Build Order

**Step 1: Templates (~10% of session)**
- Create `templates/collect-output.md` — pass-through learning item schema
- Create `templates/proposal-output.md` — change proposal format with mandatory fields and good/bad examples
- Create `templates/diagnostic-output.yaml` — copy from bulwark-research, adapt fields

**Step 2: References (~25% of session)**
- Create `references/collect-instructions.md` — parsing rules for session handoffs (Learnings, Technical Decisions, Blockers sections), MEMORY.md (Defects, Lessons, Architecture Decisions, Framework Observations), and custom paths
- Create `references/specialize-test-audit.md` — what to look for: mock detection gaps, assertion pattern additions, AST script coverage, new detection patterns from sessions
- Create `references/specialize-code-review.md` — what to look for: new security patterns, framework pattern gaps, review lens updates
- Create `references/specialize-general.md` — what to look for: instruction hardening (DEF-P4-005 pattern), workflow improvements, configuration updates

**Step 3: SKILL.md (~30% of session)**
- Pre-Flight Gate (input threshold, argument parsing)
- Stage 0-5 definitions following FR4 above
- Error handling table (FR5)
- Token budget management checkpoints
- Completion checklist
- Usage section with invocation syntax and examples

**Step 4: Sync and Validate (~10% of session)**
- Copy to `.claude/skills/continuous-feedback/` (dogfood)
- Run `/anthropic-validator` on the skill
- Update `scripts/sync-essential-skills.sh` to include the new skill

### Effort Estimation

**Total: 1 session (estimated 35-50K tokens)**

This matches the P5.14 precedent where bulwark-research and bulwark-brainstorm were each built in approximately 1 session.

## Success Criteria

1. Pipeline spawns Collector → 1-3 Analyzers (parallel) → Proposer in correct sequence
2. Collector extracts learning items with source attribution and LLM-classified skill_relevance
3. Collector preserves full learning content (pass-through, not lossy compression)
4. Analyzers spawn dynamically based on detected skill types in collected learnings
5. General Analyzer always runs regardless of detected specializations
6. Proposer produces proposals with all mandatory fields (target_file, change_type, section, proposed_content, rationale, validation)
7. Proposals are specific enough to copy-paste (not vague recommendations)
8. `--since` argument limits session scope; default is last 10
9. Pre-Flight Gate blocks on fewer than 5 session handoffs
10. Diagnostic YAML written after every invocation
11. Passes `/anthropic-validator`
12. Synced to `.claude/skills/` and standalone repo

## Risks and Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| Proposer produces vague proposals | HIGH | Mandatory template fields. Good/bad examples in prompt. Kill if <3 actionable from 62 sessions. |
| Collector token overflow on many sessions | MEDIUM | `--since` defaults to last 10. Collector uses Grep for section targeting, not full-file reads. |
| LLM routing variance (skill_relevance inconsistent) | MEDIUM | General Analyzer always runs as catch-all. Misrouted items still get analyzed. |
| Specialization references become stale | LOW | Run continuous-feedback on itself. References are short (50-80 lines). |
| Stale proposals on re-invocation | LOW | Proposer reads current skill state. Instructed to skip already-present content. |

---

## Kill Criteria (from Critical Analyst)

1. First test yields fewer than 3 actionable proposals from Bulwark's 62 sessions
2. Proposal acceptance rate below 40% across 3 invocations
3. Token consumption exceeds 60K per invocation
4. Proposals consistently require >50% rewriting to be applicable
