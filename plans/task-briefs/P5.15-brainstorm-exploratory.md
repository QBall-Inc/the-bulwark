## Problem Statement

Extend the existing bulwark-brainstorm skill with a second mode (`--exploratory`) that uses Agent Teams peer debate for collaborative idea validation. The current `--scoped` mode excels at structured analysis of well-defined problems but offers no mechanism for adversarial idea validation when the problem framing itself is uncertain.

### Mode Distinction

| Mode | Mechanism | When to Use |
|------|-----------|-------------|
| `--scoped` (default) | Sequential Task tool: SME first → 3 parallel → Critic last | Problem statement is well understood, focused implementation brainstorming |
| `--exploratory` | Agent Teams peer debate: SME first → 3 AT teammates debating | Validating whether an idea has merit, problem framing is uncertain |

### Distribution Context

The skill exists in 3 locations that must stay synchronized:
1. `skills/bulwark-brainstorm/` — source of truth
2. `.claude/skills/bulwark-brainstorm/` — dogfood copy

## Functional Requirements

### FR1: Dual-Mode Argument Parsing

- Add `--scoped` and `--exploratory` as mutually exclusive mode flags
- Default: `--scoped` when no flag provided (backward compatibility — non-negotiable)
- Updated argument signature: `<topic-or-prompt> [--research <file>] [--scoped | --exploratory]`
- `/bulwark-brainstorm <topic>` with no flag must produce identical behavior to today

### FR2: Mode Detection (Pre-Flight)

Reuse P5.13's mode detection pattern verbatim:
1. Check `$CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` env var
2. If env var SET and user requests `--exploratory`: proceed with AT mode
3. If env var SET and no mode flag: default to `--scoped` (do not auto-select AT)
4. If env var NOT SET and user requests `--exploratory`: notify user ("Agent Teams requires CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1. Using --scoped mode.") and fall back to `--scoped`
5. If env var NOT SET and no mode flag: `--scoped` (current behavior)

### FR3: Role Consolidation for --exploratory

5 roles collapse to 4 for --exploratory mode to stay within the 3-4 AT coordination cap:

| Role | --scoped Execution | --exploratory Execution |
|------|-------------------|------------------------|
| **Project SME** | Solo first (Task tool) | Solo first (Task tool) — identical |
| **Senior Product Manager** | Parallel (Task tool) | Merged into Product & Delivery Lead |
| **Senior Technical Architect** | Parallel (Task tool) | AT teammate |
| **Senior Development Lead** | Parallel (Task tool) | Merged into Product & Delivery Lead |
| **Critical Analyst** | Sequential last (Task tool) | AT teammate — active from start |

- New combined role: **Product & Delivery Lead** — absorbs PM's value/scope focus and Dev Lead's feasibility/effort focus
- New role reference file: `references/role-product-delivery-lead.md`
- Existing PM and Dev Lead references remain unchanged for --scoped mode
- Naming convention matches P5.13's "Eng & Delivery Lead" pattern

### FR4: Critic Active from Start in --exploratory

The Critical Analyst's role transforms between modes:
- **--scoped**: Sequential last gatekeeper. Receives ALL prior outputs. Maximum information, zero influence on the analysis process.
- **--exploratory**: Active AT teammate from spawn. Challenges in real time throughout the debate. Partial information early, maximum influence.

Prompt mitigation for --exploratory Critic: "Do not form a final verdict until all teammates have shared their summaries. Challenge early findings, but reserve your formal verdict for your log artifact."

This is the primary quality advantage of Agent Teams mode — the Critic's challenges shape the other teammates' thinking in real time rather than arriving after all positions are fixed.

### FR5: Enhanced Critic Problem-Framing Challenge (Both Modes)

Add a "Problem Validation" section to the Critic's role reference and output template that applies to BOTH --scoped and --exploratory:

> "Should this problem be solved at all? Is the premise valid? What evidence suggests this is worth investing in?"

This is distinct from existing focus areas:
- "Assumption challenges" — challenges assumptions within proposals
- "Kill criteria" — defines exit conditions
- "Problem validation" (new) — challenges whether the topic itself is worth brainstorming

Low effort (~1 hour), applies to all brainstorm runs.

### FR6: SA2 Compliance (Dual-Output Contract)

Adopted from P5.13 FR4:
- **--scoped mode**: All agent output to `logs/brainstorm/{topic-slug}/{NN}-{role}.md` (standard SA2)
- **--exploratory mode**: Dual-output contract in every AT teammate prompt:
  - Full analysis → `logs/brainstorm/{topic-slug}/{NN}-{role}.md` (SA2 artifact)
  - 3-5 sentence coordination summary → mailbox (AT peer messaging)
- Synthesis stage verifies all log files exist before proceeding

### FR7: AT Configuration

All settled decisions from research + P5.13:
- Display mode: In-process (WSL2 safe default, hardcoded)
- Lead mode: Delegate (coordination only, prevents lead from doing analysis)
- Communication: Selective challenge — teammates broadcast summary once, respond only to messages they disagree with (avoid O(n^2) full mesh)
- Teammate count: 3 (fixed, no custom size option in v1)

### FR8: Graceful Degradation and AT Failure Recovery

- Env var absent + --exploratory requested → notify user, fall back to --scoped
- AT teammate fails mid-debate → fall back to --scoped for remaining unfinished roles; partial AT output feeds into fallback as additional context
- Lead context compaction → documented as known platform limitation in error handling; structural mitigation is that SME runs before AT (reduces lead context pressure)

## Technical Design

### Architecture: Sandwich Structure with Stage 3 Fork

```
Stage 1: Pre-Flight (SHARED)
├── existing logic unchanged
├── ADD: --scoped / --exploratory argument parsing
├── ADD: mode detection (env var check + user choice per FR2)
├── Token budget check (warn if >30% for --scoped, >25% for --exploratory)
└── Load subagent-prompting skill + role references

Stage 2: Project SME (SHARED — Task tool, solo)
├── identical in both modes
├── uses references/role-project-sme.md (unchanged)
├── output: logs/brainstorm/{topic-slug}/01-project-sme.md
└── Token budget check

Stage 3A: Role Analysis — Task Tool Mode (--scoped)
├── existing Stage 3 logic, unchanged
├── PM + Architect + Dev Lead parallel (3 Task tool calls)
├── outputs: 02-product-manager.md, 03-technical-architect.md, 04-development-lead.md
└── Token budget check (checkpoint if >55%)

Stage 3B: Role Analysis — Agent Teams Mode (--exploratory)
├── Lead enters delegate mode
├── Spawn 3 teammates: Product & Delivery Lead, Architect, Critical Analyst
├── Each prompt includes dual-output contract (logs + mailbox)
├── Peer debate: selective challenge (broadcast summary, respond to disagreements)
├── Critic active throughout with deferred-verdict mitigation
├── AT failure recovery: fall back to Stage 3A for unfinished roles
├── outputs: 02-product-delivery-lead.md, 03-technical-architect.md, 04-critical-analyst.md
└── Token budget check (checkpoint if >45%)

Stage 4: Critical Analyst (--scoped ONLY)
├── existing Stage 4 logic, unchanged
├── receives ALL prior outputs
├── ADD: "Problem Validation" section (FR5 enhancement)
├── output: 05-critical-analyst.md
└── Token budget check
NOTE: In --exploratory, Critic is a Stage 3B teammate — Stage 4 is skipped

Stage 5: Synthesis (SHARED, mode-aware)
├── Read all log files in output directory (mode-agnostic glob)
├── Verify expected file count: 5 for --scoped, 4 for --exploratory
├── Write synthesis to logs/brainstorm/{topic-slug}/synthesis.md
├── mode field in synthesis YAML header
├── AskUserQuestion approval gate
├── Critical Evaluation Gate (unchanged — uses Task tool for follow-ups)
└── Token budget check (must be <65%)

Stage 6: Diagnostics (SHARED, extended)
├── existing fields unchanged
└── ADD: AT-specific metrics block (only in --exploratory)
```

## Session 80 Learnings (from P5.13 TC3 — apply to P5.15)

### Mandatory Execution Checklist Must Be at Top of Skill

The completion checklist at the bottom of plan-creation was ignored by Claude — it committed to an execution plan before reaching it. **The fix**: add a "Mandatory Execution Checklist (BINDING)" section near the top of SKILL.md (after Dependencies, before Usage), mirroring the create-skill pattern. This checklist must include:
- Mode detection (env var check — MUST check, no exceptions)
- AskUserQuestion for mode choice (MUST NOT default silently)
- All stage gates

The bottom checklist was removed entirely (redundant). Apply the same pattern to brainstorm.

### ANSI Color Banners Are Cosmetic-Only

RED/AMBER ANSI color codes (`\033[38;2;...`) are not rendered in Claude Code terminal output. Banner content is correct but color is cosmetic. Do not spend effort debugging color rendering — mark as known limitation.

### AT Mitigation Patterns (Proven in TC3)

These 3 patterns must be included in all AT teammate prompts:
1. **CC-to-lead**: "After any peer DM exchange, also send a 1-line summary to the lead"
2. **Task list coordination**: Use shared task list for progress tracking, not activity detection
3. **Rendezvous gate**: Lead MUST NOT call shutdown until WORK COMPLETE received from ALL teammates AND all shared tasks terminal

### Sonnet Agents Viable for AT

TC3 used Sonnet agents (user's choice) and produced high-quality output with peer debate. Offer Opus/Sonnet choice in AT confirmation flow (same as plan-creation).

### AT Value Proposition Validated

Peer debate produced structural changes that sequential mode could not (new WPs, changed technical decisions, reconciled contradictions in real-time). See `docs/AT-v-SequentialTask-comparison.md`. This validates the --exploratory mode concept.
