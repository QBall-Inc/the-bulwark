# P0.6-8: Test Audit Skills - Consolidated Task Brief

## Overview

This brief covers three related skills that form the Test Audit pipeline:

| Task | Skill | Role | Model | User-Invocable |
|------|-------|------|-------|----------------|
| P0.6 | `test-classification` | Surface classification and triage | Haiku | No |
| P0.7 | `mock-detection` | Deep mock appropriateness analysis | **Sonnet** | No |
| P0.8 | `test-audit` | Entry point + orchestration + synthesis | Sonnet | **Yes** |

---

## Architecture Decision

### Skills as Prompt Templates

P0.6 and P0.7 are **internal skills** that define prompt templates for general-purpose sub-agents. They do NOT have an `agent:` field - they are reference materials that the orchestrator loads and uses to construct 4-part prompts.

### Main Context Orchestration

P0.8 (`test-audit`) is the **user-facing entry point** that provides orchestration instructions. When invoked via `/test-audit` or by pipeline hook:

1. Orchestrator (Opus) loads P0.8 skill and follows its instructions
2. Instructions direct orchestrator to spawn general-purpose sub-agents using P0.6/P0.7 as prompt templates
3. Sub-agents (Haiku, Sonnet) perform the actual audit work
4. Orchestrator reads outputs and implements rewrites if needed

### Bias Avoidance

The orchestrator (Opus) does NOT perform audit/review work. Audit is delegated to:
- **Haiku**: Classification and triage (surface pattern matching)
- **Sonnet**: Mock detection (contextual analysis) and synthesis (prioritization)

Opus only orchestrates (mechanical) and implements fixes (writing code).

### Automatic Rewrite

Rewrite happens automatically when `REWRITE_REQUIRED: true`. This applies to both:
- **Explicit invocation**: `/test-audit tests/`
- **Implicit invocation**: Hook-triggered pipeline

The audit summary is presented before rewrites begin (user has visibility), but no confirmation is required.

## Functional Requirements

### FR1: Test Classification (P0.6)

The skill MUST provide a prompt template that enables a Haiku sub-agent to:
- Accept a test directory or file path as input
- Identify all test files in scope
- Classify each test by category (unit, integration, e2e)
- Identify real integration test indicators
- Identify mock-heavy test indicators
- Output structured YAML classification

### FR2: Mock Detection (P0.7)

The skill MUST provide a prompt template that enables a Haiku sub-agent to:
- Accept classification output from P0.6 as input
- Scan classified tests for T1-T4 rule violations
- Identify high-risk mock patterns (spawn, fs, fetch mocks)
- Document specific line numbers for violations
- Output structured YAML with violations and severity

### FR3: Test Audit Synthesis (P0.8)

The skill MUST:
- Be user-invocable via `/test-audit [path]`
- Provide orchestration instructions for the full pipeline
- Include a synthesis prompt template for Sonnet sub-agent
- Define the `REWRITE_REQUIRED` directive logic
- Trigger automatic rewrite when directive is true

| Rule | Violation Pattern | Severity |
|------|-------------------|----------|

| T2 | Verifying calls not results (`toHaveBeenCalled` without result check) | High |
| T3 | Mocking at integration boundaries (fetch, fs in integration tests) | Critical |
| T4 | No test execution verification | Medium |

## Technical Design

### P0.6: test-classification Skill

**Location:** `skills/test-classification/SKILL.md`

**Model:** Haiku (specified by P0.8 orchestration instructions)

**Role:** Surface-level classification and triage. Identifies which files need deep analysis by P0.7.

**Frontmatter:**
```yaml
---
name: test-classification
description: Prompt template for test classification stage in Test Audit pipeline
user-invocable: false
---
```

#### Classification Logic

1. **Filename pattern (primary):**
   - `*.integration.*` → integration
   - `*.e2e.*` → e2e
   - `*.test.*`, `*.spec.*`, `test_*` → unit (default)

2. **Content validation (secondary):**
   - Check imports for test framework, system modules
   - Detect mock patterns at surface level

3. **Risk detection:**
   - Single file with multiple test types → `risk: test_management`
   - Recommend splitting into separate files

#### `needs_deep_analysis` Triggers (Mock + Integration Mismatch)

| Trigger | Reason |
|---------|--------|
| `.integration.*` file with any `jest.mock()` / `vi.mock()` | Integration test shouldn't mock integration boundaries |
| Unit test with >3 top-level mocks | Unusual mock density |
| Unit test mocking core modules (`spawn`, `fs`, `fetch`, `http`) | Known risky patterns |

#### 4-Part Prompt Template

```markdown
### GOAL
Classify all test files in {target} by type and flag files needing deep analysis.

### CONSTRAINTS
- Do NOT modify any files
- Classify every test file found
- Use filename-first classification (content validates)
- Flag mock+integration mismatches for deep analysis
- Complete within 30 tool calls

### CONTEXT
- Target directory: {target}
- Test file patterns: *.test.*, *.spec.*, test_*, *.integration.*, *.e2e.*
- Classification rules: [see above]
- Deep analysis triggers: [see above]

### OUTPUT
Write to: logs/test-classification-{timestamp}.yaml
Format: [schema below]
Diagnostic: logs/diagnostics/test-classification-{timestamp}.yaml
```

#### Line Counting for Test Effectiveness

P0.6 must count "verification lines" per file for test effectiveness calculation.

**Exclude from count:**
- Comment lines (`//`, `/* */`, `/** */`)
- Import/require statements
- Empty/whitespace lines
- Setup boilerplate (`describe(`, `it(`, `test(`, `beforeEach(`, `afterEach(`, `beforeAll(`, `afterAll(`)

**Include in count:**
- Actual test logic (assertions, function calls, variable assignments within tests)
- Mock setup lines (these will be marked as violation scope by P0.7)

### P0.7: mock-detection Skill

**Location:** `skills/mock-detection/SKILL.md`

**Model:** Sonnet (specified by P0.8 orchestration instructions)

**Scope:** Flagged files only (from P0.6 `needs_deep_analysis: true`)

**Role:** Deep mock appropriateness analysis using call graph tracing to detect violations.

**Frontmatter:**
```yaml
---
name: mock-detection
description: Deep mock appropriateness analysis for Test Audit pipeline
user-invocable: false
---
```

#### Mock Appropriateness Rubric

| Test Type | Expected Mocks (OK) | Inappropriate Mocks (VIOLATION) |
|-----------|---------------------|--------------------------------|
| **Unit** | External deps (DB, HTTP, fs) to isolate unit | Mocking function/module under test (T1) |
| **Integration** | Unrelated systems | Mocking integration boundaries (T3), broken chain (T3+) |
| **E2E** | Almost never | Any mock breaking end-to-end flow |

#### T1-T4 Detection Patterns

| Rule | Pattern | Severity | Detection Logic |
|------|---------|----------|-----------------|
| **T1** | Mocking system under test | Critical | `jest.spyOn(ModuleUnderTest, ...)`, `jest.mock('./file-being-tested')` |
| **T2** | Verifying calls not results | High | `toHaveBeenCalled()` without result assertion on same function |
| **T3** | Mock at integration boundary | Critical | In integration test: `jest.mock('node-fetch')`, `jest.mock('fs')` |
| **T3+** | Broken integration chain | Critical | Call graph shows mock intercepts data between integrated components |

#### Call Graph Analysis Approach

For each flagged file:
1. **Identify test assertions** - What is the test claiming to verify?
2. **Trace data flow** - How does data flow from setup → action → assertion?
3. **Locate mocks** - Where do mocks intercept this flow?
4. **Evaluate appropriateness** - Does mock break what test claims to test?

#### 4-Part Prompt Template

```markdown
### GOAL
Analyze flagged test files for T1-T4 violations using mock appropriateness rubric and call graph analysis.

### CONSTRAINTS
- Do NOT modify any files
- Only analyze files with needs_deep_analysis: true from classification
- Use call graph analysis to detect broken integration chains
- Provide full context for each violation (line, snippet, reason, fix)
- Complete within 50 tool calls

### CONTEXT
- Classification output: {classification_yaml}
- Files to analyze: [list of needs_deep_analysis: true files]
- Mock appropriateness rubric: [see above]
- T1-T4 detection patterns: [see above]

### OUTPUT
Write to: logs/mock-detection-{timestamp}.yaml
Format: [schema below]
Diagnostic: logs/diagnostics/mock-detection-{timestamp}.yaml
```

#### Violation Scope Tracking

P0.7 must track the **scope** of each violation - not just the violation line, but all lines affected by it.

**Scope calculation:**
- For mock violations (T1, T3): All lines that use the mock (assertions, calls depending on mock)
- For T2 violations: The assertion line itself
- For T3+ (broken chain): All lines using the mocked/incorrect data

This enables accurate test effectiveness calculation in P0.8.

### P0.8: test-audit Skill

**Location:** `skills/test-audit/SKILL.md`

**Model:** Sonnet (for synthesis stage)

**Role:** User-facing entry point with orchestration instructions and two-gate REWRITE_REQUIRED logic.

**Frontmatter:**
```yaml
---
name: test-audit
description: Audit test suite quality and identify T1-T4 violations
user-invocable: true
---
```

#### Two-Gate REWRITE_REQUIRED Logic

| Gate | Condition | Trigger |
|------|-----------|---------|
| **Gate 1: Impact** | Any P0 violation (false confidence) | REWRITE_REQUIRED regardless of % |
| **Gate 2: Threshold** | P1 violations AND test effectiveness < 95% | REWRITE_REQUIRED |
| **Advisory** | P2 only OR P1 with effectiveness >= 95% | Recommendations only, no auto-rewrite |

**Priority Logic (Impact-Based):**

| Priority | Impact | Rules | Description |
|----------|--------|-------|-------------|
| **P0** | False confidence | T1, T3+ | Test passes but shouldn't - mocks hide real failures |
| **P1** | Incomplete verification | T2, T3 | Test runs real code but doesn't verify results properly |
| **P2** | Pattern issues | Other | Style, organization, minor issues |

#### Orchestration Instructions

```markdown
## Orchestration Instructions

When this skill is loaded, follow these steps:

### Step 1: Resolve Target

IF $ARGUMENTS provided:
    target = $1
ELSE:
    Look for test files in recent conversation context
    IF found: target = that path
    ELSE: Ask user for target path

### Step 2: Classification Stage (Haiku)

1. Load `test-classification` skill
2. Construct 4-part prompt using the skill's template
3. Spawn sub-agent:
   Task(subagent_type="general-purpose", model="haiku", prompt=...)
4. Read output from logs/test-classification-{timestamp}.yaml

### Step 3: Detection Stage (Sonnet)

1. Load `mock-detection` skill
2. Construct 4-part prompt using the skill's template
3. Include classification output in CONTEXT
4. Spawn sub-agent:
   Task(subagent_type="general-purpose", model="sonnet", prompt=...)
5. Read output from logs/mock-detection-{timestamp}.yaml

### Step 4: Synthesis Stage (Sonnet)

1. Construct synthesis prompt (template below)
2. Include classification + detection in CONTEXT
3. Spawn sub-agent:
   Task(subagent_type="general-purpose", model="sonnet", prompt=...)
4. Read output from logs/test-audit-{timestamp}.yaml

### Step 5: Present Summary

Display audit summary to user before proceeding.

### Step 6: Evaluate REWRITE_REQUIRED (Two-Gate)

Gate 1 (Impact):
  IF any P0 violations exist:
    REWRITE_REQUIRED = true

Gate 2 (Threshold):
  ELSE IF P1 violations exist:
    IF any file has test_effectiveness < 95%:
      REWRITE_REQUIRED = true
    ELSE:
      REWRITE_REQUIRED = false (advisory only)

Advisory:
  ELSE (P2 only):
    REWRITE_REQUIRED = false
    Provide recommendations

### Step 7: Rewrite (If Required)

IF REWRITE_REQUIRED == true:
    For each file in files_to_rewrite (ordered by priority):
        Implement rewrite using direction from audit report
        Follow T1-T4 rules strictly
ELSE:
    Report recommendations without auto-rewrite
```
