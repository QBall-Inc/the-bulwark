# P6.9: Add Persistent Memory to Sub-Agents

## Context

Claude Code 2.1.33 introduced `memory` frontmatter for custom sub-agents, giving them persistent directories that survive across conversations. Currently, Bulwark sub-agents are fully stateless - each invocation starts fresh with only the prompt context from the orchestrator. This means agents can't learn from their own past runs, remember common patterns in this codebase, or reference their own prior log outputs.

Adding `memory: project` to Bulwark sub-agents and pre-seeding their MEMORY.md files will:
- Give agents awareness of their own log file patterns (so they can reference past work)
- Build institutional knowledge over time (common bugs, codebase patterns, quality gate outcomes)
- Reduce prompt size needed from the orchestrator (agents already know the basics)

### How `memory` Works (Claude Code 2.1.33+)

- Add `memory: project` to agent frontmatter
- Creates `.claude/agent-memory/<agent-name>/` directory
- First 200 lines of `MEMORY.md` in that directory are **auto-injected into the sub-agent's system prompt**
- Read, Write, Edit tools are **auto-enabled** so the agent can manage its memory files
- Agent can create additional topic files and link from MEMORY.md (same pattern as main-thread auto-memory)

### Scope Values

| Scope | Storage Path | Use Case |
|-------|-------------|----------|
| `user` | `~/.claude/agent-memory/<name>/` | Cross-project learnings |
| `project` | `.claude/agent-memory/<name>/` | Project-specific, version-controllable |
| `local` | `.claude/agent-memory-local/<name>/` | Project-specific, not committed |

## Functional Requirements

### FR1: Add memory frontmatter to 5 pipeline sub-agents

| Agent | Scope | Rationale |
|-------|-------|-----------|
| `bulwark-implementer` | `project` | Remembers implementation patterns, quality gate outcomes, file conventions |
| `bulwark-issue-analyzer` | `project` | Remembers past bugs, root cause patterns, codebase hotspots |
| `bulwark-fix-validator` | `project` | Remembers validation patterns, common failure modes, test infrastructure |
| `bulwark-standards-reviewer` | `project` | Remembers common violations, project-specific conventions |
| `test-validator` | `project` | Remembers test infrastructure, fixture patterns |

**Excluded**: `file-counter`, `code-analyzer` (test/demo agents), `statusline-setup` (built-in)

### FR2: Pre-seed MEMORY.md for each agent

Each MEMORY.md contains:

1. **Identity & Role** - One-liner about what this agent does in the pipeline
2. **Output Conventions** - Exact log file paths and patterns this agent writes to
3. **Pipeline Context** - Which pipeline stages this agent participates in, upstream/downstream agents
4. **Self-Maintenance Instructions** - Guidance to update memory with patterns discovered during runs
5. **Patterns Learned** - Empty section that grows over time

### FR3: Generic reusable template

Create `lib/templates/agent-memory-seed.md` with placeholder sections usable in any project.

## Technical Design

### Frontmatter Change (one line per agent)

```yaml
---
name: bulwark-implementer
memory: project
# ... rest of existing frontmatter
---
```

### Pre-seeded MEMORY.md Structure

Example for `bulwark-implementer`:

```markdown
# Bulwark Implementer Memory

## My Role
Code-writing agent in the Bulwark pipeline. I implement fixes and features following Bulwark standards with quality enforcement at every step.

## My Output Paths
- Implementation logs: `logs/implementer-{id}-{YYYYMMDD-HHMMSS}.yaml`
- Diagnostics: `logs/diagnostics/bulwark-implementer-{YYYYMMDD-HHMMSS}.yaml`
- To find my past work: `ls logs/implementer-*.yaml`

## Pipeline Context
- **Fix Validation Pipeline**: I am Stage 2 (FixWriter). I receive a debug report from bulwark-issue-analyzer (Stage 1) at `logs/debug-reports/`. After me, bulwark-fix-validator (Stage 5) validates my fix.
- **New Feature Pipeline**: I am Stage 3 (Implementer). I receive a design doc from the orchestrator.
- **Quality gate**: I MUST call `implementer-quality.sh` via Bash after every Write/Edit operation.

## Self-Maintenance
After completing a task, update this memory with:
- Codebase patterns discovered (file organization, naming conventions, common utilities)
- Quality gate outcomes (what typically fails, what passes)
- Recurring issues in this codebase

## Patterns Learned
(This section grows over time as I discover codebase patterns)
```

### Gitignore Strategy

**Do NOT gitignore** the agent-memory directories:
- Pre-seeded content is part of agent configuration (must be version-controlled)
- Agent self-updates represent institutional knowledge worth preserving
- The 200-line MEMORY.md limit keeps files small and manageable
- Team members benefit from accumulated agent knowledge

### Generic Template (`lib/templates/agent-memory-seed.md`)

```markdown
# {Agent Name} Memory

## My Role
{One-line description of what this agent does and its purpose in the workflow}

## My Output Paths
{List exact log file paths and patterns this agent writes to}
- Primary output: `{path/pattern}`
- Diagnostics: `{path/pattern}`
- To find past work: `{ls command}`

## Pipeline Context
{Describe which pipelines/workflows this agent participates in}
- Upstream: {what feeds into this agent}
- Downstream: {what consumes this agent's output}
- Quality gates: {any quality checks this agent must pass}

## Self-Maintenance
After completing a task, update this memory with:
- Patterns discovered in the codebase
- Common issues and their solutions
- Infrastructure or tooling observations

## Patterns Learned
(This section grows over time)
```
