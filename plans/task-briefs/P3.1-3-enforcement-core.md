# P3.1-3: Enforcement Core - Consolidated Task Brief

## Overview

This brief covers three related components that form the CORE layer of Bulwark's Defense-in-Depth architecture:

| Task | Component | Role | Location |
|------|-----------|------|----------|
| P3.1 | Justfile Templates | Standardized `just` recipes for quality checks | `lib/templates/justfile-{lang}.just` |
| P3.2 | Scaffold Command | Project onboarding automation | `skills/bulwark-scaffold/SKILL.md` |
| P3.3 | enforce-quality.sh | Quality gate script called by hooks | `scripts/hooks/enforce-quality.sh` |

---

## Architecture Context

### Defense-in-Depth Positioning

```
┌─────────────────────────────────────────────────────────────┐
│                    OUTER RING: HOOKS                        │ ← P3.4-5 (next brief)
│  PostToolUse enforcement - lint/typecheck errors blocked    │
├─────────────────────────────────────────────────────────────┤
│                  MIDDLE RING: SUB-AGENTS                    │ ← P0, P1 (complete)
│  Specialists review in isolated context, return via logs    │
├─────────────────────────────────────────────────────────────┤
│                   INNER RING: SKILLS                        │ ← P0, P2 (complete)
│  Heuristics loaded on-demand via progressive disclosure     │
├─────────────────────────────────────────────────────────────┤
│                    CORE: EXECUTION                          │ ← P3.1-3 (THIS BRIEF)
│  `just` command runner for deterministic invocation         │
└─────────────────────────────────────────────────────────────┘
```

### Dependency Flow

```
P3.1 (Justfile templates)
      ↓
P3.2 (scaffold command) ──────┐
      ↓                       │
P3.3 (enforce-quality.sh) ←───┘
      ↓
P3.4 (Global hooks) ← calls enforce-quality.sh
```

## Design Decisions

### DD1: Strict Enforcement

enforce-quality.sh uses **Exit 2 to block** on any lint or typecheck failure. This is a strict mode with no advisory warnings.

**Rationale:** Quality gates should be hard blockers. Code that doesn't pass typecheck or lint shouldn't proceed.

### DD2: Atomic Recipes with Aggregate Alias

Justfile templates provide **separate recipes** for typecheck, lint, and test, plus an aggregate `ci` recipe (with `check` alias for manual invocation).

**Rationale:** Atomic recipes allow enforce-quality.sh to report specific failures. The `ci`/`check` aggregate provides a single command for running all checks in sequence.

### DD3: Reuse Language Detection

Scaffold command reuses **bulwark-verify's language detection** pattern.

**Rationale:** Already validated in P2.3. Search from current directory upward for manifest files.

### DD4: Skip-by-Default Conflict Handling

Scaffold skips existing Justfile by default. User must provide `--force` to overwrite (creates backup).

**Rationale:** Justfile is critical infrastructure. Overwriting without consent risks data loss.

## Functional Requirements

### FR1: Justfile Templates (P3.1)

Templates MUST provide these recipes:

| Recipe | Purpose | Exit Behavior |
|--------|---------|---------------|
| `typecheck` | Run type checker | 0=success, 1=type errors |
| `lint` | Run linter | 0=success, 1=lint errors |
| `build` | Compile/build project | 0=success, 1=build errors |
| `test` | Run tests | 0=success, 1=test failures |
| `ci` | Aggregate: typecheck → lint → build → test | 0=all pass, 1=any fail |
| `check` | Alias for `ci` (manual invocation) | Same as `ci` |
| `fix` | Auto-fix linting issues | Best effort |
| `default` | Show available recipes | `just --list` |

Templates MUST support:
- Node (TypeScript/JavaScript)
- Python
- Rust
- Generic (bash fallback)

### FR2: Scaffold Command (P3.2)

The command MUST:
- Detect project language from manifest files
- Generate appropriate Justfile from templates
- Create logs/ directory structure
- Update .gitignore with Bulwark patterns
- Handle existing file conflicts gracefully

### FR3: enforce-quality.sh (P3.3)

The script MUST:
- Run `just typecheck` and `just lint`
- Exit 2 on any failure (blocking for hooks)
- Truncate output to prevent context window bloat
- Work with any language that has Justfile recipes

## Technical Requirements

### TR1: Justfile Syntax Standards

Based on research, templates MUST use:

```justfile
# Shell configuration for cross-platform compatibility
set shell := ["bash", "-cu"]
set windows-shell := ["powershell.exe", "-c"]

# Export PATH for local tools (node_modules/.bin, .venv/bin)
export PATH := "./node_modules/.bin:" + env_var("PATH")

# Atomic recipes (not combined)
typecheck:
    @echo "Running typecheck..."
    tsc --noEmit

lint:
    @echo "Running linter..."
    eslint . --max-warnings 0

test:
    @echo "Running tests..."
    npm test

# Aggregate recipe with dependencies
ci: typecheck lint test
    @echo "✓ All quality checks passed"
```

**Key patterns:**
- Use `@` prefix for informational output (suppress command echo)
- Use `-` prefix only for truly optional operations (cleanup)
- Shebang recipes (`#!/usr/bin/env bash`) for complex logic
- Export PATH to include local tool binaries

### TR2: Language-Specific Commands

Based on research:

| Language | Typecheck | Lint | Build | Test |
|----------|-----------|------|-------|------|
| Node | `tsc --noEmit` | `eslint . --max-warnings 0` | `npm run build --if-present` | `npm test` |
| Python | `mypy .` | `ruff check .` | `pip install -e . --quiet` or no-op | `pytest` |
| Rust | `cargo check --all-targets` | `cargo clippy --all-targets -- -D warnings` | `cargo build` | `cargo test` |
| Generic | (no-op or custom) | (no-op or custom) | (no-op or custom) | (no-op or custom) |

**Exit code patterns:**
- 0 = success
- 1 = tool-specific failures
- 2+ = configuration errors

### TR3: Scaffold Language Detection

Detection order (first match wins):

| Check | Result | Manifest |
|-------|--------|----------|
| `package.json` exists | Node | `package.json` |
| `pyproject.toml` OR `setup.py` exists | Python | `pyproject.toml` |
| `Cargo.toml` exists | Rust | `Cargo.toml` |
| None of the above | Generic | - |

Search from current directory upward to project root. Stop at first manifest match.

### TR4: Scaffold File Generation

**Justfile:**
- Location: Project root
- Conflict: Skip by default, `--force` creates backup

**logs/ directory:**
```
logs/
logs/diagnostics/
logs/validations/
logs/debug-reports/
```
Each with `.gitkeep` file.

**.gitignore update:**
```gitignore
# Bulwark logs
logs/*.yaml
logs/*.log
logs/diagnostics/*.yaml
logs/validations/*.yaml
logs/debug-reports/*.yaml
!logs/.gitkeep
!logs/*/.gitkeep
```

Idempotent: Check for `# Bulwark logs` comment before adding.

### TR5: enforce-quality.sh Behavior

```bash
#!/bin/bash
# enforce-quality.sh - Quality gate for PostToolUse hooks

# Run typecheck
if ! just typecheck 2>&1 | head -100; then
    echo "::error::Typecheck failed"
    exit 2
fi

# Run lint
if ! just lint 2>&1 | head -100; then
    echo "::error::Lint failed"
    exit 2
fi

# Run build (catches compilation errors)
if ! just build 2>&1 | head -100; then
    echo "::error::Build failed"
    exit 2
fi

echo "::info::Quality checks passed"
exit 0
```

**Exit codes:**
- 0 = All checks passed (continue)
- 2 = Quality gate failed (block)

**Output truncation:** Limit to 100 lines to prevent context bloat.

**Note:** Tests are NOT run by enforce-quality.sh. Testing is handled by suggest-pipeline.sh which suggests appropriate pipelines (Code Review, Test Audit, etc.) based on change size.

### TR6: Hook Chaining Order

PostToolUse hooks execute in this order:

```
PostToolUse (Edit|Write)
        │
        ▼
┌─────────────────────────────┐
│  1. enforce-quality.sh      │  ← GATE (Exit 2 blocks)
│     typecheck → lint → build│
└──────────────┬──────────────┘
               │ (if exit 0)
               ▼
┌─────────────────────────────┐
│  2. suggest-pipeline.sh     │  ← ADVISORY (suggests next steps)
│     - Checks change size    │
│     - Suggests: Code Review,│
│       Test Audit, etc.      │
│     - Pipelines handle tests│
└─────────────────────────────┘
```

**Rationale:**
- enforce-quality.sh is the gate - fast checks before expensive operations
- suggest-pipeline.sh is advisory - suggests testing via pipelines for significant changes
- Tests run in pipeline context, not on every Edit/Write (too slow)

#### Node Template (justfile-node.just)

```justfile

set windows-shell := ["powershell.exe", "-c"]

    @echo "Running typecheck..."
    tsc --noEmit

# Linting with ESLint (fail on any warning)

    @echo "Running linter..."
    eslint . --max-warnings 0

# Build (compile TypeScript, or skip if no build script)
build:
    @echo "Running build..."
    npm run build --if-present

# Test execution (runs on pre-built code)
test:
    @echo "Running tests..."
    npm test

# Auto-fix linting issues
fix:
    @echo "Running auto-fix..."
    eslint . --fix
    @echo "✓ Auto-fix complete"

# CI: Run all quality checks in order
ci: typecheck lint build test
    @echo "✓ All quality checks passed"

# Alias for manual invocation
check: ci
```

## Execution Steps

### Step 1: Detect Project Language

Search from current directory upward for manifest files:

1. Check for `package.json` → **Node**
2. Check for `pyproject.toml` OR `setup.py` → **Python**
3. Check for `Cargo.toml` → **Rust**
4. None found → **Generic**

If `--lang` flag provided, use that instead.

### Step 2: Check for Existing Justfile

```
IF Justfile exists AND NOT --force:
    Print warning: "Justfile already exists. Use --force to overwrite (creates backup)."
    Continue to logs/ and .gitignore (skip Justfile)
ELSE IF Justfile exists AND --force:
    Create backup: Justfile.backup-{YYYYMMDD-HHMMSS}
    Proceed with generation
ELSE:
    Proceed with generation
```

### Step 3: Generate Justfile

Copy template from `$CLAUDE_PLUGIN_ROOT/lib/templates/justfile-{lang}.just` to project root as `Justfile`.

### Step 4: Create logs/ Directory Structure

```bash
mkdir -p logs/diagnostics logs/validations logs/debug-reports
touch logs/.gitkeep logs/diagnostics/.gitkeep logs/validations/.gitkeep logs/debug-reports/.gitkeep
```

### Step 5: Update .gitignore

Check if `.gitignore` exists:
- If exists: Append Bulwark patterns (if not already present)
- If not exists: Create with Bulwark patterns

**Patterns to add:**
```gitignore
# Bulwark logs
logs/*.yaml
logs/*.log
logs/diagnostics/*.yaml
logs/validations/*.yaml
logs/debug-reports/*.yaml
!logs/.gitkeep
!logs/*/.gitkeep
```

**Idempotent check:** Look for `# Bulwark logs` comment before adding.

### Step 6: Generate Hook Configuration (Default)

UNLESS `--no-hooks` flag provided:

Create `.claude/settings.json` (if not exists) or merge hooks into existing:
```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "command": "${CLAUDE_PLUGIN_ROOT}/scripts/hooks/enforce-quality.sh"
      }
    ]
  }
}
```

**Rationale:** Hooks are the core value of Bulwark - enforcement by default. Users who want Justfile without enforcement can opt out with `--no-hooks`.

Write to `logs/scaffold-{YYYYMMDD-HHMMSS}.yaml`:

```yaml

### P3.3: enforce-quality.sh

**Location:** `scripts/hooks/enforce-quality.sh`

```bash
#!/bin/bash
# enforce-quality.sh - Quality gate script for PostToolUse hooks
#
# Called by hooks to enforce typecheck, lint, and build pass before allowing changes.
# Exit codes:
#   0 = All checks passed (continue)
#   2 = Quality gate failed (block)
#
# Note: Tests are NOT run here - they are handled by suggest-pipeline.sh
# which suggests appropriate pipelines (Code Review, Test Audit) for
# significant changes. This keeps the quality gate fast.
#
# Usage: Called by hooks, not directly by users

set -e

# Configuration
MAX_OUTPUT_LINES=100
JUSTFILE_REQUIRED=true

# Color codes (if terminal supports)
RED='\033[0;31m'
GREEN='\033[0;32m'
NC='\033[0m' # No Color

# Check if Justfile exists
if [ ! -f "Justfile" ]; then
    if [ "$JUSTFILE_REQUIRED" = true ]; then
        echo -e "${RED}::error::No Justfile found. Run /bulwark:scaffold to initialize.${NC}"
        exit 2
    else
        echo "::info::No Justfile found, skipping quality checks"
        exit 0
    fi
fi

# Check if just is installed
if ! command -v just &> /dev/null; then
    echo -e "${RED}::error::just command not found. Install from https://just.systems${NC}"
    exit 2
fi

echo "::group::Typecheck"
if ! just typecheck 2>&1 | head -n $MAX_OUTPUT_LINES; then
    echo -e "${RED}::error::Typecheck failed${NC}"
    echo "::endgroup::"
    exit 2
fi
echo -e "${GREEN}✓ Typecheck passed${NC}"
echo "::endgroup::"

echo "::group::Lint"
if ! just lint 2>&1 | head -n $MAX_OUTPUT_LINES; then
    echo -e "${RED}::error::Lint failed${NC}"
    echo "::endgroup::"
    exit 2
fi
echo -e "${GREEN}✓ Lint passed${NC}"
echo "::endgroup::"

echo "::group::Build"
if ! just build 2>&1 | head -n $MAX_OUTPUT_LINES; then
    echo -e "${RED}::error::Build failed${NC}"
    echo "::endgroup::"
    exit 2
fi
echo -e "${GREEN}✓ Build passed${NC}"
echo "::endgroup::"

echo -e "${GREEN}::info::All quality checks passed${NC}"
exit 0
```

**Permissions:** Must be executable (`chmod +x`)
