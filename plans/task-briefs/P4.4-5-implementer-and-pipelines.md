## Hook Validation Results (T-044) - COMPLETED

**Session 41 (2026-02-07)**: Agent-scoped hooks validated across 3 fresh sessions.

### Finding: Agent-Scoped Hooks Do Not Fire (Confirmed Bug)

PostToolUse and Stop hooks declared in agent YAML frontmatter **do not fire** when the agent is spawned via Task tool. This was tested with:
- `code-analyzer.md` with PostToolUse (matcher: Write) and Stop hooks
- Scripts writing breadcrumbs to `/tmp/claude/` and `logs/hooks.log`
- Scripts marked executable (`chmod +x`)
- 3 separate fresh sessions (not mid-session modifications)
- With and without `tools:` restriction in frontmatter

**Evidence**: No breadcrumb files created, no `AGENT-HOOK-TEST` entries in hooks.log, no `agent-hook-debug.log` or `agent-stop-debug.log` files.

**Known GitHub Issues**: #18392 (agent frontmatter hooks not executed for subagents), #19213 (agent frontmatter hooks not executing). #18392 incorrectly closed as duplicate of plugin-specific #17688.

### Finding: Global PostToolUse Hooks DO Fire for Sub-Agent Writes

**Empirically verified** via hooks.log timestamp correlation:
```
[07:41:04Z] SubagentStart: ae14ed3
[02:41:23 ] PostToolUse: enforce-quality.sh triggered for logs/code-analyzer-20260207-120000.yaml
[07:41:25Z] SubagentStop: ae14ed3
```

The global `enforce-quality.sh` hook fires between SubagentStart and SubagentStop, for files written by the sub-agent. This means **quality enforcement works for sub-agent code writes via the existing global hook**.

### Design Pivot

| Original Design | Revised Design |
|-----------------|----------------|
| Agent-scoped PostToolUse hook | Global PostToolUse hook (already works) |
| Agent-scoped Stop hook → `finalize-implementer.sh` | Global SubagentStop → updated `track-pipeline-stop.sh` |
| Agent frontmatter `hooks:` section | No hooks in frontmatter |
| Quality enforcement via agent hook | Quality enforcement via direct `implementer-quality.sh` invocation + `just` self-validation |

### Updated track-pipeline-stop.sh

The existing `track-pipeline-stop.sh` (global SubagentStop hook) will be updated to:
1. Extract `subagent_type` from stdin JSON
2. Search `logs/` for files matching the agent name modified within last 60 seconds
3. If no log found: warn via stderr + log warning to hooks.log
4. User/orchestrator decides whether to re-run or proceed

This replaces `finalize-implementer.sh` with a **generic, agent-agnostic** log verification that benefits ALL sub-agents, not just the implementer.

**Note on hooks (Session 41 pivot)**: No `hooks:` section in frontmatter. Agent-scoped hooks do not fire for sub-agents spawned via Task tool (confirmed bug #18392/#19213). Quality enforcement is provided by:

1. **Direct script invocation** (`implementer-quality.sh`) - agent calls this via Bash after each Write/Edit. Runs typecheck, lint, build on code files. Returns pipeline suggestions the agent logs and includes in summary with MANDATORY language (SA6).
2. **Global SubagentStop hook** (`track-pipeline-stop.sh`) - verifies the agent wrote its required log output. Warns if missing.
3. **Final self-validation** (`just typecheck && just lint`) - agent runs as final check before writing output report.
4. **Global PostToolUse hook** (`enforce-quality.sh`) - still fires for sub-agent writes but output is unactioned (goes to sub-agent context, not orchestrator). Provides defense-in-depth; primary enforcement is via item 1.

### track-pipeline-stop.sh Update

**Location**: `scripts/hooks/track-pipeline-stop.sh` (existing file, to be updated)

**Purpose**: Global SubagentStop hook updated to verify sub-agent log output exists. Replaces the agent-scoped `finalize-implementer.sh` with generic, agent-agnostic log verification that benefits ALL sub-agents.

**Changes from current version**:
1. Extract `subagent_type` from stdin JSON (already parsed but only used for logging)
2. Search `logs/` for files matching the agent type name, modified within last 60 seconds
3. If no log found: warn via stderr + log warning to hooks.log
4. Remove transcript existence check (transcript always exists, created by Claude Code)

**Updated logic**:
```bash
# After existing logging...

# Verify sub-agent wrote expected log output
# Search for files matching agent type name modified in last 60 seconds
LOG_FOUND="false"
if [ "$SUBAGENT_TYPE" != "unknown" ] && [ "$SUBAGENT_TYPE" != "general-purpose" ]; then
    # Custom agents: look for files containing agent type name
    RECENT_LOG=$(find "$LOGS_DIR" -maxdepth 2 -name "*${SUBAGENT_TYPE}*" \
        -newer "$LOGS_DIR/pipeline-tracking.log" -o \
        -name "*${SUBAGENT_TYPE}*" -mmin -1 2>/dev/null | head -1)
    if [ -n "$RECENT_LOG" ]; then
        LOG_FOUND="true"
        echo "[$TIMESTAMP] SubagentStop: Log verified at $RECENT_LOG" >> "$LOGS_DIR/hooks.log"
    else
        echo "[$TIMESTAMP] SubagentStop: WARNING - No log output found for $SUBAGENT_TYPE" >> "$LOGS_DIR/hooks.log"
        echo "WARNING: Sub-agent $SUBAGENT_TYPE ($AGENT_ID) completed without writing expected log to logs/" >&2
    fi
fi
```

**Note on general-purpose agents**: Skip log verification for `general-purpose` type since these are ad-hoc agents that may not follow the naming convention. Custom agents (bulwark-implementer, bulwark-issue-analyzer, etc.) always include their name in log filenames per SA2.

### implementer-quality.sh (New Script)

**Location**: `scripts/hooks/implementer-quality.sh`

**Purpose**: Agent-invoked quality gate + pipeline suggestion script. Unlike `enforce-quality.sh` (which reads stdin JSON from the hook system), this script accepts a file path as a CLI argument so the implementer agent can call it directly via Bash and act on its output.

**Interface**:
```bash
# Called by the implementer agent after each Write/Edit
implementer-quality.sh <file_path>
```

**Behavior**:
1. **Phase 1: Quality checks** (code files only)
   - Same checks as enforce-quality.sh: `just typecheck`, `just lint`, `just build`
   - On failure: outputs error details to stdout (agent reads and self-corrects)
   - Exit code 1 on failure, 0 on success
2. **Phase 2: Pipeline suggestion** (all file types)
   - Same classification logic as suggest-pipeline.sh (file type detection, threshold)
   - Since no hook JSON is available, reads the file directly to count lines as change size proxy
   - Outputs pipeline suggestion in plain text format (not hook JSON):
   ```
   QUALITY: PASSED
   PIPELINE: Code Review
   TARGET: src/user-service.ts
   LINES: 45
   REASON: Existing code file modified, 45 lines
   ```

**Key differences from enforce-quality.sh**:

| Aspect | enforce-quality.sh | implementer-quality.sh |
|--------|-------------------|----------------------|
| Input | stdin JSON from hook system | File path as CLI argument |
| Invoked by | Global PostToolUse hook | Agent via Bash |
| Output format | Hook JSON (decision/reason/hookSpecificOutput) | Plain text (QUALITY/PIPELINE/TARGET/LINES/REASON) |
| Change size | Exact diff from hook JSON | Total file line count (proxy) |
| Who acts on output | Sub-agent context (unactioned) | Agent directly (actioned) |

**Note on double execution**: When the implementer writes a file, both scripts run: `implementer-quality.sh` (called by agent, output actioned) and `enforce-quality.sh` (fired by global hook, output unactioned in sub-agent context). This is an accepted tax for quality assurance.
