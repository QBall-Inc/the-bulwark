## Overview

This brief covers two related components that form the OUTER RING of Bulwark's Defense-in-Depth architecture:

| Task | Component | Role | Location |
|------|-----------|------|----------|
| P3.4 | Global Hooks | Plugin-level hook configuration | `hooks/hooks.json` |
| P3.5 | SessionStart Protocol | Governance injection at session start | `scripts/hooks/inject-protocol.sh` |

## Design Decisions

### DD1: Plugin Hooks vs Settings Hooks

Hooks migrate from `.claude/settings.json` (project-level) to `hooks/hooks.json` (plugin-level) for distribution.

**Key Difference:** Command paths use `${CLAUDE_PLUGIN_ROOT}` in hooks.json, while scripts continue using `$CLAUDE_PROJECT_DIR` internally for logs and flag files.

**Rationale:** Plugin hooks ship with the plugin; project paths (logs/) stay project-relative.

### DD2: Chained Execution Model

To avoid race conditions and confusing parallel output, `enforce-quality.sh` **chains to** `suggest-pipeline.sh` internally rather than registering both as separate hooks.

**Execution Flow:**
```
PostToolUse (Write|Edit)
        │
        ▼
┌─────────────────────────────────┐
│  enforce-quality.sh             │
│  ┌───────────────────────────┐  │
│  │ Is it a code file?        │  │
│  │ ├─ Yes → typecheck/lint/  │  │
│  │ │        build            │  │
│  │ │   └─ Fail? Exit 2       │  │
│  │ │   └─ Pass? Continue     │  │
│  │ └─ No → Continue          │  │
│  └───────────────────────────┘  │
│            │                    │
│            ▼                    │
│  ┌───────────────────────────┐  │
│  │ Call suggest-pipeline.sh  │  │
│  │ (pass through stdin JSON) │  │
│  └───────────────────────────┘  │
└─────────────────────────────────┘
```

**Benefits:**
- Code files: Quality gate runs first, pipeline suggestion only if quality passes
- Non-code files (docs, config): Quality checks skipped, pipeline suggestion runs directly
- Deterministic sequential execution, no race conditions
- Single hook entry in hooks.json

**Rationale:** Hooks fire in parallel by default. Chaining internally gives us sequential behavior.

### DD3: SessionStart Context Injection (Skill-Based)

SessionStart hook uses `once: true` and exit code 0 to inject governance protocol into Claude's context.

**Special Behavior:** Unlike most hooks where exit 0 stdout goes to verbose mode only, SessionStart exit 0 stdout is added to Claude's **CONTEXT** (visible to Claude, not just user).

**Architecture:**
```
SessionStart hook → inject-protocol.sh → reads skill → stdout to context
                                              ↓
                         skills/governance-protocol/SKILL.md
                         (readable, user-extensible)
```

**Benefits:**
- Governance content in readable markdown (not bash heredoc)
- User-extensible without touching bash scripts
- Consistent with Bulwark's skill-based architecture
- Discoverable alongside other skills

**Rationale:** Ensures Claude sees governance requirements from session start, while keeping the content maintainable and extensible.

### DD4: Graceful Degradation

Hooks should handle missing dependencies gracefully:
- No Justfile → Skip quality checks, warn user
- No `just` command → Skip, suggest installation

**Rationale:** Plugin should work in any project state, guiding users to proper setup.

## Functional Requirements

### FR1: Global Hooks (P3.4)

The hooks.json MUST:
- Define PostToolUse hook for quality enforcement (chains to pipeline suggestion internally)
- Define SubagentStart/Stop hooks for pipeline tracking
- Use `${CLAUDE_PLUGIN_ROOT}` for all command paths
- Include top-level description field

**Hook Events to Configure:**

| Event | Script | Purpose | Exit Behavior |
|-------|--------|---------|---------------|
| PostToolUse (Write\|Edit) | enforce-quality.sh | Quality gate + pipeline suggestion (chained) | 2=block, 0=pass |
| SubagentStart | track-pipeline-start.sh | Log pipeline progress | 0 (logging only) |
| SubagentStop | track-pipeline-stop.sh | Log pipeline completion | 0 (logging only) |

**Note:** `suggest-pipeline.sh` is NOT registered separately in hooks.json. It is called internally by `enforce-quality.sh` after quality checks pass (or are skipped for non-code files).

### FR2: SessionStart Protocol (P3.5)

The SessionStart hook MUST:
- Fire once per session (`once: true`)
- Inject governance protocol into Claude's context
- Log injection event to `logs/hooks.log`
- Handle /clear and /compact gracefully (no re-injection due to once:true)

The governance-protocol skill MUST:
- Contain the governance protocol content in readable markdown
- Be user-extensible (users can add project-specific rules)
- Include a section for project-specific customization
- Be discoverable in `skills/governance-protocol/SKILL.md`

```json

            "timeout": 60
          }
        ]
      }
    ],
    "SubagentStart": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "${CLAUDE_PLUGIN_ROOT}/scripts/hooks/track-pipeline-start.sh"
          }
        ]
      }
    ],
    "SubagentStop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "${CLAUDE_PLUGIN_ROOT}/scripts/hooks/track-pipeline-stop.sh"
          }
        ]
      }
    ],
    "SessionStart": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "${CLAUDE_PLUGIN_ROOT}/scripts/hooks/inject-protocol.sh",
            "timeout": 5,
            "once": true
          }
        ]
      }
    ]
  }

```

**Key Points:**
- Only `enforce-quality.sh` registered for PostToolUse (it chains to `suggest-pipeline.sh` internally)
- Timeout increased to 60s to accommodate quality checks + pipeline suggestion
- Matcher regex is case-sensitive
- `once: true` supported in Claude Code 2.1.1+

### TR2: Environment Variable Usage

| Variable | Scope | Used For |
|----------|-------|----------|
| `${CLAUDE_PLUGIN_ROOT}` | hooks.json commands | Script paths (where scripts live) |
| `$CLAUDE_PROJECT_DIR` | Inside scripts | Log files, flag files, project paths |
| `$CLAUDE_ENV_FILE` | SessionStart/Setup only | Persist environment variables |

**Critical:** Scripts write logs to `$CLAUDE_PROJECT_DIR/logs/`, not `${CLAUDE_PLUGIN_ROOT}/logs/`.

### TR3: Exit Code Semantics

| Exit Code | Meaning | Behavior |
|-----------|---------|----------|
| 0 | Success | JSON parsed for control; stdout to verbose mode (or CONTEXT for SessionStart) |
| 2 | Blocking error | stderr fed to Claude; blocks tool call (PreToolUse/PostToolUse) |
| Other | Non-blocking error | stderr to verbose mode; execution continues |

**SessionStart Exception:** Exit 2 does NOT block session start - only shows warning to user.

2. **track-pipeline-start.sh** (SubagentStart)

   - Same path change

**Transformation:**

**Behavior:**
1. Read `hooks/hooks.json`

**Idempotency:** Safe to run repeatedly - overwrites hooks section only, preserves other settings.

**Workflow:**
```

        ▼ sync-hooks-for-dev.sh
        │
```

### P3.5: inject-protocol.sh

**Location:** `scripts/hooks/inject-protocol.sh`

```bash
#!/bin/bash
# inject-protocol.sh - Inject governance protocol at session start
#
# Hook configuration: once: true (fires once per session)
# Exit code 0: stdout added to Claude's CONTEXT (not just verbose mode)
#
# Architecture: Reads governance content from skill file (not hardcoded)
# This keeps governance readable and user-extensible.
#
# Usage: Called by SessionStart hook, not directly by users

set -euo pipefail

# Get directories
SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
PROJECT_DIR="${CLAUDE_PROJECT_DIR:-.}"
LOGS_DIR="${PROJECT_DIR}/logs"
HOOKS_LOG="${LOGS_DIR}/hooks.log"

# Skill file location (relative to script in plugin structure)
SKILL_FILE="${SCRIPT_DIR}/../../skills/governance-protocol/SKILL.md"

# Ensure logs directory exists
mkdir -p "$LOGS_DIR"

# Log the injection event (for verification)
TIMESTAMP=$(date -Iseconds)
echo "[${TIMESTAMP}] SessionStart: Governance protocol injected (once:true)" >> "$HOOKS_LOG"

# Extract and output skill content (after frontmatter)
# Frontmatter is between first --- and second ---
if [ -f "$SKILL_FILE" ]; then
  # Use awk to extract content after the closing --- of frontmatter
  awk '
    BEGIN { in_frontmatter = 0; frontmatter_done = 0 }
    /^---$/ {
      if (!in_frontmatter) { in_frontmatter = 1; next }
      else { frontmatter_done = 1; next }
    }
    frontmatter_done { print }
  ' "$SKILL_FILE"
else
  echo "Warning: governance-protocol skill not found at ${SKILL_FILE}" >&2
  echo "## Bulwark Governance Protocol"
  echo ""
  echo "Governance skill not found. Please ensure skills/governance-protocol/SKILL.md exists."
fi

exit 0
```

**Permissions:** Must be executable (`chmod +x`)

### A4: enforce-quality.sh (Created in P3.3, Updated in P3.4)

**Location:** `scripts/hooks/enforce-quality.sh`

**Purpose:** Quality gate + pipeline suggestion (chained execution).

**Behavior (P3.4 Update):**
1. Capture stdin JSON at start (`INPUT=$(cat)`)
2. Detect if file is code file (`.ts`, `.js`, `.py`, etc.)
3. **Code files:** Run `just typecheck`, `just lint`, `just build`
   - Exit 2 on any failure (blocking)
   - If all pass, continue to step 4
4. **Non-code files:** Skip quality checks, continue to step 4
5. **Chain to suggest-pipeline.sh:** `echo "$INPUT" | suggest-pipeline.sh`
6. Exit 0 on success (suggest-pipeline.sh output flows through)

**Updated Script Structure:**
```bash
#!/bin/bash
INPUT=$(cat)
FILE_PATH=$(echo "$INPUT" | jq -r '.tool_input.file_path // ""')
SCRIPT_DIR="$(dirname "$0")"

# Phase 1: Quality checks (code files only)
is_code_file() {
  echo "$1" | grep -qiE '\.(ts|js|tsx|jsx|py|go|rs|java|cpp|c|rb|php)$'
}

if is_code_file "$FILE_PATH" && [ -f "Justfile" ]; then
  just typecheck 2>&1 | head -100 || { echo "::error::Typecheck failed" >&2; exit 2; }
  just lint 2>&1 | head -100 || { echo "::error::Lint failed" >&2; exit 2; }
  just build 2>&1 | head -100 || { echo "::error::Build failed" >&2; exit 2; }
fi

# Phase 2: Pipeline suggestion (all file types)
SUGGEST_SCRIPT="${SCRIPT_DIR}/suggest-pipeline.sh"
if [ -x "$SUGGEST_SCRIPT" ]; then
  echo "$INPUT" | "$SUGGEST_SCRIPT"
fi

exit 0
```

**New in P3.4:** Registered in hooks.json with timeout: 60 (accommodates both phases)
