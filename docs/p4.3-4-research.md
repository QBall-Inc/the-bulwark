## 2. Architecture Decision: Custom Agents (Not Skills)

**Unanimous across all 5 perspectives. User confirmed.**

Both P4.3 and P4.4 MUST be custom agents (`agents/*.md`) invoked via Task tool.

| Requirement | Agent Provides | Skill Cannot |
|---|---|---|
| Bias isolation | `context: fork` | Loads into caller context |
| Tool restrictions | `tools:` allowlist in frontmatter | Inherits all caller tools |
| Pipeline staging | Discrete `Task()` invocation + summary return | Augments context, no staging |
| Lifecycle hooks | PostToolUse, Stop hooks in frontmatter | No lifecycle hooks |
| Discrete output | Returns summary, writes to `logs/` | No standalone artifacts |

**First Principles rationale:** An auditor loaded as a skill into the author's context would inherit the author's implementation assumptions and blind spots. `context: fork` provides true independence. For the implementer, agent-scoped PostToolUse hooks are the ONLY mechanism for mandatory quality gates that cannot be bypassed.

## 4. Proposed Agent Designs

### 4.1 ~~P4.3: bulwark-code-auditor~~ DESCOPED

**Status:** Descoped in Session 39 after analysis.

**Reasons for descoping:**
1. `/code-review` skill already provides direct user invocation (all sections by default)
2. Code Review Pipeline provides bias-free isolated review via 4 parallel agents
3. Sub-agents spawned by Task tool already run in isolated context windows - they do NOT inherit the orchestrator's conversation history (confirmed via official docs: "Each subagent runs in its own context window")
4. Single-agent all-sections review is susceptible to Anthropic-documented security bias - models over-index on security findings at the expense of other review aspects
5. Pipeline's separation of concerns (one agent per section) is architecturally superior
6. Adding `context: fork` to the code-review skill is not viable because sub-agents cannot spawn other sub-agents (the pipeline's 4 agents couldn't be spawned from a forked context)

**Effort redirected to:** P4.5 (pipeline-templates parallel execution updates)

### 4.2 P4.4: bulwark-implementer

```yaml
# Frontmatter
name: bulwark-implementer
description: Code-writing agent with mandatory quality enforcement via agent-scoped hooks. Implements fixes and features following Bulwark standards.
model: opus
skills:
  - subagent-prompting
  - subagent-output-templating
  - component-patterns  # Required (not optional) for test scaffolding
tools:
  - Read
  - Grep
  - Glob
  - Write
  - Edit
  - Bash
hooks:
  PostToolUse:
    - matcher: "Edit|Write"
      hooks:
        - type: command
          command: "$CLAUDE_PROJECT_DIR/scripts/hooks/enforce-quality.sh"
  Stop:
    - hooks:
        - type: command
          command: "$CLAUDE_PROJECT_DIR/scripts/agents/finalize-implementer.sh"
```

**Key characteristics:**
- Writes code + tests following Bulwark standards
- Agent-scoped PostToolUse hook calls enforce-quality.sh on Write/Edit operations
- Hook failure handling: max 3 self-correction attempts, then escalate to orchestrator
- Pre-Flight Gate at BEGINNING with MANDATORY language (DEF-P4-005 prevention)
- Explicit output paths (DEF-P4-006 prevention)
- `component-patterns` skill is REQUIRED (not optional) for test scaffolding consistency

**Output:**
- Source code changes (primary deliverable)
- Report: `logs/implementer-{id}-{timestamp}.yaml`
- Diagnostics: `logs/diagnostics/bulwark-implementer-{timestamp}.yaml`
- Summary returned to orchestrator (100-300 tokens)

**Invocation paths:**
- Fix Validation Pipeline Stage 2: Replaces Orchestrator FixWriter
- New Feature Pipeline Stage 3: Replaces generic Opus Implementer
- Task tool: `Task(subagent_type="bulwark-implementer", prompt="...")`

**Input structure (must be specified in 4-part prompt CONTEXT):**
- For fixes: Debug report path, root cause, affected files, fix approach
- For features: Design document, requirements, existing patterns
- Context: fork prevents access to parent variables - ALL required info must be in the prompt

## 7. Open Items (Post-Compact)

### 7.1 RESOLVED: Agent-Scoped Hook Syntax

**Status:** Resolved via official documentation (https://code.claude.com/docs/en/hooks, https://code.claude.com/docs/en/sub-agents)

**Findings:**

Agent-scoped hooks are **fully supported** in YAML frontmatter. All hook events are supported. Syntax uses same structure as settings.json:

```yaml
---
name: agent-name
hooks:
  PostToolUse:
    - matcher: "Edit|Write"
      hooks:
        - type: command
          command: "$CLAUDE_PROJECT_DIR/scripts/hooks/enforce-quality.sh"
  PreToolUse:
    - matcher: "Bash"
      hooks:
        - type: command
          command: "./scripts/validate-command.sh"
---
```

**Hook interaction:** All matching hooks run **in parallel**. Both agent-scoped AND global hooks fire. Identical handlers are deduplicated. Agent-scoped hooks are cleaned up when the agent finishes.

**PostToolUse exit code 2:** Cannot truly "block" (tool already ran). Instead, stderr is shown to Claude as feedback. The agent can then self-correct. For `decision: "block"` via JSON output, this prompts Claude with the reason. This is the correct pattern for P4.4: enforce-quality.sh informs the agent of violations, the agent self-corrects.

**Path references:** `$CLAUDE_PROJECT_DIR` works for project-relative paths. Use quotes for paths with spaces.

### 7.2 RESOLVED: Stop Hooks vs SubagentStop Hooks

**Status:** Resolved via official documentation

**Findings:**

Stop hooks in agent frontmatter ARE supported. **They are automatically converted to `SubagentStop` events at runtime.** This means:

- You declare `Stop` in agent frontmatter → fires as `SubagentStop` when agent completes
- This is the mechanism for agent finalization
- The hook runs in the agent's context before returning to the orchestrator

```yaml
---
name: agent-name
hooks:
  Stop:
    - hooks:
        - type: command
          command: "$CLAUDE_PROJECT_DIR/scripts/agents/finalize-agent.sh"
---
```

**Two distinct mechanisms exist:**
1. **Agent frontmatter `Stop`** → converted to SubagentStop, runs when THIS agent finishes
2. **Global `hooks.json` `SubagentStop`** → runs in PARENT context when ANY matching sub-agent finishes

**For P4.4:** Use frontmatter `Stop` hooks for agent-level finalization (validate outputs, write metrics). Global `SubagentStop` hooks (like `track-pipeline-stop.sh`) remain for pipeline-level tracking.

**Concrete finalization actions for P4.4 (implementer):**
- Validate that expected output files exist (logs/implementer-*.yaml)
- Record execution metrics (duration, files changed, hook retry count)
- Clean up any temporary files

**Backport to P1 agents:** Recommended for consistency but deferred to P6.
