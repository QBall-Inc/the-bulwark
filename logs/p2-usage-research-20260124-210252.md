## Executive Summary

**PRIMARY CONSUMER: Test Rewriting (Test Audit Pipeline)**

The P2.1-P2.3 skills (assertion-patterns, component-patterns, verification-script) are **NOT** used in code-review or fix-validation pipelines. They are designed specifically for **test rewriting** in the Test Audit pipeline when `REWRITE_REQUIRED: true`.

**Finding**: These skills guide the orchestrator (Opus) on **how to rewrite tests** to eliminate T1-T4 violations by providing patterns for real behavior verification instead of mock-based testing.

#### 3. Test Audit Pipeline - PRIMARY CONSUMER

**Pipeline**: `skills/pipeline-templates/references/test-audit.md`

**Stages**:
1. TestClassifier (Haiku) - uses `test-classification`
2. MockDetector (Sonnet) - uses `mock-detection`
3. AuditSynthesizer (Sonnet) - uses `test-audit` synthesis template
4. **Test Rewrite (Orchestrator - Opus)** ← P2 SKILLS USED HERE

**Stage 4: Test Rewrite Details**

**Executor**: Orchestrator (Opus) - NOT a sub-agent

**Conditional**: Only executes if `audit.directive.REWRITE_REQUIRED == true`

**Two-Gate REWRITE_REQUIRED Logic**:
- Gate 1: Any P0 violation (false confidence) → REWRITE
- Gate 2: P1 violations + test effectiveness < 95% → REWRITE
- Advisory: P2 only or P1 >= 95% → recommendations only

**Behavior** (from test-audit.md lines 280-305):
```yaml
rewrites:
  completed:
    - file: tests/proxy.test.ts
      changes: "Replaced mock with real proxy spawn"
      verification: "Run: npm test -- proxy.test.ts"
```

**How P2 Skills Are Used**:

When the orchestrator (Opus) rewrites tests flagged by the audit, it needs guidance on:
1. **What assertions to use** instead of mock verifications (P2.1 assertion-patterns)
2. **How to verify behavior** for different component types (P2.2 component-patterns)
3. **How to create runnable verification scripts** for complex scenarios (P2.3 verification-script)

**Example Rewrite Scenario**:

**Before (T1 violation)**:
```javascript
jest.spyOn(child_process, 'spawn');
// Test just verifies spawn was called
expect(spawn).toHaveBeenCalled();
```

**After (using P2 patterns)**:
- P2.1 (assertion-patterns): Replace mock call assertion with real output verification
- P2.2 (component-patterns): For process spawner, verify with port check
- Result: Real spawn + port check assertion

### Usage Pattern

1. **Test Audit** identifies violations (T1: mocking spawn, T3: mocking HTTP)
2. **Audit Synthesis** outputs `REWRITE_REQUIRED: true` with rewrite guidance
3. **Orchestrator (Opus)** loads P2 skills for rewrite implementation:
   - P2.1: What assertions to use (real output vs mock calls)
   - P2.2: How to verify for this component type (process spawner → port check)
   - P2.3: If complex, generate runnable verification script

## Key Distinctions

### P2.3 verification-script vs Fix Validator's Generated Scripts

| Aspect | P2.3 verification-script | Fix Validator Strategy 3 |
|--------|-------------------------|--------------------------|
| **Purpose** | Guide test rewrites to eliminate T1-T4 violations | Validate a fix when native runners fail |
| **Consumer** | Test Audit pipeline Stage 4 (rewrite) | Fix Validation pipeline Stage 4 (validate) |
| **Context** | "How to test this component correctly" | "How to verify this specific fix" |
| **Output** | Runnable script as canonical test pattern | Temporary script for validation execution |
| **Persistence** | Becomes the new test | Deleted after execution |
| **Skills Used** | P2.1 + P2.2 patterns | Ad-hoc generation, no skill reference |

**Conclusion**: These are DIFFERENT use cases, both generate scripts, but for different purposes.

## Recommendations for Task Brief

### Consolidation Strategy

**Recommendation**: Consider consolidating P2.1-P2.3 into a **single composite skill** or keeping them separate with clear usage examples.

**Option A: Single Composite Skill**
- Name: `test-rewrite-patterns`
- Combines assertion patterns + component patterns + verification script generation
- Simpler to reference in Test Audit Stage 4

**Option B: Keep Separate (Current Plan)**
- P2.1: atomic (assertion patterns)
- P2.2: atomic (component patterns)
- P2.3: composite (orchestrates P2.1 + P2.2)
- More modular, allows independent loading

### Scope Clarity

The task briefs should clarify:

1. **NOT for code review** - Code review uses `code-review` skill sections
2. **NOT for fix validation** - Fix validator generates ad-hoc validation scripts
3. **NOT for test writing** - TestWriter follows T1-T4 rules directly
4. **FOR test rewriting** - When Test Audit identifies violations and triggers rewrites

## Summary

**PRIMARY CONSUMER**: Test Audit Pipeline Stage 4 (Test Rewrite)

**SPECIFIC USAGE**: When `REWRITE_REQUIRED: true`, the orchestrator (Opus) loads P2 skills to guide rewriting tests that have T1-T4 violations.

**NOT USED BY**:
- Code Review pipeline
- Fix Validation pipeline (TestWriter stage)
- bulwark-issue-analyzer agent
- bulwark-fix-validator agent (uses ad-hoc scripts instead)

**KEY INSIGHT**: P2 skills are pedagogical guides for the orchestrator on "how to test correctly" when rewriting violating tests. They encode patterns for real behavior verification that replace mock-based testing.
