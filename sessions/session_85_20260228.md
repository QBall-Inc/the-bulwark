## Technical Decisions

### Timeout values are in seconds — Bulwark hooks.json needs correction
- **Decision**: Change 60000→60, 30000→30, 5000→5 in both `hooks/hooks.json` and `.claude/settings.json`
- **Rationale**: Cross-referenced with lazyptc-mcp (30, 10) and clear-framework (60, 30) configs. Official docs confirm default 600 seconds.
- **Impact**: Action item for P7.1. Current dev config still works because hooks complete well within the absurd timeout.

### once:true is valid but not needed for Bulwark
- **Decision**: Do not add `once: true` to SessionStart hook
- **Rationale**: `once: true` is a valid skills-only field that removes the hook after first fire. Bulwark's governance injection SHOULD re-fire after `/clear` and compaction to maintain context. Using `once: true` would weaken governance.
- **Impact**: No change to hooks.json. Research agent was wrong — corrected via docs cross-reference.

### Copy-on-init is the only viable governance delivery model
- **Decision**: Rules.md copied to `.claude/rules/` during init, not @imported from plugin path
- **Rationale**: `${CLAUDE_PLUGIN_ROOT}` does not expand in Markdown files (GitHub #9354 OPEN). `.claude/rules/` auto-loads all .md files recursively with path-scoped frontmatter support.
- **Impact**: Init script design confirmed. Version header in copied files enables drift detection.

### LSP setup added as P6.11 with init integration
- **Decision**: Create setup-lsp skill (P6.11) and integrate as recommended optional step in P7.1 init
- **Rationale**: LSP replaces grep/glob searches (30-60s) with semantic intelligence (~50ms). The `ENABLE_LSP_TOOL=1` flag discovery + troubleshooting guide makes it viable. Fits Bulwark's quality enforcement principles.
- **Impact**: P7.1 now depends on P6.11. Init pipeline: mandatory (governance) → recommended (LSP) → optional (Justfile, assets).

### Task brief approach for setup-lsp (not draft skill)
- **Decision**: Created task brief to feed into `/create-skill` pipeline next session
- **Rationale**: create-skill handles interview, classification, template selection, validation. Draft skill = double work.
- **Impact**: Next session can invoke `/create-skill` with task brief as context input.

## Learnings

- **Cross-reference over web search for verified configs**: The 4 Sonnet research agents got 2 out of 4 findings wrong (once:true, timeout units) that were trivially resolved by reading existing plugin configs in the workspace. Reference projects (lazyptc-mcp, clear-framework) are more reliable than web searches for Claude Code plugin behavior.
- **Slimmed CLAUDE.md template confirms design direction**: User's template (~50 lines) validates the architectural bet — minimal CLAUDE.md with @Rules.md import, session protocols, project assets, modes of operation. No embedded rules.
- **Init pipeline has natural priority tiers**: mandatory (governance files → .claude/rules/) → recommended (LSP setup) → optional (Justfile, project assets). This emerged from combining Finding 7 (zero-init enforcement value) with the LSP discovery.
- **LSP viability unlocked by hidden flag**: `ENABLE_LSP_TOOL=1` is undocumented (community-discovered via #15619) but makes the entire LSP stack work. Without knowing this flag, LSP appears broken. This is exactly the kind of gotcha a setup skill should handle.
- **Rules needed to enforce LSP preference**: Without explicit CN rules, Claude defaults to training-pattern text search (Grep/Glob) even when LSP is configured. Proposed CN1/CN2 rules for Rules.md. Note: Claude Code's Grep tool already uses ripgrep internally — no separate rg install needed for in-session search.
