## Technical Decisions

### Three AT mitigation patterns for Stage 3B
- **Decision**: Apply CC-to-lead (Pattern A), task list coordination (Pattern B), and rendezvous gate (Pattern C) to all AT teammate spawn prompts
- **Rationale**: TC2 revealed delegate-mode lead has zero visibility into peer DM traffic and teammate activity state. These are architectural constraints of AT, not configurable.
- **Impact**: Stage 3B design now includes explicit shutdown gate (WORK COMPLETE from all teammates + all tasks terminal before requestShutdown)

### AT confirmation flow with model class choice
- **Decision**: RED banner (#FF9A96) + AskUserQuestion (Opus/Sonnet/Switch to Task tool) before AT spawn; AMBER banner (#FFF4B0) after completion
- **Rationale**: AT is experimental, token-expensive, and can exhibit coordination issues. Users need upfront warning, escape hatch, and cost control (Sonnet option). Colors match ez-statusline for consistency.
- **Impact**: AskUserQuestion is blocking (no timeout), so AT requires explicit user confirmation to proceed

### Workpackage single-session sizing
- **Decision**: Every WP must be independently implementable/verifiable AND fit within one 200K token session. `estimated_sessions` is always 1; confidence indicates fit tightness.
- **Rationale**: Multi-session WPs are ambiguous — they don't map cleanly to Claude Code's session model. Single-session WPs make total effort = WP count minus parallel savings.
- **Impact**: Eng Lead agent updated. Future plans will have more granular WPs. QA/Critic can stress-test Low-confidence WPs.

### Plan output location: `plans/{slug}/plan.md`

- **Rationale**: Groups plan with future associated artifacts, avoids flat file sprawl

## Learnings

- **AT peer DMs are architecturally invisible to the lead**: No config, hook, or API changes this. The only mitigations are prompt-level (CC convention) and coordination-level (task list + rendezvous gate). Confirmed by both codebase research and official docs.
- **False stall detection from activity blind spot**: Lead can't distinguish "teammate is idle" from "teammate is executing a long tool call." TeammateIdle hook fires on actual idle, but task status can lag 10-15 seconds. Time-based stall heuristics are unreliable.
- **Premature shutdown is the compound failure**: Peer DM blind spot + activity blind spot = lead triggers shutdown while teammates are actively working on peer-dispatched tasks. Rendezvous gate (explicit WORK COMPLETE from all teammates) is the structural fix.
- **WP sizing should be session-scoped, not dependency-count-based**: A WP with 8 dependencies but trivial work (grep+replace) fits in one session. A WP with 0 dependencies but architectural decisions may not. The constraint is "fits in 200K tokens," not "has few dependencies."
- **QA/Critic found a genuine gap in TC1**: The plan-execution runtime observation was PM-Essentials-specific context, not a plan-creation skill flaw. The Critic read actual SKILL.md source and discovered that Stage 5 creates plans with all-`not_started` — no transitions occur. This is exactly the adversarial value the role is designed to provide.
