## Technical Decisions

### Agents vs Skills Architecture
- **Decision**: Agents are in `.claude/agents/`, skills in `.claude/skills/`
- **Rationale**: Agents run isolated via Task(), skills load into main context
- **Impact**: test-validator moved from tests/skills/ to tests/agents/

### Task-Type Model Selection (P0.1 Update)
- **Decision**: Objective rubric based on task type, not complexity
- **Rationale**: Complexity is subjective; task type (lookup/review/write) is objective
- **Impact**: Haiku=Lookups/Execute, Sonnet=Review/Analyze, Opus=Write/Fix

### Global Hooks for Pipeline Validation
- **Decision**: PreToolUse/SubagentStart/SubagentStop hooks in `.claude/settings.json` (dev) migrated to `hooks/hooks.json` (plugin) in P3.4
- **Rationale**: Skill-scoped hooks only fire when skill is loaded; global hooks ensure deterministic invocation
- **Impact**: P0.3 creates development hooks, P3.4 migrates to plugin

### Progressive Disclosure for Pipelines
- **Decision**: SKILL.md references 6 separate files in references/
- **Rationale**: Keeps main skill under 500 lines, loads templates on-demand
- **Impact**: 6 pipeline template files instead of inline content

### 6 Pipeline Templates

- **Rationale**: Cover core software development patterns
- **Impact**: Extensible structure for adding more templates later

## Learnings

### Sub-agent Skill Loading
Sub-agents invoked via Task() don't automatically load skills. The orchestrator must either:
1. Include skill content in the prompt, OR
2. Define custom agents with `skills:` in frontmatter

### Hook Architecture
- **PreToolUse**: Before tool runs, can validate/block/modify
- **SubagentStart**: When sub-agent begins, good for tracking
- **SubagentStop**: When sub-agent ends, good for verification
- Matchers: Can filter by tool name (e.g., "Task" for sub-agents)

### Custom Agent Override
If a custom agent has `agent:` in frontmatter, that model is used instead of task-type rubric. This was added to P0.1 guidance.
