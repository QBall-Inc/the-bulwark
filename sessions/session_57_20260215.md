## Technical Decisions

### File-Scope Mock Detection Architecture
- **Decision**: Create one T3 lead per file-scope mock per integration/e2e section (cross-product)
- **Rationale**: Each file-scope mock independently contaminates each integration section. A file with 2 file-scope mocks and 2 integration sections produces 4 leads.
- **Impact**: Increases AST precision for mixed-type files like router.test.ts

### Affected Lines Deduplication — Two-Layer Fix
- **Decision**: Added dedup instruction to both mock-detection (file_summaries) and synthesis (effectiveness calculation)
- **Rationale**: Defense in depth — mock-detection should pre-deduplicate, but synthesis should also handle it in case detection doesn't
- **Impact**: Prevents effectiveness inflation when multiple violations share the same scope

### AST Verification Lines as Ground Truth in Classification
- **Decision**: Classification Haiku must use AST-provided verification_lines, not re-count
- **Rationale**: Haiku sometimes echoes total_lines instead of counting correctly. AST is deterministic.
- **Impact**: Prevents inflation like knowledge-linker.test.ts (571/571)

## Learnings

- **File-scope mocks are invisible to section-scoped AST**: The original integration-mock-detector only looked inside describe blocks. File-scope `jest.mock()` is a different pattern — the mock is declared globally but contamination is per-section. Requires cross-reference logic.
- **Double-counting is a LLM synthesis issue, not detection**: The detection agent correctly reports per-violation scope. The synthesis agent naively sums them. Adding explicit union instruction fixes it without changing the detection output schema.
- **Haiku verification_lines unreliable without AST anchoring**: At scale (14+ files), Haiku takes shortcuts. Providing AST data as MANDATORY ground truth eliminates the heuristic counting path entirely.
