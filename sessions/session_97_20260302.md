## Technical decisions

### 3-tier agent invocation guidance
- **Decision**: Agent READMEs classify each agent as standalone-ready, skill-preferred, or pipeline-only
- **Rationale**: User identified that agents can be invoked directly via `claude --agent` or `claude -p --agent`, so docs should guide when that's appropriate vs when to use the parent skill
- **Impact**: Each Sonnet agent assessed its own tier by reading the agent definition. Most product-ideation agents are pipeline-only. Market-researcher is standalone-ready. Plan-creation architect is skill-preferred.

### Humanizer pass skipped for skill READMEs
- **Decision**: User reviewed tone directly and approved, skipping the humanizer skill pass
- **Rationale**: Tone matched the main README.md well enough. Humanizer would add a session with minimal improvement.
- **Impact**: Saved ~15-20K tokens.

### Warning block format
- **Decision**: Used `> [!WARNING]` GitHub admonition format instead of blockquote
- **Rationale**: User requested warning logo. GitHub-flavored markdown renders this as a styled warning block with icon.
- **Impact**: 7 skill READMEs updated.

## Learnings

- Sonnet agents are sufficient for README writing when given a precise prompt template with structure, style rules, and auto-invocation details. Opus not needed for documentation.
- Agent tier classification (standalone-ready / skill-preferred / pipeline-only) emerged as a genuinely useful framework. Agents that read prior stage logs are almost always pipeline-only. Agents with broad analytical roles (market researcher, PO, architect) have standalone value.
- Batch-of-5 with user checkpoint between batches is the right cadence for parallel doc generation. Lets the user review and course-correct without wasting tokens.
- Social preview image on GitHub serves as the visual for LinkedIn posts. No need for separate image uploads.
