## Technical Decisions

### Mandatory Execution Checklist replaces Pre-Flight Gate
- **Decision**: Merged Pre-Flight Gate (MUST/MUST NOT lists + anti-thought traps) and Completion Checklist into a single "Mandatory Execution Checklist (BINDING)" at the top of the skill, right after Usage
- **Rationale**: Checklist is scannable (binary yes/no), position-dominant (first thing read), and mode-agnostic (doesn't predict which thought pattern causes a skip). Anti-thought traps were verbose and the LLM found new rationalizations around them.
- **Impact**: Both create-subagent and create-skill restructured. Stage 3 also retains FIRST/THEN ordering as a structural reinforcement within the stage definition itself.

### Defect pattern: substitution compliance
- **Pattern**: LLM performs manual checks, mentally marks the stage complete, silently drops the tool invocation
- **Root cause**: Manual checks gave false sense of completion. The anti-thought trap addressed "skipping entirely" but not "substituting a weaker version"
- **Fix layers**: (1) Checklist at top with binding language, (2) Stage 3 FIRST/THEN ordering, (3) Stage 3 exit gate with explicit validator invocation check

## Learnings

- **Substitution compliance is a distinct failure mode from skipping**: The LLM doesn't skip the step — it substitutes a weaker version (manual review for tool invocation) and mentally marks it done. Anti-thought traps addressing "skipping" don't catch this. The fix is a mandatory checklist that names the specific action required (e.g., "invoked via Skill tool") so partial compliance is visually distinguishable from full compliance.
- **Checklist position matters more than prose volume**: Moving the checklist from bottom to top had more impact than adding paragraphs of justification in the Pre-Flight section. The LLM reads the checklist first and uses it as a tracking mechanism throughout execution.
- **Generated scaffolds inherit the bare-path bug**: Skills generated by create-skill/create-subagent will have bare `logs/` paths unless the templates they read already use `$PROJECT_DIR/`. The template fix in Session 73 prevents future occurrences, but existing generated output needs manual fixing.
