## Technical Decisions

### Removed inline object literal detection
- **Decision**: Inline objects passed directly as function arguments (`createOrder({ customerId: 'x' })`) are NOT flagged as T3+ violations
- **Rationale**: These are input parameters to start a chain, not broken integration chains. The task brief algorithm specifies "variable used as function argument" — not inline objects.
- **Impact**: Eliminates massive false positives (16 in the validation fixture). Only variables traced to object/array literals are flagged.

### Added property-access tracing
- **Decision**: When an inline object argument contains property accesses like `{ orderId: mockOrder.id }`, trace `mockOrder` back to its source and flag if it's a manually-constructed literal
- **Rationale**: Real-world code often extracts properties from manually-constructed objects rather than passing variables directly. Without this, the analyzer missed violations in the unbiased fixture.
- **Impact**: Catches property-access patterns (`mockOrder.id`), direct argument patterns (`fn(mockOrder.id)`), and direct value patterns (`{ data: mockOrder }`).

### Extracted helper functions for variable flagging
- **Decision**: Created `flagVariableIfLiteral()` and `getRootIdentifier()` helpers to avoid duplication across the 3 argument-checking cases
- **Rationale**: Same flagging logic needed for direct variable args, property-access args, and inline object property values
- **Impact**: Cleaner code, single source of truth for violation flagging

## Learnings

- **User validation catches real issues**: The unbiased fixture (created by Sonnet without reading analyzer code) exposed two significant bugs — inline false positives and missing property-access detection. Automated tests alone would not have caught these because they were written by the same agent that wrote the analyzer.
- **Inline objects are input parameters, not violations**: `createOrder({ customerId: 'x' })` is how you start a chain. The T3+ violation is when a variable is manually constructed to replace upstream output.
- **Property-access tracing is essential**: Real-world code rarely passes entire mock objects directly. Instead, developers extract properties (`mockOrder.id`, `mockOrder.totalAmount`) into new objects. Without property-access tracing, the analyzer misses the majority of real violations.
- **Test-fixture-creation skill works**: Using a separate agent (Sonnet) that cannot read the implementation produces genuinely unbiased fixtures. This should be standard practice for all validation.
