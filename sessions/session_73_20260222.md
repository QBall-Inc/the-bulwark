## Technical Decisions

### $PROJECT_DIR prefix for all output paths
- **Decision**: All output paths in skill templates MUST use `$PROJECT_DIR/` prefix instead of bare relative paths
- **Rationale**: During skill execution, CWD is often the skill directory (`.claude/skills/<name>/`). Bare `logs/` resolves relative to CWD, causing output to be written into the skill directory instead of the project root.
- **Impact**: All create-skill templates, create-subagent templates, bulwark-research, and bulwark-brainstorm updated

### artifacts/ for deliverables (not output/)
- **Decision**: User-facing deliverables (synthesis, reports, generated code) go to `$PROJECT_DIR/artifacts/{skill-name}/{slug}/` — separate from `$PROJECT_DIR/logs/` (intermediate/diagnostics)
- **Rationale**: `artifacts/` is more intuitive than `output/` (which sounds dev-centric). Artifacts can be anything: code, config, docs. Clear separation from diagnostic logs.
- **Impact**: Synthesis documents that previously went to `logs/{skill-name}/synthesis.md` now go to `artifacts/{skill-name}/{slug}/synthesis.md`

## Learnings

- **Bare paths resolve to CWD in skill execution**: Skills loaded via `.claude/skills/` execute with CWD as the skill directory. Any bare relative path (`logs/`, `output/`) creates files inside the skill directory. Always use `$PROJECT_DIR/` prefix in templates that generated skills will read.
- **Blanket pipeline deferral remains effective**: 13+ non-code file edits this session, all deferred. Pattern continues to save significant token budget for doc-heavy work.
- **cp over rsync for dogfood on WSL/NTFS**: rsync fails with "Read-only file system" on `.claude/skills/` (NTFS via WSL) for some temp file operations. Direct `cp` works reliably. Use rsync only for standalone repo sync (which lives on ext4).
