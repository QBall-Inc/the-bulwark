## Technical Decisions

### CLAUDE.md is a standalone template, not an injection
- **Decision**: Create a complete CLAUDE.md from template instead of injecting sections into existing CLAUDE.md
- **Rationale**: User requirement — the sample-claude.md was always intended as a replacement template, not an injection. The brainstorm phase misframed this as injection due to incorrect input to agents.
- **Impact**: `init-project-rules.sh` (injection script) is fully superseded. `init.sh` creates CLAUDE.md from `lib/templates/claude.md`.

### Scope selection: both files at same scope level
- **Decision**: CLAUDE.md and rules.md always installed at the same scope (project or user)
- **Rationale**: CLAUDE.md `@`-mentions rules.md, so they must be co-located for the reference to resolve
- **Impact**: Project scope: `./CLAUDE.md` + `./.claude/rules/rules.md`. User scope: `~/.claude/CLAUDE.md` + `~/.claude/rules/rules.md`.

### User scope strips Project Assets section
- **Decision**: When `--scope=user`, the Project Assets section is removed from CLAUDE.md via sed
- **Rationale**: User-level CLAUDE.md applies to all projects — project-specific assets don't belong there. Post-init checklist suggests creating a project-level CLAUDE.md separately.
- **Impact**: sed pattern `/^## Project Assets/,/^---/d` strips the section.

### File locations confirmed against official Claude Code docs
- **Decision**: Project CLAUDE.md at `./CLAUDE.md`, user at `~/.claude/CLAUDE.md`, rules in `.claude/rules/` at each scope
- **Rationale**: Confirmed via claude-code-guide sub-agent fetching official docs + user's firsthand experience on work machine
- **Impact**: All paths in init scripts match official conventions

### Rules.md slimming must precede distribution
- **Decision**: Move OR/SA rules from Bulwark's CLAUDE.md into Rules.md, slim both, update `lib/templates/rules.md` before plugin distribution
- **Rationale**: `lib/templates/rules.md` is what ships to users. Current version doesn't contain OR/SA rules. Template CLAUDE.md references rule IDs (OR1, SA1-SA6) that must exist in Rules.md.
- **Impact**: S92 priority. Distribution pushed to S93-S94.

### Init skill is a priority gap
- **Decision**: Must create an init skill (`skills/init/SKILL.md`) so users can invoke `/the-bulwark:init` from within Claude Code
- **Rationale**: Plugin users invoke functionality via slash commands, not bash scripts. The init skill wraps `scripts/init.sh`, checks if already initialized, and informs the user.
- **Impact**: S92 priority #2 after Rules.md slimming.

## Learnings

- **Brainstorm input framing matters**: The brainstorm agents were given the wrong framing (injection into existing CLAUDE.md) when the user's intent was a standalone replacement template. This caused the entire init design to be built around the wrong assumption. Always verify the problem statement with the user before passing to sub-agents.
- **CRLF persists across sessions on NTFS**: Files created in S90 on `/mnt/c/` had CRLF. Must convert immediately after creation, not assume the Write tool preserves LF on NTFS paths. cleanup-stale.sh was affected.
- **sed range patterns need CRLF awareness**: `sed '/^pattern$/,/^---$/d'` fails silently with CRLF because `$` matches before `\n` but `\r` is before it. Use `/^pattern/,/^---/d` (no `$` anchor) for robustness on mixed-ending files.
- **Plugin testing requires explicit paths**: `claude plugin validate .` and `claude --plugin-dir ./` are ambiguous with multiple plugins in the same parent directory. Always use absolute paths: `claude plugin validate /mnt/c/projects/the-bulwark`.
