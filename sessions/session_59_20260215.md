## Technical Decisions

### Two-skill split (bulwark-research + bulwark-brainstorm)
- **Decision**: Split the draft research-brainstorm into two independent skills
- **Rationale**: Token budget (5 agents + synthesis consumes most of 200K), CS1 single responsibility, independent invocation
- **Impact**: Each skill runs in its own session; cross-skill composition via pipeline template

### Diagnostic templates extracted to templates/ directory
- **Decision**: Move diagnostic YAML schemas from inline in SKILL.md to `templates/diagnostic-output.yaml`
- **Rationale**: User feedback — diagnostics are output materials like other templates
- **Impact**: SKILL.md references template instead of embedding schema

### SME autonomous codebase exploration
- **Decision**: Project SME agent discovers relevant files via Glob/Grep/Read, no hardcoded paths
- **Rationale**: Hardcoding architecture.md/tasks.yaml makes skill Bulwark-specific; portability matters
- **Impact**: Skill works in any project (tested via smart-todo empty project)

### argument-hint added to both skills
- **Decision**: Added `argument-hint: "<topic, filepath, or directory>"` per validator suggestion
- **Rationale**: Improves autocomplete UX in `/` menu
- **Impact**: Users see expected input format when invoking

### model field skipped in frontmatter
- **Decision**: Did not add `model` field despite validator suggestion
- **Rationale**: Both skills run in main context (not `context: fork`), so `model` has no effect. Sub-agents get their model via Task tool `model` parameter.
- **Impact**: None — avoids misleading metadata

## Learnings

- **Diagnostic schemas belong in templates/**: User correctly identified that diagnostic output YAMLs are output materials, not SKILL.md content. Extracting to `templates/diagnostic-output.yaml` is cleaner and consistent with the other templates.
- **argument-hint improves discoverability**: Small frontmatter addition with outsized UX benefit. Should be standard for all user-invocable skills with arguments.
- **model field semantics matter**: `model` in frontmatter controls the orchestrator model (only meaningful with `context: fork`). For main-context skills, sub-agent models are set via Task tool. Adding `model` without `context: fork` is misleading.
- **Rsync prefix stripping pattern is reusable**: Section 2b in sync script uses the same `src:dest` mapping array as ez-statusline. Pattern works for any bulwark-prefixed skill destined for standalone repo.
