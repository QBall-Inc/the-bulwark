## Technical Decisions

### Fixture placement strategy
- **Decision**: Source code in `scripts/modules/` (typechecked), YAML data in `tests/fixtures/implementer/` (excluded from typecheck), test files in `tests/modules/` (typechecked)
- **Rationale**: `scripts/**/*.ts` and `tests/**/*.ts` are in tsconfig include; `tests/fixtures/**` is excluded. Source and test files must compile; YAML data files don't need compilation.
- **Impact**: All fixtures pass Phase 1. Implementer agent can modify source files and run implementer-quality.sh successfully.

### Shared fixtures for direct and pipeline tests
- **Decision**: Direct invocation (P4.4-1/2) and pipeline invocation (P4.4-4/5) share the same source files. Pipeline tests use different prompts and context.
- **Rationale**: The source code is identical; only the invocation pattern differs. Avoids fixture duplication.
- **Impact**: Fixtures must be restored (`git checkout`) between direct and pipeline tests.

### Test Execution & Fix FixWriter is orchestrator
- **Decision**: Clarified in SKILL.md quick reference that FixWriter in Test Execution & Fix pipeline is orchestrator, not bulwark-implementer
- **Rationale**: Orchestrator sees PostToolUse hook output directly (no workaround needed), LOOP iteration is natural, fixes are simpler (failing tests, not root-cause debugging)
- **Impact**: Only Fix Validation Stage 2 and New Feature Stage 3 use bulwark-implementer

### Completion policy
- **Decision**: Added to MEMORY.md: never mark tasks complete until fully validated
- **Rationale**: Implementation without testing = "in progress". Repeated incorrect completion status in previous sessions.
- **Impact**: All future tasks follow implementation + validation + user confirmation = complete
