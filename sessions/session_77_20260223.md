## Technical Decisions

### AT validation via separate skill before plan-creation integration
- **Decision**: Create dev-team skill in PM-Essentials to test AT mechanics independently, before adding AT mode to plan-creation
- **Rationale**: AT is untested in any Bulwark/PM-Essentials skill. Building it directly into plan-creation (complex 4-agent pipeline) means debugging AT mechanics AND skill logic simultaneously. Separate skill isolates AT mechanics.
- **Impact**: T-010 blocked on TC2 results. If AT fails, plan-creation ships with Task tool mode only.

### Agent Teams teammates don't use agent files
- **Decision**: AT teammates are spawned via natural language role descriptions, not `.claude/agents/*.md` files
- **Rationale**: AT teammates are independent Claude sessions spawned by the lead, not Task tool sub-agents. Agent files are for Task tool `subagent_type` invocations. AT uses inline role descriptions at spawn time.
- **Impact**: dev-team skill has no separate agent files — all teammate definitions are embedded in SKILL.md spawn instructions.

### URL shortener as dev-team test case (not calculator)
- **Decision**: Use URL shortener CLI tool instead of calculator app for TC2
- **Rationale**: Calculator is a known tutorial project — Claude would recognize it as a test fixture and potentially shortcut with memorized responses. URL shortener has real security surface (open redirect, SSRF, slug injection), genuine edge cases (unicode domains, collisions), and meaningful architectural decisions.
- **Impact**: Better AT validation — teammates have real issues to debate, not toy problems.

## Learnings

- **AT is natural language, not API**: Agent Teams has no structured tool calls. Teammates are spawned, messaged, and coordinated entirely through natural language instructions. SKILL.md writes imperative prose, not code. This is fundamentally different from Task tool sub-agents.
- **Separate AT test skill prevents compound debugging**: Testing AT mechanics (mailbox, task list, peer debate) separately from domain complexity (4-role plan creation) isolates failure modes. If AT fails in dev-team, we know it's AT, not our pipeline logic.
- **Three project paths must not be confused**: Bulwark (dev), Essential Skills (standalone rsync target), PM-Essentials (test harness). Previous sessions sometimes conflated the standalone repo with PM-Essentials.
