## Technical Decisions

### Two-Gate REWRITE_REQUIRED Logic
- **Decision**: Gate 1 (Impact) triggers on any P0 violation; Gate 2 (Threshold) triggers on P1 + <95% effectiveness
- **Rationale**: P0 violations (false confidence) are unacceptable regardless of percentage; P1 needs volume check
- **Impact**: More nuanced than simple "any critical triggers rewrite"

### Test Effectiveness Metric
- **Decision**: test_effectiveness = (verification_lines - affected_lines) / verification_lines
- **Rationale**: Treats violation-affected lines as "dead test code" - provides meaningful coverage metric
- **Impact**: P0.6 counts verification_lines, P0.7 tracks violation_scope, P0.8 calculates percentage

### P0.7 Model Upgrade to Sonnet
- **Decision**: Changed P0.7 from Haiku to Sonnet
- **Rationale**: Mock appropriateness analysis requires contextual judgment (call graph, data flow), not just pattern matching
- **Impact**: Pipeline is now Haiku → Sonnet → Sonnet instead of Haiku → Haiku → Sonnet

### Impact-Based Priority
- **Decision**: P0 = false confidence (T1, T3+), P1 = incomplete verification (T2, T3), P2 = pattern issues
- **Rationale**: Prioritize by impact on test reliability, not just rule severity
- **Impact**: Affects which gate triggers and rewrite ordering

## Learnings

### Test Effectiveness as Coverage Metric
- Treating violation-affected lines as "ineffective" provides a meaningful metric
- A 5-line mock that affects 90 lines of assertions = 5% effective, not 95%
- Violation scope tracking is key to accurate calculation

### Two-Gate Approach
- Separates "unacceptable" (P0) from "concerning if pervasive" (P1)
- Prevents over-triggering on isolated minor issues
- Provides clear rationale for rewrite decisions
