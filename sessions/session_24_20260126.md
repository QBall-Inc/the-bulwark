## Technical Decisions

### DD2: Chained Execution Model
- **Decision**: enforce-quality.sh chains to suggest-pipeline.sh internally instead of parallel hooks
- **Rationale**: Hooks fire in parallel by default; chaining gives sequential behavior, prevents race conditions
- **Impact**: Only enforce-quality.sh registered in hooks.json; suggest-pipeline.sh called via stdin passthrough

### TR6: Development Sync Script
- **Decision**: Create sync-hooks-for-dev.sh to copy hooks.json to settings.json with path transformation
- **Rationale**: Plugin hooks use ${CLAUDE_PLUGIN_ROOT} but dogfooding needs ${CLAUDE_PROJECT_DIR}
- **Impact**: Single source of truth (hooks.json), automated sync for local testing

### TR7: Skill-Based SessionStart Protocol
- **Decision**: Governance content lives in `skills/governance-protocol/SKILL.md`, inject-protocol.sh reads it
- **Rationale**: User-extensible without bash editing, readable markdown, consistent with skill architecture
- **Impact**: Users can add project-specific rules to skill's "Project-Specific Rules" section

## Learnings

- Claude Code hooks execute in **parallel** by default, not sequentially
- Stop hook fires once at end of response, cannot be used as per-tool quality gate
- SessionStart exit 0 stdout goes to Claude's CONTEXT (special behavior vs other hooks)
- `once: true` supported in Claude Code 2.1.1+ for skills and global hooks
