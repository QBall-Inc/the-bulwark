## Technical Decisions

### implementer-quality.sh as Separate Script
- **Decision**: Create new `scripts/hooks/implementer-quality.sh` that accepts file path as CLI argument, rather than modifying `enforce-quality.sh`
- **Rationale**: enforce-quality.sh has a contract with the hook system (reads stdin JSON). Dual-mode input parsing violates CS2 (No Magic). Clean separation of concerns.
- **Impact**: Two scripts with similar logic. Double execution accepted (global hook + agent script = 2x typecheck/lint per Write/Edit). Extract shared core later if maintenance becomes a burden.

### SA6 Rule Added to Rules.md
- **Decision**: New rule SA6 requiring orchestrator to evaluate pipeline suggestions from code-writing sub-agents (execute or document deferral)
- **Rationale**: Without a Rules.md entry, MANDATORY language in sub-agent summaries is just a convention that could erode. SA6 makes it permanent and auditable.
- **Impact**: Applies to all future code-writing agents, not just bulwark-implementer. The "or document deferral" escape valve prevents unnecessary pipeline runs for trivial changes.

### Pipeline Suggestions as Skill-Level Pattern
- **Decision**: Added `pipeline_suggestions` to subagent-output-templating skill (not just the implementer agent)
- **Rationale**: Any future code-writing agent should follow this pattern. Establishing it at the skill level ensures consistency.
- **Impact**: subagent-output-templating YAML schema, minimal log template, and summary checklist all updated. Code-writing agents MUST include; read-only agents omit.

### Double Execution Accepted
- **Decision**: Accept that both `enforce-quality.sh` (global hook) and `implementer-quality.sh` (agent-invoked) run on every Write/Edit by the implementer
- **Rationale**: Global hook output goes to sub-agent context (unactioned). implementer-quality.sh is the primary enforcement channel. The tax is 2x typecheck/lint - acceptable for quality assurance.
- **Impact**: No changes to global hooks. No detection logic needed. Clean separation.

## Learnings

- **4-5 sessions to finalize P4.4 task brief**: Hook validation (T-044) uncovered agent-scoped hooks bug, which cascaded into design pivots. The live calculator test in Session 41 revealed that even global hook output is unactioned in sub-agent context. Each discovery required re-evaluating the design. This is the cost of building on undocumented/buggy platform features.
- **Three-channel enforcement pattern**: For sub-agent quality enforcement, a single channel is fragile. The final design uses three: (1) direct script invocation by agent, (2) pipeline suggestions in implementation report log, (3) MANDATORY language in summary to orchestrator. Belt, suspenders, and SA6 rule.
- **Skill-level vs agent-level patterns**: Adding `pipeline_suggestions` to subagent-output-templating (not just the implementer agent) establishes a reusable pattern. Future code-writing agents inherit it automatically.
