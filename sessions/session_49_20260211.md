## Technical Decisions

### ts-morph as AST tooling
- **Decision**: Use ts-morph (v27.x) for all three AST scripts
- **Rationale**: Type-aware analysis critical for data-flow-analyzer and verification-counter. ts-morph wraps TypeScript Compiler API with 10x better ergonomics. tree-sitter lacks type information. Raw Compiler API too verbose.
- **Impact**: 1.47MB dependency bundled inside skills/test-audit/scripts/

### Justfile recipes for script invocation
- **Decision**: Scripts invoked via `just verify-count`, `just skip-detect`, `just ast-analyze` — not `npx tsx` directly
- **Rationale**: V2 compliance (use just for execution). Abstracts runtime — can swap tsx for compiled JS later without changing callers.
- **Impact**: Three new recipes in Justfile. Must propagate to scaffold templates in P6.2/P7.1.

### Phased implementation with validation gates
- **Decision**: 4 phases, each with a validation gate that must pass before proceeding
- **Rationale**: Viewpoint analysis unanimously recommended against building everything at once. Phase 1 validates tooling stack. Phase 2 validates most complex script. Dual-mode (Phase 4) only after scripts proven.
- **Impact**: Slower rollout but lower risk of building on unproven foundations.

### Graceful degradation on AST failure
- **Decision**: If AST scripts fail, audit falls back to LLM-only analysis (current behavior)
- **Rationale**: Viewpoint 1 flagged undefined failure handling. AST enhancement shouldn't break existing functionality.
- **Impact**: Stage 0 is best-effort, not blocking.
