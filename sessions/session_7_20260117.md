## Technical Decisions

### Skill Argument Handling
- **Decision**: Use `$ARGUMENTS`/`$1` in skill content, not frontmatter
- **Rationale**: Claude Code skills access arguments via placeholders in content, no special frontmatter field
- **Impact**: Added Step 0 to anthropic-validator to resolve target from arguments

### Batch Validation Strategy
- **Decision**: Process each skill individually, then aggregate results
- **Rationale**: Frontloading all skills into one reviewer context causes shallow analysis due to context overload
- **Impact**: Updated Batch Validation section with explicit "DO NOT frontload" warning

### Hooks Config Structure
- **Decision**: Our current hooks config is correct; validator produced false positive
- **Rationale**: `hooks` array wrapper is required for ALL event types, even without matcher
- **Impact**: Updated hooks-checklist.md with correct structure documentation

### Issue-Debugging Usability
- **Decision**: Added explicit "When to Use" section with trigger patterns
- **Rationale**: Claude wasn't automatically loading skill because triggers weren't clear
- **Impact**: Skill now has table of trigger patterns (bug investigation, test failures, etc.)

## Learnings

### Validator Self-Discovery
- Running anthropic-validator on our own assets revealed real issues (issue-debugging usability) and false positives (hooks structure)
- The validator needs better hooks knowledge - corrected in fallback checklist

### Skill Argument Passing
- Skills and commands share invocation mechanism but remain structurally separate
- `user-invocable: true` + `$ARGUMENTS` in content is the pattern for user-facing skills with arguments

### Batch Processing Anti-Pattern
- Combining multiple assets into single reviewer context causes shallow analysis
- Always process individually, aggregate at end
