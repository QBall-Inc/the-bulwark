## Technical decisions

### Parallel Opus sub-agents for README writing
- **Decision**: Spawn 5 Opus agents in parallel per batch instead of writing sequentially
- **Rationale**: Each skill README requires reading the full SKILL.md (300-500 lines). Reading 10 skills in the main context would consume 60%+ of tokens. Sub-agents each get their own context window.
- **Impact**: All 10 meaty READMEs completed in one session instead of spanning two.

### Token warning block for multi-agent skills
- **Decision**: Add a warning block after the title for skills that spawn 3+ sub-agents (7 skills)
- **Rationale**: User feedback. Skills like product-ideation (6 agents) are token-intensive. Users should check their budget before triggering.
- **Impact**: 7 targeted edits in S97.

### Prompt template preserved for batch 2
- **Decision**: Stored the full prompt template used for meaty skills in `tmp/s96-readme-review-comments.md`, along with a simplified template for simple skills
- **Rationale**: Ensures consistency across sessions. S97 can reuse the exact same prompting pattern.
- **Impact**: No re-derivation needed. Batch 2 can start immediately.

## Learnings

- **Opus sub-agents are effective README writers when given precise structure specs.** All 10 agents produced style-compliant output with correct Mermaid diagrams, cross-links, and tone matching. The key is giving each agent: exact structure order, style rules, source file path, destination path, and line count target.
- **Reading all source files in main context is wasteful for doc generation.** Reading 10 SKILL.md files (300-500 lines each) consumed 60% of context before any writing started. Delegating to sub-agents that each read one file is far more efficient.
- **Batch-of-5 with user checkpoint is the right cadence.** Lets the user review output quality and provide feedback before the next batch runs. Prevents wasted work if the structure needs adjustment.
