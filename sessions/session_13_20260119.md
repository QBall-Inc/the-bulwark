## Technical Decisions

### P1.2 User-Invocable Design
- **Decision**: Make bulwark-issue-analyzer user-invocable: true
- **Rationale**: Agent serves dual purpose - direct invocation via `/bulwark-issue-analyzer` for ad-hoc analysis AND pipeline stage 1 for Fix Validation
- **Impact**: Agent appears in `/` menu, users can analyze issues without running full pipeline

### $ARGUMENTS Uncertainty
- **Decision**: Document $ARGUMENTS as "to be validated" with fallback to CONTEXT
- **Rationale**: $ARGUMENTS works for skills but uncertain if custom sub-agents receive arguments
- **Impact**: Implementation includes fallback - if $ARGUMENTS doesn't work, orchestrator passes path in CONTEXT

### Test Fixtures No-Bias Pattern
- **Decision**: Test fixtures contain ONLY code - NO explanatory comments about bugs
- **Rationale**: Prevents confirmation bias. Agent must detect issues through actual analysis, not by reading hints.
- **Impact**: Expected results stored separately in `issue-analyzer-expected/` folder

### No PreToolUse Hook for Prompting
- **Decision**: Skip PreToolUse hook on Task for enforcing 4-part template
- **Rationale**: By time PreToolUse fires, prompt already composed. Instead, added subagent-prompting to agent's skills list.
- **Impact**: Agent has access to 4-part template via loaded skill

## Learnings

- **Timestamp standardization prevents pipeline failures**: The {timestamp}/{ts} ambiguity was causing test-audit stages to not find each other's outputs. Standardizing to {YYYYMMDD-HHMMSS} for file paths and {ISO-8601} for YAML fields resolves this.
- **Bias avoidance in test fixtures**: Following the P0.6-8 pattern of separating code from expected results ensures agent analyzes without hints.
