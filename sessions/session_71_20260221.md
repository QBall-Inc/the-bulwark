## Technical Decisions

### Pipeline agents are architecturally infeasible as sub-agents
- **Decision**: Sub-agents (invoked via Task tool) cannot spawn other sub-agents. Pipeline orchestrator agents only work via `claude --agent <name>` (main context). Rename create-agent → create-subagent; pipeline orchestration belongs in skills.
- **Rationale**: TC3 validation proved the constraint. A pipeline agent created by create-agent would produce a non-functional asset when invoked as a sub-agent.
- **Impact**: P5.5a revision — rename + simplify to single-agent template + enhance create-skill for pipeline sub-agent generation.

### tmp/ working directory for creator skills
- **Decision**: Both create-skill and create-subagent generate/refine in `tmp/create-{type}/{name}/` during Stages 2-4, deploying to final location only at Stage 5.
- **Rationale**: User experienced 20+ edit approval prompts due to .claude/ sandbox restrictions during iterative refinement loops.
- **Impact**: Reduces approval prompts from 20+ to 1 batch at deploy time.

### Anthropic-validator known conventions
- **Decision**: Added imperative register patterns (Pre-Flight Gate, MUST/MUST NOT, Protocol sections, Permissions Setup, Completion Checklists) as known conventions in the reviewer's CONTEXT. Classified as informational, not high-severity violations.
- **Rationale**: Empirically validated across Opus and Sonnet — behavioral register instructions get ignored, imperative register gets followed. Standards reviewer was producing false-positive high-severity findings for our established patterns.
- **Impact**: Eliminates 2 high + 1 low false positives per validation run.

### Invocation section removal is acceptable
- **Decision**: Standards reviewer recommended removing Invocation section from generated agents. Accepted as valid — it's caller-facing documentation in the agent's system prompt (wastes tokens). But kept the requirement in our agent-conventions reference for future generations (negligible token cost, useful when orchestrator lacks subagent-prompting skill).
- **Rationale**: The Invocation section serves no behavioral purpose for the agent itself.
- **Impact**: TC1 check 9 reclassified as PASS.

## Learnings

- **Pipeline agents infeasible as sub-agents**: `.claude/agents/*.md` invoked via Task tool run as sub-agents which cannot spawn further sub-agents. Pipeline orchestration must be implemented as skills, not agents. This is a fundamental Claude Code constraint.
- **tmp/ working directory eliminates edit storms**: Skills that iteratively generate/refine files in `.claude/` cause 20+ approval prompts. Working in `tmp/` during refinement and deploying at the end reduces this to 1 batch.
- **Imperative register > behavioral register**: Both Opus and Sonnet reliably comply with "You MUST verify..." but inconsistently follow "You always verify...". This is empirically validated (DEF-P4-005) and should be treated as a known convention, not a violation of Anthropic's behavioral-register recommendation.
- **Validator findings can improve generated output**: The standards reviewer's recommendation to remove the Invocation section was legitimate — caller-facing docs don't belong in the agent's system prompt. Validation-driven refinement works as designed.
