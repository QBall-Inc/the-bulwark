## Technical Decisions

### Main Context Orchestration Pattern
- **Decision**: P0.6/P0.7/P0.8 are all skills (not agents), with NO `agent:` field
- **Rationale**: Orchestrator (Opus) must spawn sub-agents; sub-agents can't spawn sub-agents. Skills serve as prompt templates.
- **Impact**: Consistent with anthropic-validator pattern. P1.1 (bulwark-test-auditor) may be redundant.

### Bias Avoidance via Model Separation
- **Decision**: Haiku/Sonnet do audit work, Opus only orchestrates and implements fixes
- **Rationale**: Opus reviewing its own work introduces bias. Audit should be external.
- **Impact**: All three skills spawn general-purpose sub-agents with explicit model selection.

### Automatic Rewrite on REWRITE_REQUIRED
- **Decision**: No user confirmation required; rewrite happens automatically
- **Rationale**: Same flow for explicit (/test-audit) and implicit (hook) invocation. Simpler. User can interrupt.
- **Impact**: Pipeline includes LOOP(max=2) to prevent infinite cycles.

## Learnings

### Skills as Prompt Templates
- Skills without `agent:` field serve as reference materials
- Orchestrator loads skill content, constructs 4-part prompt, spawns general-purpose agent
- Clean separation: skill defines WHAT, orchestrator handles HOW

### Main Context Orchestration Pattern
- Matches anthropic-validator design
- User-invocable skill with orchestration instructions
- Orchestrator follows instructions, spawns sub-agents sequentially
- Sub-agents write to logs/, orchestrator reads and chains
