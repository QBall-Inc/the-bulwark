## Technical Decisions

### Hook Instruction Chain Made Explicit
- **Decision**: Updated `suggest-pipeline.sh` to require loading `subagent-prompting` and `subagent-output-templating` skills before spawning sub-agents
- **Rationale**: Previous instructions allowed loophole where sub-agents returned inline instead of writing to logs
- **Impact**: Closes loophole, ensures audit trail and artifact flow

### Hook Disabled by Default
- **Decision**: Added flag file mechanism (`.claude/enable-pipeline-hooks`) to enable/disable PostToolUse pipeline hook
- **Rationale**: Hook was firing on every edit during development, not just validation scenarios
- **Impact**: Enable only for manual tests or before writing scripts

### Skill Loading Trigger
- **Decision**: Word "debug" in user prompt triggers skill loading, but "look" or "investigate" does not
- **Rationale**: Skill description uses "debugging" terminology for semantic matching
- **Impact**: Users should use "debug" when they want the methodology applied

### Severity Threshold for P0.5
- **Decision**: Any Critical = Fail, but report ALL findings with severity levels
- **Rationale**: User needs full visibility to make informed decisions on what to fix
- **Impact**: Comprehensive reports, user agency preserved

## Learnings

### Hook Instruction Design
- Instructions must be explicit about the FULL workflow, not just "orchestrate sub-agents"
- Loopholes occur when intent is implied rather than stated
- Make dependency chain explicit: skill A → skill B → action → verification

### Skill Semantic Matching
- Skill description keywords affect when skill is loaded
- "Debug" triggers issue-debugging skill, but synonyms may not
- Consider including common synonyms in description for better matching

### Test Fixture Value
- Intentional bugs in fixtures enable real methodology verification
- Debug reports demonstrate skill works as designed
- Manual verification catches issues automated tests might miss
