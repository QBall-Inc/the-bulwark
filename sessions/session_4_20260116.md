## Technical Decisions

### Skill vs Agent for P0.4
- **Decision**: P0.4 is a skill (methodology), P1.2/P1.3 are agents that use it
- **Rationale**: Skills provide knowledge/methodology; agents provide execution context
- **Impact**: Clear separation - skill defines "how to debug", agents execute it

### Debug Report as Separate File
- **Decision**: Debug report at `logs/debug-reports/{issue-id}.yaml`, not embedded in log
- **Rationale**: Cleaner separation, easier for FixValidator to reference
- **Impact**: IssueAnalyzer summary includes path to debug report

### Tiered Validation Plan
- **Decision**: P1 (must run), P2 (should run), P3 (nice-to-have)
- **Rationale**: Full regression (4000+ tests) is impractical; prioritize by relevance
- **Impact**: FixValidator runs tests by tier based on available budget

### Role-Based Agents for Code Review
- **Decision**: Code Review pipeline uses general-purpose agents with roles, each loading code-review skill sections
- **Rationale**: More flexible than custom agents per section; skill has 4 sections (Security, Type Safety, Linting, Coding Standards)
- **Impact**: Pipeline orchestrates role-based agents; bulwark-code-auditor for standalone use

### FixValidator Added to Pipeline
- **Decision**: New stage between TestWriter and CodeReviewer
- **Rationale**: Needed to execute validation plan from debug report and assess confidence
- **Impact**: P1.3 task added; Fix Validation pipeline now has 5 stages

## Learnings

### Architecture Patterns
- Skills provide methodology (knowledge); agents provide execution
- Pipeline stages can be role-based general-purpose agents with skill references
- Custom sub-agents (like bulwark-issue-analyzer) encapsulate stage behavior

### Debug Report as Contract
- Debug report is the contract between IssueAnalyzer → FixWriter → FixValidator
- Summary includes path; downstream stages read the full report
- Confidence rubric enables objective fix assessment
