## Technical Decisions

### Inline context, no sub-agents
- **Decision**: Skill runs inline (no fork), orchestrator executes all stages directly via tools
- **Rationale**: Multiple user interaction points (confirm languages, restart prompt). No distinct analysis needing separate agents — each step is a Bash/Read/Write/AskUserQuestion call.
- **Impact**: Simple execution model, no agent permissions to configure

### Exit+resume instead of restart
- **Decision**: Stage 5 checkpoint uses `claude --resume ${CLAUDE_SESSION_ID}` instead of generic "restart"
- **Rationale**: User clarified that restart = new session = lost context. Must exit and resume to preserve skill continuity.
- **Impact**: `${CLAUDE_SESSION_ID}` is a documented substitution variable — no CLI command needed to fetch it

### Mandatory checklist at top (not bottom)
- **Decision**: Moved completion checklist to top of SKILL.md as "Mandatory Execution Checklist (BINDING)"
- **Rationale**: DEF-P4-005 + create-skill pattern — bottom checklists are ignored because Claude commits to execution plan before reaching them
- **Impact**: Consistent with all other Bulwark pipeline skills

### `--lang` flag for explicit language specification
- **Decision**: Added `--lang <languages>` flag that skips detection and uses user-specified languages directly
- **Rationale**: Enables testing in empty projects without manifests, and gives users who know their stack a faster path
- **Impact**: Pre-Flight Gate routes to Stage 2 with EXPLICIT_LANGUAGES; registry lookup replaces scanning

### CN rules placement in Rules.md
- **Decision**: CN1 (prefer LSP for semantic ops) and CN2 (search tool hierarchy table) added between SC and SR sections
- **Rationale**: Operational rules that apply during code navigation. Without explicit rules, Claude defaults to Grep/Glob even with LSP available.
- **Impact**: Takes effect immediately for any project with LSP configured

## Learnings

- **create-skill pipeline works well for reference-heavy skills**: The adaptive interview extracted answers from the task brief efficiently. Classification was straightforward (inline, no sub-agents, reference-heavy template). Sonnet generator produced structurally sound output on first pass.
- **`${CLAUDE_SESSION_ID}` is a documented substitution variable**: Available in skill content — no need for CLI commands to fetch session ID. Discovered via claude-code-guide standards fetch.
- **Validator caught real issues**: `$PROJECT_DIR` is not a documented substitution variable (use relative paths or `${CLAUDE_PROJECT_DIR}`). Duplicate stage numbering was a genuine ambiguity. These would have caused confusion during execution.
- **Empty-detection path is essential for testing**: Without explicit handling for "no manifests found", the skill would silently produce an empty list and proceed to install nothing. Manual language selection makes the skill usable in any project context.
